The 1-Year Core Update Cadence

Predictable, planned, and scripted — how the yearly core component cycle eliminates surprise breaking changes and gives every team time to adopt on their terms.

Why a Predictable Cadence Matters

In enterprise software, surprise major-version upgrades create emergencies. Teams scramble, sprints derail, and technical debt compounds when a framework vendor drops a breaking release without warning. The 1-year core update cadence eliminates that chaos entirely.

📅 Core Principle A predictable cadence lets teams plan. Surprise major versions create emergencies. Every core release is planned twelve months in advance, documented thoroughly, and shipped with automated migration scripts.

Core components are the shared building blocks every domain relies on: header, footer, buttons, forms, data tables, navigation, alerts, modals, and layout primitives. These are published as W3C-standard Web Components, distributed via CDN, and versioned independently from any single domain.

  • One new edition per year — a WCFramework edition plus a React compatibility edition, published simultaneously
  • Security patches ship quarterly — minor releases within the current core version that require zero domain changes
  • Accessibility improvements bundled — WCAG updates are folded into the annual release, not shipped as surprise patches
  • Performance optimizations included — each edition benchmarks faster than the last, with no API changes required
  • New shared components — community-requested additions to the core library, vetted by the platform team
Industry Standard: Predictable Release Trains Organizations like Google (Chrome), Ubuntu (LTS), and Node.js (LTS) all use time-based release cadences. Teams know exactly when to expect changes and have documented timelines for adoption. This is the same model applied to internal shared components.

How the Annual Cycle Works

Each year follows a structured, repeatable process. The platform team begins working on the next core edition in Q1. By Q3, the pre-release is available for domain teams to test. In Q4, the stable edition ships with full documentation and automated migration scripts.

Q1–Q2: Design & Implementation Platform team builds

The platform team publishes an RFC (Request for Comments) in January. Domain teams have 30 days to provide feedback. By March, the API surface is locked and implementation begins. All changes are backward-compatible within the same major version.

  • RFC published to internal documentation site with full change catalog
  • Feedback window: 30 days, tracked in issue tracker
  • Breaking change threshold: zero within minor versions, documented for major
  • New component proposals vetted through the App Catalog governance process
Q3: Beta & Early Adoption Domain teams test

Pre-release packages are published to a beta CDN channel. Early adopter domains (typically Catalog and Cart) integrate and report issues. The change-mod scripts are finalized based on real migration feedback.

  • Beta packages available via CDN at /core/v{N+1}-beta/
  • Automated compatibility tests run against all registered domains
  • Migration script dry-run mode: shows what would change without applying
  • Bug reports prioritized with 48-hour response SLA
Q4+: General Adoption & Gradual Rollout Teams adopt at their pace

The stable release ships in September. Domain teams adopt on their own schedule — there is no hard deadline. Teams on last year's core continue to receive security patches. Teams two years back also continue to function; the CDN serves all supported versions.

  • Stable packages at /core/v{N+1}/
  • Change-mod scripts fully tested and documented
  • Adoption dashboard shows which domains are on which version
  • No forced migration — teams upgrade when it fits their sprint cadence

Change-Mod Scripts: Automated Upgrades

The single most important tool in the cadence process is the change-mod script. It automates the mechanical work of upgrading each domain's core component references. No manual find-and-replace. No guesswork. No missed imports.

How It Works

  1. Run the script — npx @core/change-mod --target v{N+1}
  2. Review the diff — the script generates a standard git diff showing every change
  3. Test locally — run each domain's existing test suite against the updated components
  4. Deploy — push through our normal domain deployment pipeline
✅ That's It Run the script → review the diff → test → deploy. Four steps. No meetings. No coordination. No release trains.

What the Script Actually Changes Technical detail

Import Path Updates

The script updates CDN import paths from the previous core version to the new one. For example:

// Before
import '@core/v4/button.js';
import '@core/v4/data-table.js';

// After (change-mod applied)
import '@core/v5/button.js';
import '@core/v5/data-table.js';

API Signature Adjustments

If a component's public API changed (e.g., a property was renamed or a new required attribute was added), the script updates every usage in each domain's templates and code.

Deprecation Cleanup

Components or APIs deprecated in the previous version are replaced with their documented successors. The migration manifest contains the full mapping.

Dry-Run Mode Preview before applying

Run npx @core/change-mod --target v{N+1} --dry-run to see exactly what would change without modifying any files. The output is a standard unified diff that we can review, share with our team, or pipe to a code review tool.

  • Zero side effects — reads our source, computes the diff, writes nothing
  • Includes line-by-line explanations for each change
  • Generates a compatibility report: which tests might be affected
  • Can be integrated into CI as a "readiness check" before the actual migration

Version Support & CDN Policy

A common fear with shared components is: "What if we're not ready to upgrade and the old version disappears?" The answer is simple: it doesn't.

  • Current version (vN) — full support, security patches, bug fixes
  • Previous version (vN-1) — security patches only, CDN served, fully functional
  • Two versions back (vN-2) — CDN served, no active patches, still works
  • End of life — versions older than N-2 are archived but still accessible via pinned URLs
⚠️ No Surprise Removals The CDN never removes a published version. Even end-of-life versions remain accessible at their pinned URL. The "end of life" designation means no further patches — not that the code disappears.

Real-World Adoption Patterns

Not every domain adopts on the same day — and that's by design. Here's what the typical adoption wave looks like across the organization:

Wave 1: Early Adopters (Month 1–2) Platform-adjacent teams

Teams closest to the platform (Catalog, Search) adopt first. They have the most experience with change-mod scripts and provide early feedback that benefits later adopters.

  • Typically 2–3 domains in the first wave
  • Feedback loop: issues found here are fixed before general availability
  • Adoption time: 1–2 sprints including testing
Wave 2: General Adoption (Month 2–4) Most domain teams

The majority of domain teams adopt during this window. Change-mod scripts are battle-tested, documentation is refined, and any edge cases from Wave 1 are resolved.

  • Typically 5–8 domains in this wave
  • Adoption time: 1 sprint for most domains
  • Support channel active for migration questions
Wave 3: Late Adopters (Month 4–8) Complex or low-priority domains

Some domains have complex customizations or lower release frequency. They adopt later — and that's perfectly fine. The previous core version continues to receive security patches throughout this window.

  • Typically 2–4 domains in this wave
  • Often coincides with other planned refactoring work
  • No pressure, no deadlines — just the team's own sprint cadence
Industry Standard: Adoption Waves Google's internal component library uses the same tiered adoption model. Early adopters validate, general teams follow, and late adopters have extended windows. The key is that no one is forced — the tooling makes adoption easy enough that teams choose to upgrade.
