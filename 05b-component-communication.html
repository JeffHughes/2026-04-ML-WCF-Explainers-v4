Page 5b — Communication & State

How Components Talk: Communication & State

The complete guide to inter-component communication, cross-domain messaging, and state management in a domain-driven architecture.

□ The Communication Model at a Glance

When we split a monolith into independent domains, the first question every architect asks is: “How do the pieces talk to each other?” The answer is four layers, each with clear rules and zero hidden coupling.

▣

Layer 1: Within a Component

Internal state is private. The component manages its own rendering. Nobody outside can reach in. Standard DOM attributes flow in, custom events flow out.

▢

Layer 2: Within a Domain

Components in the same domain share state through a lightweight reactive store. The store is scoped to the domain—invisible to other domains.

▭

Layer 3: Between Domains

Domains communicate via a typed message bus built on standard custom events. Fire-and-forget. No direct dependencies. Fully documented contracts.

◇

Layer 4: Frontend to Backend

Each domain calls its own BFF (Backend-for-Frontend). Typed data contracts define every request and response. No direct backend calls from components.

Three Rules That Govern Everything

  1. Components communicate via standard DOM: attributes in, events out. No framework-specific bindings. Pure HTML and JavaScript.
  2. Domains communicate via a lightweight message bus: typed custom events on a shared event target. Explicit, documented, decoupled.
  3. Frontend-to-backend uses standardized data contracts via BFF: each domain owns its data layer. No cross-domain backend calls.
Figure 5b-1: The four-layer communication model — from component internals to backend services
Industry Standard: Layered Communication Layered communication boundaries are the foundation of every successful micro-frontend architecture. Google, IKEA, Spotify, and Zalando all use this exact pattern: component isolation at the bottom, event-driven messaging between domains, and typed contracts to the backend.

▣ Within a Component: Attributes and Events

The foundational communication pattern is the simplest one: a single Web Component receives data through HTML attributes and emits behavior through custom DOM events. This is how the web has worked since day one.

Inputs: HTML Attributes and JavaScript Properties

Every Web Component declares the attributes it accepts. Parents set these attributes in HTML—no special syntax, no framework binding language, just standard markup:

<ml-product-card
  sku="ABC123"
  name="Exam Gloves, Nitrile"
  qty="500"
  in-stock="true">
</ml-product-card>

Internally, the component observes attribute changes via attributeChangedCallback—the W3C-standard lifecycle hook. When a parent updates an attribute, the component re-renders. No virtual DOM diffing, no zone detection, no digest cycle. Just a direct callback.

Outputs: Custom DOM Events

When the user interacts with the component—clicks a button, submits a form, selects an option—the component dispatches a standard CustomEvent:

this.dispatchEvent(new CustomEvent('add-to-cart', {
  bubbles: true,
  composed: true,
  detail: { sku: 'ABC123', qty: 1, unitPrice: 12.50 }
}));

The parent listens with addEventListener—the same API every developer has used for decades. No framework-specific event binding. No @Output() decorators. No onChange props. Pure DOM.

Slots: Projected Content

Web Components use the <slot> element to let parents project content into designated areas:

<ml-product-card sku="ABC123">
  <div slot="actions">
    <ml-add-to-cart-btn sku="ABC123"></ml-add-to-cart-btn>
    <ml-save-for-later-btn sku="ABC123"></ml-save-for-later-btn>
  </div>
</ml-product-card>

This is how we compose components without tight coupling. The product card doesn’t need to know about the add-to-cart button’s implementation. It just provides a slot for actions. Any component can be projected into that slot.

Figure 5b-2: Web Component anatomy — attributes and slots flow in, custom events flow out
Industry Standard: Attributes In, Events Out The attributes-in/events-out pattern is the W3C-endorsed communication model for Web Components. It mirrors how native HTML elements have always worked—an <input> accepts a value attribute and dispatches a change event. Web Components simply extend this pattern to custom elements.
Framework Agnostic by Design Because this pattern uses only standard DOM APIs, any framework can host these components internally. A Web Component could use WCFramework internally, or any other rendering approach. The outside world never needs to know—it only sees attributes, events, and slots.

▢ Within a Domain: Shared State

Individual components handle their own rendering, but components within the same domain often need to share business state. The Cart page needs to show the same cart items in both the line-item list and the summary totals. This is where the domain store comes in.

The Domain Store Pattern

Each domain has a lightweight reactive store built on the standard observer pattern:

  • Single source of truth for domain-specific business state
  • Observable slices: components subscribe only to the state they care about
  • Reactive updates: when state changes, subscribed components re-render automatically
  • Domain-scoped: the store is private to the domain—other domains cannot read or write to it
  • No global state library: no Redux, no NgRx, no MobX. A simple observable pattern with typed interfaces.

Example: The Cart Domain Store

// cart-store.ts — domain-scoped reactive store
interface CartState {
  items: CartItem[];
  subtotal: number;
  itemCount: number;
  lastUpdated: number;
}

class CartStore {
  private state: CartState = { items: [], subtotal: 0, itemCount: 0, lastUpdated: 0 };
  private listeners = new Map<string, Set<Function>>();

  select(slice: keyof CartState) {
    // Returns an observable for a specific slice
  }

  dispatch(action: CartAction) {
    // Updates state and notifies only affected subscribers
  }
}

The ml-cart-summary component subscribes to subtotal and itemCount. The ml-cart-line-item component subscribes to items. When a user changes a quantity, the store updates and only the affected components re-render.

Industry Standard: Observable Store Pattern The reactive observable store is the standard state management pattern across the industry. Whether implemented as a simple EventTarget with typed events or a lightweight library, the pattern is the same: single source of truth, selective subscriptions, and unidirectional data flow. This is how every major platform manages domain state.
Deep Dive: Why Not a Global Store? The case for domain-scoped state

Global state management (a single store for the entire application) is the pattern most teams reach for by default. It works well in small applications. It becomes a nightmare at scale:

  • Invisible dependencies: Any component can read any state. A component in the Cart domain might accidentally depend on state from the Account domain. When Account refactors, Cart breaks—and nobody understands why.
  • Testing complexity: To test a single component, developers need to set up the entire global state tree. Mocking becomes a full-time job.
  • Deployment coupling: If two domains share a global store, they must deploy together to keep the store schema in sync.
  • Performance: Global stores with 84 MFEs subscribing to the same state tree mean unnecessary re-renders across the entire application on every state change.

Domain-scoped stores eliminate all of these problems. Each domain owns its state. The boundary is clear and enforced.

▭ Between Domains: The Message Bus

Domains are independent. They have separate repositories, separate deployments, separate stores. But sometimes they need to coordinate. When a user adds an item to the cart, the mini-cart badge in the header needs to update. When checkout confirms an order, the account domain needs to refresh the order history.

The answer is a lightweight message bus: a shared event target where domains publish and subscribe to typed events.

How It Works

// message-bus.ts — the entire cross-domain communication layer
const bus = new EventTarget();

// Cart domain publishes
bus.dispatchEvent(new CustomEvent('cart-updated', {
  detail: { itemCount: 5, subtotal: 62.50 }
}));

// Mini-Cart domain subscribes
bus.addEventListener('cart-updated', (e: CustomEvent) => {
  updateBadge(e.detail.itemCount);
});

That’s it. The entire cross-domain communication mechanism is a standard EventTarget with typed CustomEvent payloads. No library, no framework, no magic.

The Rules

  • Fire-and-forget: Publishers don’t know or care who subscribes. Subscribers don’t know who published.
  • No shared state objects: Events carry data in their payload. There is no shared mutable object between domains.
  • Typed contracts: Every event has a documented name and payload schema, registered in the App Catalog.
  • No request/response: The bus is for notifications, not queries. If Domain A needs data from Domain B, it calls its own BFF, which orchestrates backend calls.
  • Loose coupling: If the Cart domain is removed, the Mini-Cart simply stops receiving events. Nothing crashes. Nothing throws.

Industry Standard: Event-Driven Architecture Event-driven architecture is the industry standard for decoupled communication in distributed systems. From Apache Kafka at LinkedIn to AWS EventBridge at Amazon, every large-scale platform uses publish/subscribe messaging to decouple producers from consumers. Our message bus applies the same principle at the frontend layer.
Deep Dive: Add-to-Cart Flow Across Domains Tracing a user action through the entire system

Let’s trace what happens when a user clicks “Add to Cart” on a product detail page. This single user action touches three domains and four communication layers:

  1. Layer 1 (Component): The ml-add-to-cart-btn dispatches a CustomEvent('add-to-cart', { detail: { sku, qty } })
  2. Layer 2 (Domain Store): The Cart domain’s event handler catches the event, dispatches an ADD_ITEM action to the Cart Store, which recalculates totals
  3. Layer 3 (Message Bus): The Cart Store publishes a cart-updated event to the message bus with the new itemCount and subtotal
  4. Layer 3 (Message Bus): The Mini-Cart domain receives cart-updated and updates the badge count in the header
  5. Layer 4 (BFF): The Cart Store calls the Cart BFF to persist the updated cart to the backend

Five steps. Three domains. Four layers. Zero direct dependencies between domains. If the Mini-Cart domain is unavailable, the cart still works. If analytics is down, the user’s experience is unaffected.

◇ Frontend to Backend: BFF Data Contracts

The final communication layer connects the frontend to backend services. Every domain has its own BFF (Backend-for-Frontend)—a thin service layer that sits between components and backend microservices.

Why a BFF?

Components never call backend services directly. Instead, they call their domain’s BFF, which handles:

  • Aggregation: A single BFF call can fan out to multiple backend services and combine the results
  • Authorization: The BFF validates tokens and enforces access rules before forwarding requests
  • Response shaping: The BFF transforms backend responses into exactly the shape the frontend needs—no over-fetching, no under-fetching
  • Caching: The BFF can cache frequently-accessed data to reduce backend load and improve response times
  • Versioning: BFF contracts are versioned. The App Catalog scans them. Breaking changes are detected automatically before deployment.

The Data Contract

// catalog-bff-contracts.ts
interface ProductDetailRequest {
  sku: string;
  includeReviews?: boolean;
  includeRelated?: boolean;
}

interface ProductDetailResponse {
  sku: string;
  name: string;
  description: string;
  price: { amount: number; currency: string };
  images: { url: string; alt: string }[];
  availability: 'in-stock' | 'low-stock' | 'out-of-stock';
  reviews?: ReviewSummary;
  related?: ProductCard[];
}

These contracts are the only coupling between frontend and backend. As long as the contract is honored, either side can refactor freely. The backend team can rewrite their service in a different language. The frontend team can swap component implementations. The contract is the boundary.

Figure 5b-6: BFF request flow — component requests data via store, BFF aggregates from multiple backend services, returns shaped response
Industry Standard: Backend-for-Frontend Pattern The BFF pattern was pioneered by Sam Newman at ThoughtWorks and is now the standard approach at Netflix, Airbnb, SoundCloud, and hundreds of enterprise platforms. It solves the fundamental mismatch between what backend services provide and what frontend components need, without creating a monolithic API gateway.

■ State Boundaries: What’s Shared, What’s Private

Clear state boundaries prevent the invisible coupling that kills monoliths. Every piece of state in the system has a defined scope, mechanism, and owner.

State TypeScopeMechanismExample
Component-Private Single component Internal class properties, Shadow DOM Dropdown open/closed, active tab index, scroll position, animation state
Domain-Scoped All components in one domain Domain reactive store (observer pattern) Cart items and totals, search results, user profile fields
Cross-Domain Published events only Message bus (typed custom events) Cart item count for badge, order confirmation for account refresh
URL State Current route Shell router (path params, query strings) /catalog/gloves?page=2&sort=price, /account/orders/ORD-9923
Session State Entire application Shell-managed, passed as attributes to domains Auth token, user identity, locale, feature flags

The Golden Rule

State Flows Down, Events Flow Up The shell passes session state down to domains as attributes. Domains pass domain state down to components as attributes. Components dispatch events up. Domains publish events to the bus. Nobody reaches across boundaries to read someone else’s state directly.

Visual: State Boundaries in the Architecture

This diagram shows exactly where each type of state lives and how the boundaries are enforced:

State Boundary Visualization Detailed architecture diagram

The shell owns session state (auth, locale, feature flags) and passes it down as attributes to each domain. Each domain owns its business state in an isolated store. Components own their rendering state privately in Shadow DOM. Cross-domain communication happens exclusively through the message bus—never through shared objects.

  • ▣ Shell layer: Manages URL routing, session state, and the message bus instance
  • ▢ Domain layer: Each domain has an isolated store. The store reads from the BFF and publishes events to the bus.
  • □ Component layer: Components subscribe to their domain store and manage internal rendering state privately

The key insight: no state object is ever shared between domains. Domains share events (data copies), not references. This means Domain A can never hold a stale reference to Domain B’s internal state. Every cross-domain interaction is explicit, typed, and documented.

▸ Why This Works Better Than Global State

The monolith pattern is global state: one giant store, every component can read and write anything. It works fine with 5 components. It becomes unmanageable with 84 micro frontends.

The Problem with Global State at Scale

AspectGlobal State (Monolith)Domain-Scoped State (Target)
Dependencies Invisible. Any component can read any state slice. Changes ripple unpredictably. Explicit. Components only access their domain store. Cross-domain uses documented events.
Testing Must mock the entire global state tree to test one component. Mock only the domain store. Isolated, fast, reliable tests.
Deployment Store schema changes require synchronized deployments across all consumers. Domain store changes deploy independently. Event contracts are versioned.
Performance Every state change can trigger re-renders in all 84 MFEs that subscribe. State changes only affect components in the owning domain. Other domains are untouched.
Debugging “Who changed this state?” could be any of 84 MFEs. Good luck. State changes come from within the domain or from documented bus events. Clear audit trail.
Onboarding New developers must understand the entire state tree to work on any feature. New developers learn their domain’s store and its event contracts. Scoped, manageable.
Industry Standard: Bounded Context Communication Domain-Driven Design (DDD) calls this principle “Bounded Contexts.” Each domain is a bounded context with its own internal model. Communication between contexts happens through well-defined contracts—never through shared internal state. This is the pattern used by Amazon’s service teams, Spotify’s squads, and every successful microservices architecture.

The Bottom Line

Explicit Is Always Better Than Implicit In a domain-scoped architecture, if Cart needs to know something from Checkout, there is a documented event for it in the App Catalog. The contract is typed. The payload is versioned. The subscribers are listed. There is no hidden coupling, no mystery dependencies, no “it worked on my machine.” Every communication path is visible, testable, and governable.
Deep Dive: Migration Path from Global to Domain State How to get there incrementally

There’s no need to rip out global state on day one. The migration is incremental:

  1. Identify domain boundaries: Map which state slices belong to which domain. The domain map (next page) defines these boundaries.
  2. Extract domain stores: For each domain, create a scoped store that owns its state slices. The global store becomes a thin proxy that delegates to domain stores.
  3. Replace direct reads with events: Where domains currently read each other’s state directly, introduce message bus events. The consuming domain subscribes to the event and maintains its own local copy of the data it needs.
  4. Remove the global store: Once all state is owned by domain stores and all cross-domain communication uses the bus, the global store can be deleted. Each domain is fully independent.

Each step is independently valuable. Even step 1 alone gives us a clear picture of state ownership that we don’t have today.
