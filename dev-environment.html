Day in the Life: The Developer Experience

What a developer’s morning-to-merge looks like in the evolved platform — from opening a browser to shipping production code in under 30 minutes.

□ The Web IDE: Zero Setup, Instant Start

The first thing that changes in the evolved platform is where we code. There is no local setup. No “clone the repo, install dependencies, configure environment variables, pray the build works.” Instead, we open a URL.

The platform provides a browser-based VS Code environment — identical to vscode.dev / GitHub Codespaces — hosted at ide.medline.com. Every developer gets a cloud workspace pre-configured with:

▣

Pre-Cloned Repositories

Our domain repo is already cloned. Our branch from yesterday is still there. No git clone, no waiting for a 2 GB monolith to download.

▢

Dependencies Installed

Node modules, build tools, type definitions — all pre-installed in the cloud workspace image. npm install happens once when the image is built, not every time we start.

▩

Platform Plugin Pre-Loaded

The Medline Platform Plugin is pre-installed and configured. Every shift-left check — lint, format, scan, test, validate — is active from the moment we open the editor.

◇

Live Preview Available

A built-in browser preview renders our components in isolation. Change code, save, see the result instantly. No manual server start, no port configuration.

Why Browser-Based?

Browser-based development eliminates the “works on my machine” problem entirely. Every developer works in an identical environment. Onboarding drops from 2 days of setup to 2 minutes of opening a URL. Security is centralized — source code never touches a local disk. And when a developer’s laptop breaks, they log in from another device and continue exactly where they left off.

Industry Standard: Cloud Development Environments GitHub Codespaces, Gitpod, AWS Cloud9, and Google Cloud Shell are all production-grade cloud IDE platforms used by thousands of engineering organizations. Shopify moved 11,000 developers to cloud environments in 2022. Stripe, Uber, and Slack have all adopted similar approaches. The pattern is proven at massive scale.

▣ The Platform Plugin: Shift-Left Everything

The heart of the developer experience is a single VS Code extension — the Medline Platform Plugin. It handles everything that used to happen 20 minutes later in CI. Now it happens on save, in the editor, before the code ever leaves the developer’s hands.

What the Plugin Does on Every Save

When a developer presses Ctrl+S, the plugin runs a pipeline of checks in parallel — all completing in under 2 seconds:

▣

Auto-Lint

ESLint with domain-specific rules. Framework conventions, naming standards, import restrictions — all enforced instantly. Violations appear as inline squiggles with auto-fix suggestions.

▢

Auto-Format

Prettier runs on every save. Formatting debates are over. Tabs vs. spaces, semicolons, trailing commas — all decided once, enforced everywhere, automatically.

▩

Security Scanning

Dependency audit checks for known vulnerabilities. SAST (Static Application Security Testing) scans the saved file for injection risks, hardcoded secrets, and insecure patterns. Findings appear inline.

◇

Accessibility Validation

axe-core rules run against the component preview. Missing ARIA labels, insufficient color contrast, keyboard navigation gaps — all flagged before commit.

■

Unit Test Runner

Affected tests run on save. Results appear inline — green checkmarks next to passing assertions, red markers on failures. No manual npm test required.

◆

Contract Validation

Component contracts (attributes-in/events-out) are checked against the spec in the App Catalog. API contracts (BFF responses) are validated against declared schemas. Mismatches are caught instantly.

▭

Bundle Size Guard

The plugin calculates the component’s bundle size impact. If the component exceeds its budget (e.g., 15 KB for a product card), a warning appears immediately. No waiting for CI to tell us.

□

All Before Commit

Every check above completes in the editor. The developer sees instant, actionable feedback. Problems are fixed in seconds, not discovered 20 minutes later in a failed CI build.

The platform plugin pushes every possible check as far left as it can go — into the editor, at save time, before the code leaves the developer’s hands.

▢ Git Routing: Smart Commits, Automatic Flow

In the evolved platform, the plugin knows which domain repository we’re in. When we commit, it handles everything — commit format validation, repository routing, branch management, and PR creation.

Conventional Commits — Enforced, Not Requested

Every commit message must follow the Conventional Commits specification. The plugin enforces this at commit time — not with a CI check 20 minutes later, but immediately in the editor:

feat: add hover preview to product card       ← minor version bump
fix: correct price rounding on bulk orders     ← patch version bump
chore: update dev dependencies                 ← no version bump
BREAKING CHANGE: remove legacy cart endpoint   ← major version bump

If the commit message doesn’t match the pattern, the commit is blocked with a helpful error explaining the required format. No ambiguity. No “please remember to use conventional commits.” The system enforces it.

Automatic Repository Routing

The plugin knows which domain we’re working in based on the workspace configuration. On commit:

  • Changes are routed to the correct domain repository automatically. No manual git remote add, no remembering which repo owns which code.
  • Trunk-based development with short-lived feature branches. No long-lived branches, no complex branching strategies, no merge conflicts that span weeks.
  • PR templates are auto-filled with the contract diff (what attributes/events changed), test coverage delta, and bundle size delta. Reviewers see exactly what matters.

▩ Semantic Versioning — Fully Automated

No human ever manually bumps a version number. The platform handles it entirely based on conventional commit types.

How It Works

  1. Commit type determines version bump: feat: → minor, fix: → patch, BREAKING CHANGE: → major
  2. Version is calculated at merge time, not at commit time. Multiple commits in a PR are analyzed together. The highest-impact commit type wins.
  3. CHANGELOG is auto-generated from the conventional commit messages. Grouped by type (Features, Fixes, Breaking Changes).
  4. App Catalog registry is updated automatically. The new version appears in the catalog within seconds of merge.
  5. Import maps are updated. Consumers using latest get the new version. Consumers pinned to a specific version are unaffected.

■ The Outer Loop: Merge to Production

The outer loop is everything that happens after a PR is approved — from merge to production. In the evolved platform, this is fully automated and takes under 10 minutes.

Figure DEV-5: The outer loop — from PR approval to production deployment with canary testing and automatic rollback

Outer Loop Steps

  1. PR Approved → Merge: Teammate reviews the auto-generated PR with contract diff, coverage delta, and bundle size delta. Approves. Merged to main.
  2. Auto-Version: Semantic version is calculated from conventional commits. Git tag created. No human involvement.
  3. Auto-CHANGELOG: CHANGELOG.md is generated from commit messages, grouped by type. Committed automatically.
  4. Production Build: Domain-specific build runs. Only the changed domain is built — not the entire monolith.
  5. Canary Deploy: New version is deployed to 5% of traffic. Real users validate the change with real data.
  6. Smoke Tests: Automated smoke tests verify critical paths. If any fail, the canary is rolled back automatically.
  7. Full Production Deploy: If smoke tests pass, traffic shifts to 100%. The old version remains available for instant rollback.
  8. App Catalog Updated: The new version is registered. Import maps are updated. Consumers see the new version immediately.

A developer on the Catalog domain team needs to add a hover preview feature to the ml-product-card component. This adds a new preview-url attribute to the component contract.

Key Principles

▣

Shift Left

Every check that can run in the editor, runs in the editor. CI validates — it doesn’t discover. By the time code reaches CI, the developer already knows it’s clean.

▢

Domain Isolation

CI runs only domain-specific tests. Deploys only the changed domain. No monolith build. No cross-team coordination. Fast, focused, independent.

▩

Full Automation

From version bump to CHANGELOG to canary deploy to App Catalog update — every step after merge is fully automated. Zero manual intervention.

◇

Safe Deployments

Canary releases catch issues before they affect all users. Automatic rollback on smoke test failure. Blue-green infrastructure means instant rollback is always available.

Deep Dive: DORA Metrics Impact How this maps to elite engineering performance

The DORA (DevOps Research and Assessment) framework identifies four key metrics that distinguish elite engineering teams:

  • Deployment Frequency: Elite = multiple times per day. Our target: each pod deploys independently, multiple times per day.
  • Lead Time for Changes: Elite = less than one hour. Our target: 35 minutes from code to production.
  • Mean Time to Recovery (MTTR): Elite = less than one hour. Our target: instant rollback via blue-green + canary.
  • Change Failure Rate: Elite = 0–15%. Our target: shift-left validation catches issues before deployment; canary catches the rest.

The evolved developer experience is specifically designed to hit elite-level performance on all four DORA metrics.
