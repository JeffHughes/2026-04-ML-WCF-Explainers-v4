Component Strategy

The App Catalog: Our Single Source of Truth

A central registry that knows every component, every version, every dependency, and every breaking change — before it reaches production

Why We Need a Catalog

With 84+ micro frontends, each independently deployed, each consuming shared components at their own pinned versions, visibility becomes our most critical capability. Without a catalog, we are flying blind.

The App Catalog is the central registry that answers every question about our component ecosystem in real time:

  • What components exist? — Every ml- component, every version, every published artifact
  • Who uses what? — Which domains consume which components at which version
  • What changed? — Version history, attribute changes, event contract modifications
  • What will break? — Breaking change detection before deployment, not after
  • What needs attention? — Domains running outdated versions, components with known issues
“Without a catalog, we’re flying blind with 84 micro frontends.” Every enterprise that operates micro frontends at scale — Zalando, IKEA, Spotify — runs a component registry. The catalog is not optional infrastructure. It is the nervous system of the platform.

Catalog Architecture

The App Catalog is composed of three primary subsystems: the Registry that stores component metadata, the Scanner that discovers and validates components automatically, and the Dashboard that provides real-time visibility to every team.

App Catalog Architecture — Scanner discovers components from multiple sources; Registry stores contracts; Dashboard provides visibility; Diff engine prevents breaking changes

Component Dependency Graph

The catalog maintains a live dependency graph showing which domains share which components and at which versions. This graph is the foundation for breaking change detection and upgrade planning.

Component Dependency Graph — Catalog sees on v2.1 of ml-header while others are on v2.3; Cart has not yet adopted ml-search-bar v3.0
Industry Standard: Component Registries Backstage (Spotify’s open-source developer portal), Bit.dev, and Chromatic all provide component catalog functionality. The pattern of a central registry with automated scanning and dependency tracking is well-established across the industry. Medline’s App Catalog applies this pattern specifically to CDN-distributed Web Components.

What Gets Tracked

Component Metadata Identity, ownership, classification

Every component in the registry carries a complete metadata profile:

  • Tag name: ml-product-card
  • Owning domain: Catalog
  • Owning team: Catalog Frontend
  • Classification: Shared / Domain-specific / Internal
  • Technology: WCFramework / React (wrapped) / Hybrid
  • Published versions: Complete version history with dates and changelogs
  • CDN paths: All version URLs for each published artifact
  • Bundle size: Raw and gzipped sizes per version
  • Dependencies: Other ml- components this component depends on
Contract Definitions Attributes, events, slots, properties

Every component’s public interface is documented and versioned:

  • Attributes: Name, type, default value, required/optional, description
  • Properties: Name, type, description (for complex data that can’t be stringified as attributes)
  • Events: Event name, detail payload schema, when it fires
  • Slots: Slot name, expected content type, fallback behavior
  • CSS Custom Properties: Theme tokens the component exposes for consumer customization

Contract changes between versions are automatically diffed. Any change to a public attribute, event, or slot is classified as additive (safe), modified (review), or removed (breaking).

Consumer Map Who uses what, at which version

For every component, the catalog knows exactly who consumes it:

  • Domain name: Which deployed domain includes this component
  • Pinned version: The exact version the domain’s import map points to
  • Usage count: How many pages within the domain render this component
  • Last scan date: When the scanner last verified this mapping
  • Upgrade availability: Whether a newer version exists and what changed

This map is what powers breaking change detection. When a component author modifies an attribute, the catalog instantly knows which domains depend on that attribute.

Breaking Change Detection Know before we deploy

The catalog’s contract diff engine compares every new version against its predecessor:

  • Removed attributes: Any consumer using this attribute will break — blocked by default
  • Changed attribute types: A string becoming a number can break consumers — flagged for review
  • Removed events: Any consumer listening for this event will break — blocked by default
  • Modified event payloads: Changed detail shape can break consumers — flagged for review
  • Removed slots: Any consumer projecting into this slot loses content — blocked by default

Breaking changes are not forbidden — they’re surfaced with full impact analysis. The author sees which domains are affected, which teams to notify, and what version the consumers can stay on while they adapt.

Automated Discovery

Components and API contracts are discovered automatically — not manually registered. The scanner runs continuously, ensuring the catalog is always accurate.

📦

CDN Artifact Scanning

When a component is published to CDN, the scanner extracts its metadata from the module: tag name, observed attributes, event dispatches, slot definitions.

🚧

CI Pipeline Events

Every component build pipeline emits a publish event. The scanner captures the event and updates the registry before the component is even available on CDN.

🌐

Production Page Scanning

A headless browser visits production pages periodically, inspecting the DOM for ml- custom elements, their versions (from import maps), and their usage patterns.

🔌

API Contract Extraction

Running API services are scanned for their data contracts. The catalog links frontend component data dependencies to the backend services that serve them.

Zero Manual Registration The catalog is populated entirely by automation. Teams don’t fill out forms. They don’t update spreadsheets. They build components, publish them, and the scanner takes care of the rest. If a component exists on CDN, it’s in the catalog.

Rollback & Version History

Every version of every component ever published remains on CDN and in the catalog. Rolling back to a previous version is a one-operation change:

  1. Open the domain’s import map configuration
  2. Change the version pointer: ml-product-card: "2.1.0" → ml-product-card: "2.0.4"
  3. Deploy the import map change (no rebuild, no retest of the component itself)
  4. The previous version is served immediately from CDN cache

The catalog’s version history shows every change between versions, so teams can understand exactly what they’re rolling back to and what they’re giving up.

Immutable Artifacts, Mutable Pointers The artifacts on CDN never change. Only the import map pointers change. This means rollback is instant, risk-free, and doesn’t require coordination with the component author. The old version is already deployed and cached.
