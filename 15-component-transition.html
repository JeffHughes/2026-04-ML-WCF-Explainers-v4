Component Strategy

Component Transition: React to HTML-First

A phased, zero-pressure path from React 17 to standards-based components — at every team’s own pace

The Transition Philosophy

This is not a rewrite mandate. This is not a “drop everything and migrate” directive. This is a four-phase transition where every phase is optional for every team, and the only forcing function is the team’s own assessment of what serves them best.

React 17 components work today. They will work tomorrow. They will work five years from now — wrapped in Web Component shells, served from CDN, independently versioned. The transition to HTML-first MV* patterns is about expanding what’s possible, not replacing what works.

The Core Promise No team will ever be forced to rewrite a working component. The system is designed so that React components and WCFramework components coexist indefinitely. The import map doesn’t care what’s inside the Shadow DOM.

The Four Phases

Phase 1: Wrap & Ship

Every existing React component gets wrapped in a Web Component shell. This is automated — a build-time transformation, not a manual rewrite. The React component stays identical inside. The outside world sees a standard ml- Custom Element.

  • React 17 internals remain untouched
  • Wrapper translates attributes → props, CustomEvents → callbacks
  • Shadow DOM encapsulates React’s CSS
  • Published to CDN with the standard version path
  • Existing tests continue to pass — nothing changed inside

Phase 2: New Components Go HTML-First

All new components are built with the WCFramework — an HTML-first MV* pattern where the starting point is HTML and CSS, not JSX and a virtual DOM. Developers who know HTML, CSS, and JavaScript can build production components on day one.

  • No JSX transpilation needed
  • No virtual DOM diffing overhead
  • No build-step complexity — the component IS the deliverable
  • Standard template literals, standard event listeners, standard DOM APIs
  • Smaller bundle sizes — no framework runtime shipped per component

Phase 3: Selective Rewrite

High-traffic, high-change components that benefit from smaller bundles and simpler maintenance are rewritten in WCFramework — at the team’s discretion and pace. This is optimization, not obligation.

  • Product cards, search results, and navigation components are prime candidates
  • Components that change weekly benefit from the simpler authoring model
  • Performance-critical components benefit from no virtual DOM overhead
  • The old React version stays on CDN as a rollback target

Phase 4: Long-Tail Stability

Low-traffic, low-change components (admin panels, account settings, legacy integrations) can stay React indefinitely. There is no Phase 5 that says “everything must be WCFramework.” If it works, it ships.

  • React compatibility wrappers are maintained alongside WCFramework components
  • Two component sets published yearly: one WCFramework, one React compatibility
  • Five-year-old components are first-class citizens — the import map treats them identically

Transition Timeline

Transition Timeline — Four overlapping phases; React compatibility extends indefinitely
Industry Standard: Incremental Migration Google’s migration from GWT to modern web components, Shopify’s transition from jQuery to web-native Polaris, and GitHub’s move from jQuery to Catalyst components all followed this same pattern: wrap first, build new with the target, selectively rewrite, and let the long tail live. Forced rewrites fail. Gradual transitions succeed.

Why HTML-First Matters

The argument for HTML-first development is not about technology preference. It’s about accessibility of the development experience to the widest possible talent pool.

“Patterns that require a PhD in build tooling to create a button component are not accessible to 80% of the world’s frontend developers.” When a component starts with an HTML template, a CSS stylesheet, and JavaScript event handlers — that’s a language every web developer on the planet already speaks.

Dual Publishing Model

Twice a year, the shared component team publishes two complete sets of every shared component:

WCFramework Component Set HTML-first, zero-runtime

The primary component set, built with the standards-based component framework:

  • No framework runtime shipped — just the component code
  • Smaller bundle sizes (typically 60-70% smaller than React equivalents)
  • Browser-native rendering performance
  • Standard debugging with browser DevTools
  • Recommended for all new development
React Compatibility Set For existing React-based domains

A fully maintained React wrapper set for teams that haven’t transitioned:

  • Identical visual output and behavior to the WCFramework set
  • React-friendly prop interfaces alongside standard attributes
  • Same CDN distribution, same versioning, same rollback capability
  • Published and supported indefinitely — no sunset date
  • Teams migrate when they choose, not when they’re forced
Version Synchronization How both sets stay aligned

Both component sets share the same version number and the same visual test suite:

  • Version 2.3.0 of the WCFramework set and 2.3.0 of the React set produce identical rendered output
  • Pixel-perfect visual regression tests run against both sets
  • API contract tests verify attribute/event/slot parity
  • If a bug is fixed in one set, the same fix is applied to the other

What Teams Choose, Teams Own

The transition is not managed top-down. Every team makes their own choices based on their own context. The platform’s job is to make every option easy, safe, and reversible.

🎯

Last Year’s Stack Is Fine

Components built last year work identically today. The import map doesn’t care when a component was authored.

⏳

Five Years Ago Is Fine

Legacy components wrapped in Web Component shells have the same CDN presence and the same deployment model as brand-new WCFramework components.

🔄

The Import Map Doesn’t Care

Whether the component is React inside or WCFramework inside, the import map resolves the same ml- tag to the same CDN path. The shell is agnostic.

🛡

Rollback Is Instant

Every transition step produces a new CDN artifact. The previous version is still there. Rollback is a version pointer change, not a rebuild.

The Anti-Pattern We’re Avoiding Mandating a company-wide rewrite from one framework to another is the single most common cause of multi-year platform stalls. Teams stop delivering features to rewrite working code. The new framework ships bugs the old one had already fixed. Morale drops. Deadlines slip. This transition model eliminates that entirely.
