Track: Component Author

Seven modules that take we from understanding Web Components to publishing production-ready ML- components used across every domain. We build the building blocks.

Track Overview

Component Authors build the shared ML- components that every domain team consumes. We own the component contract: attributes, events, slots, and CSS custom properties. Our components run inside Shadow DOM, encapsulated and isolated. When we publish, every domain that uses our component gets the update via the CDN.

This track has 7 modules. Expected completion: ~4 weeks.

Module 1: Component Anatomy

What we'll learn: The structure of a Custom Element—Shadow DOM, attributes, events, slots, and lifecycle callbacks.

Module content Custom Elements, Shadow DOM, lifecycle

Key Concepts

  • Custom Element registration — customElements.define('ml-my-component', MyComponent)
  • Shadow DOM — encapsulated DOM tree attached via this.attachShadow({mode: 'open'})
  • Lifecycle callbacks — connectedCallback, disconnectedCallback, attributeChangedCallback
  • Observed attributes — declare which attributes trigger change callbacks
  • Slots — named and default slots for content projection from the consumer

Hands-on Exercise

Create a new ml-info-card component from scratch. It should accept title and variant attributes, emit a card-click Custom Event, and expose a default slot for body content. Register it as a Custom Element.

Sign-off Criteria

  • Component registered with ml- prefix
  • Shadow DOM properly attached
  • Attributes observed and reflected
  • Custom Event dispatched with correct detail payload
  • Slot renders projected content

Module 2: Styling

What we'll learn: CSS custom properties, Shadow DOM style encapsulation, and how to ensure zero global CSS leakage.

Module content CSS custom properties, encapsulation, theming

Key Concepts

  • Shadow DOM encapsulation — styles inside Shadow DOM don't leak out; external styles don't leak in
  • CSS custom properties — the only styling hook: --ml-card-bg, --ml-card-radius, etc.
  • :host selector — style the component's outer element from within Shadow DOM
  • ::part() pseudo-element — expose named parts for limited external styling
  • Theme support — components respond to platform-level CSS custom properties for light/dark mode
  • No global styles — components must never import global stylesheets or modify document-level CSS

Hands-on Exercise

Extend our ml-info-card with full styling: define 5+ CSS custom properties for consumer theming, support light/dark mode via platform CSS variables, and verify zero style leakage using the style isolation test suite.

Sign-off Criteria

  • All styling uses CSS custom properties (no hardcoded colors)
  • Style isolation test passes (no leakage in or out)
  • Light/dark mode both render correctly
  • All CSS custom properties are documented in a comment block
Industry Standard: Shadow DOM Style Encapsulation Shadow DOM provides browser-native style encapsulation—no build tools, no CSS-in-JS runtime, no naming conventions to enforce. It's the W3C standard adopted by every major browser vendor. CSS custom properties as the theming API is the recommended pattern from the Web Components community.

Module 3: The Component Contract

What we'll learn: How to define and document the complete API surface of an ML- component: attributes, events, slots, and CSS custom properties.

Module content API surface, documentation, contract enforcement

Key Concepts

  • Attribute documentation — every attribute has a name, type, default value, and description
  • Event documentation — every Custom Event has a name, detail type, and when it fires
  • Slot documentation — every slot has a name and description of expected content
  • CSS custom property documentation — every styling hook has a name, default value, and description
  • Contract file — component.contract.json declares the full API surface; the pipeline validates it

Hands-on Exercise

Write the component.contract.json for our ml-info-card. Ensure every attribute, event, slot, and CSS custom property is documented. Run the contract validation tool to verify completeness.

Sign-off Criteria

  • Contract file is complete (all attributes, events, slots, CSS properties)
  • Contract validation tool passes with no warnings
  • Contract matches actual component implementation

Module 4: Publishing

What we'll learn: How to build, version, publish to CDN, and register our component in the App Catalog.

Module content Build, CDN, catalog registration, versioning

Key Concepts

  • Build process — single-file ES module output, tree-shaken, minified
  • CDN publish — built artifact is pushed to CDN with a semver-tagged URL
  • Catalog registration — the App Catalog records the new version, its contract, and its dependencies
  • Semver — breaking changes = major, new features = minor, fixes = patch
  • Consumer notification — domains using our component are automatically notified of new versions

Hands-on Exercise

Build our ml-info-card, publish it to the sandbox CDN, register it in the App Catalog, and verify it appears in the component browser with full documentation.

Sign-off Criteria

  • Component builds to a single ES module file
  • Published to CDN with correct semver tag
  • Registered in App Catalog with complete contract
  • Visible in the component browser with documentation

Module 5: WCFramework Internals

What we'll learn: The HTML-first MV* pattern that powers ML- components. Data binding, change detection, template rendering.

Module content HTML-first pattern, data binding, change detection

Key Concepts

  • HTML-first MV* pattern — templates are HTML with declarative bindings, not JSX or string interpolation
  • Data binding — one-way binding from data model to template; updates flow down, events flow up
  • Change detection — attribute changes trigger targeted re-renders of bound sections only
  • Template syntax — conditional rendering, list rendering, and slot composition using standard HTML patterns
  • Performance — fine-grained updates minimize DOM operations; no virtual DOM diffing overhead

Hands-on Exercise

Build an ml-data-table component using WCFramework internals: data binding for rows, conditional rendering for empty states, list rendering with keyed updates, and a sort event that triggers targeted re-render.

Sign-off Criteria

  • Component uses WCFramework template binding (no manual DOM manipulation)
  • Re-renders are targeted (verified via render count instrumentation)
  • Empty state and populated state both render correctly
  • Sort triggers efficient update (not full re-render)
Industry Standard: Standards-Based Component Framework Building on W3C Custom Elements and Shadow DOM means components work in any browser without a framework runtime. The HTML-first MV* pattern extends this with minimal, targeted data binding—no virtual DOM, no framework lock-in, no runtime dependency on React, Vue, or any other library.

Module 6: React Compatibility Wrapper

What we'll learn: How to publish a React-compatible version of our ML- component so existing React domains can consume it natively.

Module content React wrapper, prop mapping, event bridging

Key Concepts

  • React wrapper generator — an automated tool that creates a React wrapper from our component contract
  • Prop mapping — HTML attributes map to React props automatically
  • Event bridging — Custom Events map to React on* callback props
  • Ref forwarding — React refs forward to the underlying Custom Element
  • TypeScript types — the wrapper exports full TypeScript definitions generated from the contract

Hands-on Exercise

Generate a React wrapper for our ml-info-card. Use it in a sandbox React application. Verify props, events, and refs all work correctly. Publish the wrapper alongside the Web Component.

Sign-off Criteria

  • React wrapper generated from component contract
  • All props map correctly to attributes
  • All events map correctly to callback props
  • TypeScript types are complete and accurate
  • Wrapper published to CDN alongside the Web Component

Module 7: Documentation

What we'll learn: How to create a component playground, write usage examples, and publish API documentation to the catalog.

Module content Playground, examples, API docs

Key Concepts

  • Component playground — interactive demo page where consumers can try our component with different attributes and slot content
  • Usage examples — copy-paste code snippets for common use cases (basic, themed, with slots, event handling)
  • API documentation — auto-generated from component.contract.json plus hand-written descriptions
  • Changelog — version history showing what changed, what broke, and what was added
  • Search integration — component documentation is indexed in the catalog's search

Hands-on Exercise

Create a complete documentation page for ml-info-card: interactive playground, 4 usage examples, full API reference, and a changelog entry. Publish to the catalog and verify it appears in search results.

Sign-off Criteria

  • Playground renders with interactive attribute controls
  • At least 4 usage examples (basic, themed, slots, events)
  • API docs match the component contract exactly
  • Component is findable via catalog search
Industry Standard: Living Documentation Component documentation generated from contracts and published alongside the component is the industry standard for design systems at scale. Living docs stay in sync with the code because they're generated from the same source of truth. Teams like Google (Material), Microsoft (Fluent), and Salesforce (Lightning) all follow this pattern.
