Certification: Prove It, Ship It

Complete our track, pass the automated exercises, earn our badge, get production access. No manager approval needed. The system validates competency.

How Certification Works

Certification is the outcome of completing our training track. It's not a test we schedule. It's not a meeting we attend. It's the natural result of passing all our module exercises through the automated pipeline.

📚

Step 1: Complete Modules

Work through all modules in our track (Domain Developer: 6, Component Author: 7, Platform Engineer: 6). Each module has a hands-on exercise.

🔬

Step 2: Pass Exercises

Each exercise is validated by the automated pipeline. The same seven gates that guard production code guard our exercises. Pass = module complete.

🏅

Step 3: Earn Our Badge

When all modules in our track are complete, the system automatically issues our certification badge. No waiting. No approval chain.

🚀

Step 4: Production Access

Our badge grants production deploy access for our role. Domain Developer badge = deploy our domain. Component Author badge = publish components. Platform Engineer badge = manage infrastructure.

Certification Flow

Certification flow: complete modules → system validates → badge issued → production access granted
Industry Standard: Competency-Based Access Control Competency-based access control removes bottlenecks and ensures only qualified developers touch production. Instead of relying on job titles or manager approvals, access is granted based on demonstrated skills. This is the model used by leading DevOps organizations worldwide.

Badge Types

Three badges correspond to three training tracks. Each badge grants specific production privileges.

BadgeTrackProduction AccessVisible In

🟪 Component Author 7 modules Publish ML- components to CDN & catalog App Catalog — component ownership
🟧 Platform Engineer 6 modules Manage shell, import maps, catalog, CDN App Catalog — platform team roster
Can a developer earn multiple badges? Multi-track certification

Yes. A developer can complete multiple tracks and earn multiple badges. Common combinations:

  • Domain Developer + Component Author — for senior developers who both build features and contribute shared components
  • Component Author + Platform Engineer — for tech leads who build components and maintain infrastructure
  • All three — for architects and principal engineers who need full platform access

Each track is independent. Completing one does not require completing another.

Badges in the App Catalog

Certification badges are not just status symbols. They're functional. The App Catalog uses badges to enforce access control.

How badge-based access works Functional access control
  • Deploy pipelines check for the appropriate badge before allowing a production deployment
  • Component publish pipelines check for the Component Author badge
  • Infrastructure changes require the Platform Engineer badge
  • The App Catalog dashboard shows each developer's badges and what they're authorized to deploy
  • Team leads can see at a glance which team members are certified and which are still in training

This is not honor-system access control. The pipeline enforces it. A developer without the right badge literally cannot trigger a production deployment.

Industry Standard: Automated Access Governance Automated, role-based access control tied to demonstrated competency is the industry standard for regulated and high-scale environments. It satisfies audit requirements, eliminates privilege creep, and ensures every production deployment is performed by someone who has proven they know what they're doing.

Recertification

When the core cadence annual update happens, a short recertification module is published. This ensures everyone stays current with platform changes.

How recertification works Annual update + short module

The Process

  1. Annual core update ships — the platform engineering team releases the new core version
  2. Recert module published — a focused module covering what changed, what's new, and what developers need to know
  3. Developers complete the recert — typically 2–4 hours, covering only the delta (not re-doing the full track)
  4. Badge refreshed — badge is updated to the current year's version
  5. Grace period — developers have 30 days from the annual update to complete recertification. During the grace period, existing access continues.

What If I Don't Recertify?

After the 30-day grace period, production deploy access is suspended until recertification is complete. Staging access remains. This is a safety measure, not a punishment—the platform has changed, and we need to ensure everyone operating on it understands the changes.

Recertification: annual update triggers a short module. Complete within 30 days to keep production access.

No Manager Approval Needed

This is worth repeating. The entire certification and access control system is automated. There is no step where a developer waits for a manager to click “approve.”

The Guarantee If we complete the exercises and the pipeline validates our work, we earn our badge. Period. No politics. No waiting. No subjectivity. The system validates competency, not relationships.
Why remove managers from the sign-off loop? Scaling across timezones

Three reasons:

  1. Timezone scaling — with teams across multiple timezones, waiting for a manager in a different timezone creates 24-hour delays. The pipeline runs 24/7.
  2. Objectivity — automated gates are objective and reproducible. Two developers with the same skill level get the same result every time.
  3. Manager time — managers should spend time on leadership, mentoring, and strategic work—not reviewing training exercises that a pipeline can validate better.

Managers still see all certification data in the App Catalog. They still know who's certified and who isn't. They just don't need to be in the critical path.

Industry Standard: Self-Service Developer Platforms Self-service developer platforms—where developers can onboard, train, certify, and deploy without waiting for manual approvals—are the defining pattern of modern platform engineering. Gartner, Forrester, and ThoughtWorks Technology Radar all identify this as a key enabler of developer productivity at scale.
