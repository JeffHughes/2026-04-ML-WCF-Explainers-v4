Migration Mechanics

Cart, Mini-Cart, and Checkout Domains

Three tightly related but independently deployable domains — connected by events, separated by responsibility

Why Three Separate Domains?

Cart, Mini-Cart, and Checkout handle different user journeys, carry different risk profiles, and ship at different cadences. Bundling them together means a checkout bug blocks cart fixes and a mini-cart tweak risks breaking the payment flow. Separating them gives each team full autonomy.

🛒

Cart Domain

Full cart page, line item management, quantity changes, promotions, saved carts, bulk ordering. Changes weekly as merchandising teams adjust promo rules.

📦

Mini-Cart Domain

Header overlay, quick-add confirmation, item count badge, recently added items. Tiny surface area, ships independently as part of the shared header experience.

💳

Checkout Domain

Multi-step flow: shipping, payment, review, confirmation. Address validation, payment gateway integration, order submission. Highest-risk domain — changes require extra review and canary deployments.

Dimension Cart Mini-Cart Checkout
Release cadence Weekly As needed Bi-weekly with canary

Test requirements Unit + integration Unit + visual Unit + integration + E2E + PCI compliance

Cross-Domain Communication

These three domains talk to each other through a lightweight event bus using browser-native CustomEvent API. No direct imports, no shared state stores, no tight coupling. Each domain publishes events and subscribes to events — that’s the entire contract.

Cart / Mini-Cart / Checkout domain interactions — all communication via browser-native CustomEvent bus

Event Catalog

Every cross-domain interaction is a named event with a typed payload. Here are the events that flow between Cart, Mini-Cart, and Checkout:

Event Name Publisher Subscribers Payload
ml:add-to-cart Catalog (PDP) Cart { sku, qty, variant? }
ml:cart-updated Cart Mini-Cart, Checkout { itemCount, subtotal, items[] }
ml:remove-from-cart Cart, Mini-Cart Cart { lineItemId }
ml:proceed-to-checkout Cart Checkout { cartId }
ml:order-placed Checkout Cart, Mini-Cart { orderId, orderTotal }
ml:open-mini-cart Catalog, Cart Mini-Cart {} (no payload)
💡 No Shared State Store Events replace shared Redux/state stores. Each domain manages its own internal state. The event bus is the only coupling. This means we can swap out the Cart domain’s internal state management without affecting Mini-Cart or Checkout at all.

The Mini-Cart Domain

The Mini-Cart is the smallest domain — just 2–3 components. It lives in the header and shows a quick summary of the user’s cart without navigating to the full cart page.

Mini-Cart Components

  • <ml-mini-cart> — Slide-out overlay showing last 3 items, subtotal, and “View Cart” / “Checkout” buttons
  • <ml-cart-badge> — Item count badge displayed on the cart icon in the header
  • <ml-add-confirm> — Quick confirmation toast when an item is added from PDP

Why It’s Separate

The Mini-Cart is part of the header experience, which is rendered by the shell on every page. It subscribes to ml:cart-updated events and updates reactively. It never calls the Cart BFF directly — it gets all its data from events. This means:

  • Mini-Cart can deploy without touching the Cart domain
  • Cart can change its internal data model without affecting Mini-Cart
  • Mini-Cart works even if the Cart BFF is temporarily slow — it shows the last known state from events
Industry Standard: Event-Driven Micro Frontends Event-driven communication between micro frontends is the pattern recommended by Martin Fowler’s micro frontend architecture, the Thoughtworks Technology Radar, and implemented by Zalando, IKEA, and Spotify. It eliminates shared state and enables true independent deployment.

The Checkout Domain

Checkout is the highest-risk domain in the platform. It handles payments, address validation, tax calculation, and order submission. Every change to Checkout goes through additional review gates and canary deployment.

Checkout Flow as a State Machine

The checkout flow is modeled as a finite state machine. Each step has explicit entry conditions, exit conditions, and error states. This makes the flow predictable, testable, and resilient:

Checkout state machine — every transition is explicit, every error state has a recovery path
Checkout Components 15–20 components across the checkout flow

Shipping Step

  • <ml-checkout-shipping> — Shipping step container
  • <ml-address-form> — Address entry with autocomplete and validation
  • <ml-address-book> — Select from saved addresses (data from Account domain)
  • <ml-shipping-options> — Available shipping methods with estimated delivery dates

Payment Step

  • <ml-checkout-payment> — Payment step container
  • <ml-payment-form> — Credit card entry (PCI-compliant, tokenized)
  • <ml-saved-payments> — Select from saved payment methods
  • <ml-billing-address> — Billing address (defaults to shipping, can override)
  • <ml-po-number> — Purchase order number entry (enterprise B2B feature)

Review & Confirmation

  • <ml-order-review> — Full order summary before submission
  • <ml-order-items> — Read-only line item display
  • <ml-order-totals> — Subtotal, tax, shipping, total breakdown
  • <ml-place-order> — Submit button with loading state and idempotency guard
  • <ml-order-confirmation> — Success page with order number and tracking info

Handling Cross-Domain Data Dependencies

Checkout depends on data from other domains, but it never imports their code directly. Instead, it reads from its own BFF, which aggregates the required data:

Data Needed Source Domain How Checkout Gets It
Cart items and pricing Cart Checkout BFF reads cart state via internal API
Saved addresses Account Checkout BFF fetches from Account service
Saved payment methods Account Checkout BFF fetches from Account service
Product availability Catalog Checkout BFF re-validates with Inventory service
Shipping rates Logistics Checkout BFF calls Logistics rate calculator
Tax calculation Tax service Checkout BFF calls Tax calculation engine
💡 BFF as the Integration Point The Checkout BFF is the only place where cross-domain data is aggregated. The frontend never makes calls to Cart, Account, or Catalog services directly. This keeps the frontend simple and testable — it talks to one API with one contract.

Extraction Sequence

Cart, Mini-Cart, and Checkout are extracted in order, after Catalog proves the pattern:

Step 1: Extract Cart Second domain after Catalog
  1. Copy monolith → medline-cart
  2. Strip everything except cart pages, cart components, cart state
  3. Build Cart BFF with cart-specific endpoints
  4. Wire ml:add-to-cart event handling
  5. Route /cart traffic to the new service
  6. Verify: Cart works independently; Catalog “Add to Cart” events are received correctly
Step 2: Extract Mini-Cart Smallest extraction — header overlay only
  1. Mini-Cart is just 2–3 Web Components loaded into the shell header
  2. No separate service needed — these components are published to CDN
  3. They subscribe to ml:cart-updated events and render reactively
  4. Verify: Adding items from Catalog updates the Mini-Cart badge and overlay
Step 3: Extract Checkout Highest-risk domain — canary deployment
  1. Copy monolith → medline-checkout
  2. Strip everything except checkout flow components
  3. Build Checkout BFF with all checkout endpoints
  4. Implement the checkout state machine
  5. Canary deploy: 1% → 5% → 10% → 25% → 50% → 100%
  6. Verify: Full checkout flow completes, orders are placed successfully, payment processing works
Industry Standard: Strangler Fig Pattern Extracting domains incrementally while the monolith continues serving traffic is the Strangler Fig pattern described by Martin Fowler. We grow the new architecture around the old one, gradually routing traffic until the monolith has nothing left to serve. Every major enterprise migration at scale uses this approach.

Testing Cross-Domain Interactions

The most important tests for Cart/Mini-Cart/Checkout are the integration tests that verify events flow correctly across domain boundaries:

Critical Test Scenarios

Scenario Domains Involved What to Verify
Add item from PDP Catalog → Cart → Mini-Cart Cart state updates, Mini-Cart badge increments, overlay shows new item
Remove item from Mini-Cart Mini-Cart → Cart Cart state updates, Mini-Cart re-renders, cart page reflects removal
Proceed to checkout Cart → Checkout Checkout receives cart ID, loads cart items, begins at shipping step
Place order successfully Checkout → Cart → Mini-Cart Cart cleared, Mini-Cart shows zero items, confirmation page renders
Payment gateway error Checkout (internal) State machine transitions to PaymentError, user can retry, cart intact
Cart domain unavailable Mini-Cart fallback Mini-Cart shows “last known state” from cached events, no crash
