What Is a Domain?

In Domain-Driven Design, a domain is a bounded context—a self-contained area of business functionality with clear boundaries, its own data, and its own rules. Domains communicate with each other through well-defined contracts, never by reaching into each other's internals.

For our e-commerce platform, we've identified 8 primary domains as a starting point. Each domain maps to a business capability that can be owned, developed, tested, and deployed independently. A domain committee will refine these boundaries as the migration progresses.

Domain-Driven Design (Eric Evans, 2003) Bounded contexts are the foundational building block of Domain-Driven Design, the industry-standard approach to decomposing complex systems. The principle: each bounded context has a clearly defined boundary, owns its data, and communicates with other contexts only through explicit contracts. This eliminates the implicit coupling that creates monolithic behavior.

Data Flow Between Domains

The critical design principle: domains share data contracts, not data stores. Each domain owns its database (or its slice of the database). When Domain A needs data from Domain B, it calls Domain B's API. It never reads Domain B's database directly.

What It Owns

Special Considerations

The Admin domain is typically the last to be extracted because it has the widest data access requirements and the smallest user base (internal staff only). It's also the lowest risk to leave in the monolith longest because admin traffic is minimal compared to customer-facing traffic.

Shared / Core Domain Header, footer, authentication, analytics

How Other Domains Consume Core

The Core domain publishes its components as Web Components:

<ml-header></ml-header>
<ml-footer></ml-footer>
<ml-auth-guard></ml-auth-guard>

Each domain's shell loads these Web Components. They're versioned and published to the App Catalog. Domains can pin to a specific version of Core components, ensuring updates are controlled.

Extraction Order: Simplest First, Hardest Last

The actual extraction order will be determined by complexity—we start with whichever domain has the fewest dependencies and clearest boundaries, and work our way toward the most coupled. The domain committee decides the sequence when the time comes.

Domain Boundary Principles

🔒

Own Your Data

Each domain owns its data store. No domain reads another domain's database. All cross-domain data access goes through APIs. This is non-negotiable.

📄

Contract-First Communication

Before two domains can communicate, they define a contract: endpoint, request shape, response shape, error codes. The contract is versioned and tracked in the App Catalog.

💥

Independent Failure

If the Catalog domain goes down, Cart still works. If Search is slow, Account is unaffected. More importantly, each domain can be individually owned, debugged, fixed, and tuned—clear boundaries mean clear ownership.

🚀

Independent Deployment

Each domain deploys on its own schedule. No waiting for other teams. No coordinated release trains. Your domain ships when your tests pass.

The Boundary Test If you can't deploy your domain without coordinating with another domain, the boundary is wrong. Go back and redraw it. True domain independence means true deployment independence.
