Migration Mechanics

Copy, Split, Strip: The Safest Migration in the World

Copy the monolith, delete what isn't needed, deploy the same code with less of it. Zero risk because it's the same code our teams already trust.

Why Copy-Split-Strip Is the Safest Approach

Most migration strategies involve rewriting code. Rewriting is dangerous because new code has new bugs. It hasn't been battle-tested by real traffic. It hasn't survived edge cases that took years to discover.

Copy-split-strip is different. We're not writing new code. We're taking code that already works in production and removing the parts that don't belong to the target domain. The code that runs after the extraction is a subset of the code that was running before. Same logic. Same tests. Same infrastructure. Just less of it.

The Safety Guarantee If the monolith works today, the extracted domain works today. We're running the same code with fewer routes, fewer components, and fewer dependencies. The only change is subtraction.
Industry-Standard Decomposition Copy-split-strip is a variant of the "branch by abstraction" technique documented by Paul Hammant and widely used at Google, Facebook, and other large-scale engineering organizations. The key insight: decomposition by subtraction is always safer than decomposition by construction.

The Five Steps

Figure 5-1: Copy-split-strip flow — the monolith remains as fallback at every step

Step 1: Copy the Entire Repository

Create a complete clone of the monolith repository. This is a git clone followed by removing the .git directory and initializing a new repository. The new repo contains everything the monolith contains—all MFEs, all routes, all components, all tests.

Why copy everything? Because it's simpler and safer than trying to extract only what's needed. When starting with everything, we can verify at any point that the full system still works. Then subtract.

Step 2: Rename to the Domain

Rename the repository to reflect its domain identity: mfe-catalog, mfe-cart, mfe-checkout, etc. Update the package.json name, the K8s service name, and the CI/CD pipeline name. The code inside is still identical to the monolith.

Step 3: Strip Non-Domain Code

This is the core step. Systematically delete everything that doesn't belong to this domain:

  • Delete MFE directories for other domains
  • Delete route definitions for other domains
  • Delete page components for other domains
  • Delete test files for other domains
  • Run npm prune to remove unused dependencies
  • Remove unused shared utilities (tree-shaking will catch most of this)

Step 4: Deploy Exactly As-Is

Deploy the stripped domain using the same infrastructure as the monolith: same K8s configuration (adjusted for lower resource needs), same SSR setup, same environment variables, same secrets. The only difference is that this service handles fewer routes.

Step 5: Verify and Route

Run the domain's test suite. It should pass because the code is identical to what was tested in the monolith. Then gradually route traffic for this domain's URLs from the monolith to the new domain service.

The Rule of Thumb

When in doubt, keep it. It's safer to have extra code that's unused than to accidentally delete something the domain needs. Tree-shaking and dead-code elimination will handle unused imports in the build. Teams can always delete more later.

What Stays The shared infrastructure that every domain keeps

Every Domain Keeps

  • K8s configuration: Deployment manifests, service definitions, ingress rules (adjusted for the domain's routes)
  • CI/CD pipeline: Build, test, deploy scripts (simplified for one domain instead of 84 MFEs)
  • Shared utilities: Authentication helpers, API client wrappers, logging utilities (initially kept as-is, later extracted to shared packages if needed)
  • Shared components: Header, footer, navigation components that the domain renders (these are candidates for the shared/core domain in later phases)
  • Configuration: Environment variables, feature flags, configuration files relevant to this domain
  • Monitoring: Health check endpoints, metrics exporters, logging configuration

Gradual Cleanup

After the initial extraction, the domain team can incrementally clean up remaining unused code. This is low-priority because the unused code doesn't affect runtime behavior—it just takes up space in the repository. Clean it up when convenient, not urgently.

Risk Mitigation What could go wrong and how we handle it

Risk: We deleted something the domain needs

Mitigation: The build will fail. If it builds but tests fail, the tests catch it. If tests pass but runtime fails, the staging environment catches it. At every stage, the monolith is still running and serving traffic. Restore the file from the monolith commit hash and re-strip.

Risk: The domain has a hidden dependency on another domain's code

Mitigation: This is actually the most valuable discovery in the process. It reveals coupling that wasn't documented. When found, there are two choices: (a) keep the dependency temporarily and plan to decouple it, or (b) extract the shared code into a shared package that both domains can consume.

Risk: Performance is different as a standalone service

Mitigation: The standalone service is doing less work than the monolith (fewer routes, fewer components). Performance should be the same or better. If it's worse, it reveals a resource contention issue in the monolith that was masking individual domain performance.

Risk: The team doesn't know which files belong to their domain

Mitigation: Start with an overly generous definition of the domain (keep more than needed). The domain team knows their routes and their pages. Delete other domains' routes and pages first. Everything else can be cleaned up iteratively. See Page 6: Domain Map for detailed domain definitions.

The Fallback Is Always There At every step of the copy-split-strip process, the original monolith is still running and serving traffic. If the extracted domain has any issues, traffic routes back to the monolith. There is no point of no return until we explicitly decommission the monolith routes—and that only happens after weeks of validated production traffic on the domain service.

Common Questions

What about shared state between domains? How do Cart and Catalog communicate?

Domains communicate through well-defined API contracts, not shared state. When the Catalog domain needs to add an item to the Cart, it calls the Cart domain's API endpoint. The Cart domain owns its own state.

For real-time UI updates (like "item added to cart" notifications), domains communicate through browser-level events (Custom Events on the window object). This is a standards-based, framework-agnostic pattern.

See Page 6: Domain Map for detailed data flow between domains.

What about shared components like the header? Who owns the shared UI?

The header, footer, navigation, and authentication components belong to the "Shared/Core" domain. This domain is extracted just like any other. It provides shared components as Web Components that other domains can consume.

Initially, each extracted domain carries its own copy of shared components (remember, we copied the entire monolith). Over time, these are replaced with the shared domain's Web Component versions. But there's no rush—the copied versions work fine.

Do we need to extract all domains at once? Can we do one at a time?

Absolutely not. Extract one domain at a time. Start with the domain that has the clearest boundaries and the most to gain from independence (typically Catalog or Search). Prove the pattern. Build team confidence. Then extract the next domain.

The monolith continues to serve all un-extracted domains. As domains are extracted one by one, the monolith shrinks. Eventually, it serves no traffic and can be decommissioned.
