Track: Platform Engineer

Six modules for the team that maintains the shell, the App Catalog, the CDN, and the infrastructure that makes everything else possible. You keep the platform running.

Track Overview

Platform Engineers are the team behind the scenes. You manage the shell that composes domain micro frontends, the App Catalog that tracks every component and version, the API scanning that catches breaking changes, and the CDN that delivers it all. Without you, domains can't deploy and components can't be discovered.

This track has 6 modules. Expected completion: ~3 weeks.

Platform Engineer Responsibilities

Platform Engineer responsibilities mapped to six system areas

Module 1: Shell Architecture

What you'll learn: How the CSR shell composes domain micro frontends, manages the import map, and handles route registration.

Module content CSR composition, import maps, routing

Key Concepts

  • CSR composition — the shell is a thin orchestration layer that loads domain bundles on demand via the import map
  • Import map — a JSON file that maps module specifiers to CDN URLs. This is how the shell knows where each domain lives.
  • Route registration — domains declare their routes in domain.config.json. The shell aggregates them at build time.
  • Lazy loading — domains are loaded only when their route is matched. No upfront download of all domains.
  • Shell lifecycle — bootstrap, route match, domain load, mount, unmount

Hands-on Exercise

Register a new sandbox domain in the shell's route table. Configure its import map entry pointing to the CDN. Verify the shell loads the domain when its route is navigated. Test lazy loading by confirming no other domains are downloaded.

Sign-off Criteria

  • New domain registered in route table
  • Import map entry points to correct CDN URL
  • Domain loads correctly on route navigation
  • Lazy loading verified (no unnecessary downloads)
Industry Standard: Import Map Composition Import map-based micro frontend composition is the industry standard for client-side rendering platforms. Used by IKEA, Spotify, and other enterprise teams, it provides runtime module resolution without a monolithic build step. Each domain deploys independently; the shell simply updates a JSON file.

Module 2: App Catalog Operations

What you'll learn: How the App Catalog scans, registers, and tracks every component and domain version across the platform.

Module content Scanning, registration, version tracking, alerts

Key Concepts

  • Component scanning — automated pipeline scans published components, extracts contracts, and updates the catalog
  • Version tracking — every component version is tracked: who published it, when, what changed, and who consumes it
  • Dependency mapping — the catalog knows which domains depend on which component versions
  • Breaking change alerts — when a component contract changes in a breaking way, the catalog alerts all consuming domains
  • Health status — each component and domain has a health indicator based on test results, deployment status, and error rates

Hands-on Exercise

Publish a new component version with a contract change. Verify the catalog detects the change, updates the version record, identifies consuming domains, and sends a breaking change alert to the sandbox domain that depends on it.

Sign-off Criteria

  • Catalog detects the new version automatically
  • Contract diff is generated correctly
  • Consuming domains are identified
  • Breaking change alert is sent to affected domains

Module 3: API Scanning

What you'll learn: How automated API scanning discovers contracts, generates diffs, and notifies consumers of changes.

Module content Contract discovery, diff generation, notifications

Key Concepts

  • Contract discovery — the scanner inspects BFF endpoints and generates OpenAPI contracts automatically
  • Diff generation — when a contract changes, a structured diff shows exactly what was added, removed, or modified
  • Breaking vs. non-breaking — the scanner classifies changes: adding a field is non-breaking, removing or renaming is breaking
  • Consumer notification — affected domain teams are notified with the diff and remediation guidance
  • Contract versioning — old contracts remain available until all consumers have migrated to the new version

Hands-on Exercise

Modify a sandbox BFF endpoint: add a new field (non-breaking) and rename an existing field (breaking). Run the API scanner. Verify it generates correct diffs, classifies the changes correctly, and sends the right notifications.

Sign-off Criteria

  • Scanner detects both changes
  • Non-breaking change classified correctly
  • Breaking change classified correctly with remediation guidance
  • Notifications sent to consuming domains
Industry Standard: Automated Contract Testing Automated API contract scanning and consumer-driven contract testing are the industry standard for microservice and micro frontend architectures. Tools like Pact, Spectral, and custom OpenAPI validators are used by enterprises to catch breaking changes before they reach production.

Module 4: CDN & Import Map Management

What you'll learn: How to manage the CDN publish workflow, version pinning, and import map rollback.

Module content CDN workflow, version pinning, rollback

Key Concepts

  • CDN publish workflow — built assets are pushed to CDN with unique version-hashed URLs
  • Import map management — updating the master import map to point domains at specific versions
  • Version pinning — locking a domain to a specific component version to prevent unexpected updates
  • Rollback — reverting the import map to a previous version. Old versions remain on CDN indefinitely.
  • Cache strategy — immutable URLs with content hashes for infinite caching; import map has short TTL

Hands-on Exercise

Publish a sandbox domain to CDN. Update the import map. Verify the new version loads. Pin a component to a specific version. Perform a rollback. Verify the previous version is restored. Confirm cache headers are correct.

Sign-off Criteria

  • Domain published to CDN with correct version hash
  • Import map updated and shell loads new version
  • Rollback restores previous version within seconds
  • Cache headers verified (immutable for assets, short TTL for import map)

Module 5: Core Cadence

What you'll learn: The annual core update process—how change-mod scripts are authored, how updates are staged, and how rollouts happen domain-by-domain.

Module content Annual updates, change-mod scripts, staged rollout

Key Concepts

  • 1-year cadence — core framework updates happen once a year, not continuously
  • Change-mod scripts — automated scripts that transform domain code to be compatible with the new core version
  • Script authoring — how to write a change-mod script: AST transforms, config updates, dependency bumps
  • Staged rollout — core update rolls out domain-by-domain, starting with the smallest/simplest domain
  • Validation — each domain validates in staging before the update goes to production
  • Rollback — if a domain fails validation, it stays on the old version until the script is fixed

Hands-on Exercise

Write a change-mod script that renames a deprecated API call across all domain code. Run it against the sandbox domain. Verify the transformation is correct. Stage the updated domain. Run validation tests.

Sign-off Criteria

  • Change-mod script runs without errors
  • All deprecated API calls are renamed correctly
  • Domain passes all validation tests after transformation
  • Script is idempotent (running twice produces the same result)
Annual core update workflow: release → scripts → transform → validate → rollout
Industry Standard: Automated Codemods Automated code transformation scripts (codemods) are the industry standard for large-scale framework updates. Facebook pioneered this approach with jscodeshift for React upgrades. Google uses similar tools for their monorepo migrations. The key: updates are automated, repeatable, and verifiable.

Module 6: Monitoring & Health

What you'll learn: How to monitor domain health, track deployments, and diagnose platform-wide issues.

Module content Health dashboard, deployment tracking, diagnostics

Key Concepts

  • Domain health dashboard — real-time view of every domain: deployment status, error rates, latency, and test results
  • Deployment tracking — history of every deployment: who, when, what version, which import map change
  • Error aggregation — errors are grouped by domain, component, and severity so you know where to look first
  • Performance baselines — each domain has performance budgets; violations trigger alerts
  • Incident response — when something goes wrong: how to identify the root domain, rollback, and post-mortem

Hands-on Exercise

Deploy a sandbox domain version that intentionally violates a performance budget. Verify the monitoring dashboard detects the violation. Perform a rollback. Confirm the dashboard reflects the restored healthy state. Write a one-paragraph post-mortem.

Sign-off Criteria

  • Performance violation detected by monitoring
  • Alert triggered correctly
  • Rollback performed and verified
  • Post-mortem documents root cause and resolution
