Migration Mechanics

The Migration Strategy

Strangler fig pattern, four phases, zero downtime. Each phase delivers value independently—we can stop at any point and still be better off.

The Strangler Fig Pattern

The strangler fig is a tropical tree that grows around an existing tree. Over time, the fig's roots replace the host tree's structure. The host tree eventually decomposes, leaving the fig standing in its place. At no point did anyone cut down the host tree. At no point was there a gap where no tree existed.

This is exactly how we migrate the platform. The old system keeps running while the new system grows around it. Traffic gradually shifts from old to new. When the new system has fully replaced the old, the old system is turned off—not because it broke, but because nobody needs it anymore.

Industry-Standard Migration Pattern The Strangler Fig pattern was named by Martin Fowler in 2004 and has become the de facto standard for enterprise system migration. It has been successfully applied at Amazon (monolith to microservices), the UK Government Digital Service (legacy to cloud), BBC (monolith to distributed services), and hundreds of other large-scale organizations. The core principle: never turn off the old system until the new system is proven.

The Four Phases

Our migration is organized into four distinct phases. Each phase has a clear entry criteria, exit criteria, and value delivery. Critically, each phase is independently valuable. If we stop after Phase 1, we've gained independent deployment. If we stop after Phase 2, we've also gained a standards-based shell. There is no phase that requires all subsequent phases to be valuable.

Figure 4-2: Four-phase overview — each phase delivers independent value and has its own success criteria

Phase 1: Domain Extraction (Copy-Split-Strip)

What Happens in Phase 1 The safest migration step

The Process

For each identified domain (see Page 6: Domain Map):

  1. Copy the entire monolith repository into a new domain-specific repository
  2. Split by identifying which files, routes, and components belong to this domain
  3. Strip everything that doesn't belong. Delete unused MFEs, unused routes, unused components.
  4. Deploy the domain exactly as the monolith deploys—same K8s config, same SSR, same infrastructure
  5. Verify by running the domain's existing tests. They pass because the code is identical.
  6. Route traffic for this domain's URLs from the monolith to the new domain service

Why It's Safe

The extracted domain runs the exact same code as the monolith. It was tested in the monolith. It works in the monolith. It will work as a standalone service because it is literally the same binary with fewer routes.

What We Get

  • Independent Git repository per domain
  • Independent build pipeline (5-minute builds instead of 25-minute builds)
  • Independent test suite (domain tests only, 3-5 minutes instead of 45+)
  • Independent deployment (deploy when the domain's tests pass)
  • Smaller blast radius (a bug in Catalog doesn't affect Cart)
Phase 1 Value Proposition Even if we stop after Phase 1 and never do anything else, we've transformed a monolithic deployment pipeline into independent domain pipelines. Teams ship faster, test faster, and fail smaller.

Phase 2: Shell Swap

What Happens in Phase 2 Standards-based orchestration

The Process

  1. Build a new thin shell using W3C Web Components. This shell is a lightweight orchestrator that loads domain micro frontends via <custom-element> tags.
  2. Wrap each existing domain's entry point in a Web Component container. The internal React code doesn't change—it just mounts inside a shadow DOM boundary.
  3. Deploy the new shell alongside the old Next.js shell. Both serve the same content.
  4. Test the new shell with synthetic traffic, then gradually shift real traffic using feature flags or weighted routing.
  5. Retire the Next.js shell once the Web Component shell serves 100% of traffic.

Why Web Components

  • Browser-native: No framework runtime needed. Custom Elements and Shadow DOM are built into every browser.
  • Framework-agnostic: Web Components can host React, WCFramework, vanilla JS, or any future framework. The shell doesn't care what's inside each component.
  • Stable API: The Web Component spec has been stable since 2018. No breaking version upgrades. No migration scripts needed for the shell.
  • Isolation: Shadow DOM provides style and DOM isolation between domains. One domain's CSS can't leak into another.

What We Get

  • No more SSR runtime (CSR from CDN + BFF backends)
  • No framework lock-in for the orchestration layer
  • Style isolation between domains via Shadow DOM
  • Simpler infrastructure (static files + API services)

Phase 3: App Catalog & Governance

What Happens in Phase 3 Automated visibility and control

The Process

  1. Deploy the App Catalog registry—a service that tracks every domain, every micro frontend, every component, and every API endpoint across the platform.
  2. Register each domain with the catalog: its version, its dependencies, its API surface, its component inventory.
  3. Enable scanning: Automated CI hooks that report to the catalog on every deployment. The catalog knows what's running everywhere, in real-time.
  4. Set policies: Define rules for version freshness, security compliance, accessibility standards, and performance budgets. The catalog enforces them automatically.

What We Get

  • Single pane of glass showing every deployed micro frontend and its status
  • Automated detection of outdated dependencies, security vulnerabilities, and breaking API changes
  • Component inventory: what exists, who owns it, what version, where it's used
  • Governance without meetings: policies are automated, not manual reviews

Phase 4: Component Evolution

What Happens in Phase 4 Incremental component migration

The Process

  1. Identify components for migration using the App Catalog's component inventory. Start with the most widely-used, most duplicated components.
  2. Build the WCFramework version of each component following the standards-based component model (see Page 14).
  3. Deploy the new component alongside the old one. Feature flags control which version renders.
  4. Validate with A/B testing, accessibility audits, and performance benchmarks.
  5. Retire the React version once the WCFramework version is validated.

Why WCFramework

WCFramework is a standards-based component framework built on W3C Web Components. It provides the developer ergonomics teams need (templating, data binding, dependency injection) while compiling to native Custom Elements. The output is browser-standard code, not framework-specific bundles.

The 1-Year Core Cadence

Once on WCFramework, core updates follow a predictable 1-year cycle:

  • January: New core version released with automated migration schematics
  • February–March: Teams run the migration scripts and validate in staging
  • April: Domain-by-domain rollout to production
  • May–December: Feature development on the stable core
Phase 4 Value Proposition Components are now standards-based, cataloged, scanned, and on a predictable update cadence. The platform is future-proof because it's built on browser standards, not framework-specific patterns.

Zero Downtime Guarantees

At no point in this four-phase migration does the platform go offline. Here's why:

Stop at Any Phase

One of the most powerful aspects of this strategy is that each phase delivers value independently. There's no need to commit to all four phases upfront. We can evaluate results at each phase boundary and decide whether to continue.

What if we stop after Phase 1? Domain extraction only

We have independent domain repositories, independent build pipelines, and independent deployment. Teams deploy on their own schedule. Build times drop from 25 minutes to 5. Test times drop from 45 minutes to 5. The blast radius for any failure shrinks from "entire platform" to "one domain."

What we don't get: SSR is still in play. The shell is still Next.js. Components are still React. We're on the upgrade treadmill for the framework, but at least each domain can upgrade independently.

What if we stop after Phase 2? Domain extraction + shell swap

Everything from Phase 1, plus: SSR is eliminated. The shell is standards-based Web Components. Infrastructure is simpler (CDN + BFF). There's no framework lock-in at the shell level.

What we don't get: No automated governance. No component catalog. Components are still React (wrapped in Web Components). The 1-year cadence hasn't started yet.

What if we stop after Phase 3? Full visibility but React components

Everything from Phases 1-2, plus: full visibility into what's deployed everywhere. Automated scanning catches problems. Governance policies are automated. We know exactly what components exist, who owns them, and what version they're on.

What we don't get: Components are still React. The 1-year core cadence hasn't started. We're still on the React upgrade treadmill for individual components (though each domain manages this independently now).

What if we complete all four phases? Full evolution

Independent domains. Standards-based shell. Full governance and visibility. Standards-based components on a predictable 1-year update cadence. Automated migration scripts. No framework lock-in at any level. The platform is built on browser standards and can evolve for the next decade without another major migration.

This is the target state. Every architectural choice compounds in our favor over time instead of against us.
