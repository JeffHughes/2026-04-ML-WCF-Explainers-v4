The Big Picture

Where We Are Today

A detailed breakdown of the current monolithic architecture, its coupling patterns, and the operational friction your teams face every day.

The Platform at a Glance

Your current e-commerce platform is a Next.js 12.3 monolithic shell that hosts 84+ micro frontends. On paper, it's a micro frontend architecture. In practice, every MFE shares the same SSR runtime, the same build pipeline, the same node_modules, and the same failure domain.

This means that in day-to-day operations, the "micro" in micro frontend is aspirational rather than actual. Teams experience the coordination overhead of a monolith while carrying the conceptual complexity of a distributed system—big-bang releases, dependency inconsistencies across shared node_modules, and weird UI flicker issues from competing SSR hydration and style conflicts.

Understanding the Pain Points

The Monolith Coupling Problem Why 84 MFEs behave like one

What Coupling Looks Like in Practice

Imagine the Catalog team needs to update a date-formatting library. In a truly independent architecture, they'd update it in their repo, run their tests, and deploy. Done.

In the current monolith, here's what actually happens:

  1. The Catalog team updates date-fns in the shared package.json
  2. The change propagates to all 84 MFEs because they share node_modules
  3. The full build pipeline runs (15-25 minutes)
  4. The full regression test suite runs (45+ minutes)
  5. Three unrelated MFEs fail because they used an undocumented API in the old version of date-fns
  6. The Catalog team's PR is blocked until three other teams fix their code
  7. The three other teams are in different time zones and different sprint cycles
  8. What should have been a 10-minute change takes 2-3 days

This is the daily reality of a "micro frontend" architecture that shares a monolithic runtime.

The Shared Failure Domain

When all MFEs share the same Node.js SSR runtime, a memory leak in one MFE affects all MFEs. A slow API call in Checkout blocks SSR rendering for Catalog. An unhandled exception in Account can crash the entire shell.

There is no isolation. The "micro" in micro frontend is a label, not a boundary.

The SSR Tax Why server-side rendering costs more than it gives

The Promise of SSR

Server-side rendering was originally adopted for two reasons: SEO (search engines needed pre-rendered HTML) and perceived performance (first paint appears faster). Both of these reasons have significantly diminished:

  • SEO: Modern search engine crawlers (Google, Bing) execute JavaScript and render SPAs. Dedicated product pages can use static generation or prerendering without a full SSR runtime.
  • Performance: The SSR "fast first paint" comes at the cost of a large JavaScript bundle that must hydrate after the initial HTML arrives. The Time to Interactive (TTI) often ends up slower than a well-optimized CSR approach with code splitting.

The Cost of SSR at Scale

For an e-commerce platform with 84+ MFEs, SSR imposes a heavy tax:

  • Shared runtime process: All MFEs render in the same Node.js process. One slow component blocks the entire render pipeline.
  • Memory pressure: SSR requires maintaining component state on the server for every concurrent request. At scale, this means significant memory allocation per request.
  • Hydration cost: After SSR sends HTML to the browser, React must "hydrate" the page—re-attaching event handlers and reconstructing the virtual DOM. For complex pages, hydration can take 2-5 seconds.
  • Deployment coupling: Because all MFEs share the SSR runtime, you can't deploy one MFE's server-side code without deploying all of them.
The Hidden Cost SSR infrastructure is more complex to operate, harder to debug, and more expensive to scale than CSR + CDN. Every request requires compute. CSR serves static files from the edge.
The Testing Bottleneck Why a 10-line change triggers a 45-minute test suite

The Current Testing Flow

Because all 84 MFEs share a single repository and build pipeline, the CI/CD system has no way to determine which MFEs are affected by a change. The safe default: run everything.

  1. Developer pushes a PR touching 3 files in the Catalog MFE
  2. CI triggers full build of all 84 MFEs (15-25 minutes)
  3. Unit tests run for all MFEs (10-15 minutes)
  4. Integration tests run for all MFEs (15-20 minutes)
  5. E2E tests run across the full platform (10-15 minutes)
  6. Total: 45-75 minutes for a change that affects one domain

The Impact

  • Developers context-switch while waiting for CI. Productivity drops.
  • Flaky tests in unrelated MFEs block unrelated PRs. Teams waste time investigating failures that aren't theirs.
  • The feedback loop is too long. Developers batch changes instead of shipping small, frequent updates.
  • The cost of running the full suite for every PR scales linearly with team size. More developers = more queued builds = longer wait times.

The Fix Preview

With independent domain repositories, each domain runs only its own tests. A change in Catalog triggers Catalog's test suite (3-5 minutes), not the entire platform's. The feedback loop shrinks from 45 minutes to 5 minutes.

The Component Sprawl Nobody knows what exists or what version is running

The Problem

With 84+ MFEs, there are hundreds of React components spread across the codebase. Many of them do the same thing with slight variations:

  • 6 different button components with different styling approaches
  • 4 different modal implementations
  • 3 different table components, each with their own sorting and pagination logic
  • 2 different date picker implementations
  • Multiple product card layouts that should be standardized but aren't

No one has a complete inventory. There is no component catalog. When a new team member asks "do we have a date picker?", the answer is "check Slack" or "search the codebase."

The Consequences

  • Duplicate work: Teams build components that already exist because they can't find them.
  • Inconsistent UX: Different implementations of the same pattern lead to different behavior for end users.
  • Accessibility gaps: Without standards, accessibility support varies wildly between implementations.
  • Maintenance burden: When a design system change is needed, there's no way to know which components need updating.

The Fix Preview

The App Catalog (Page 16) will register every component with its version, ownership, and interface contract. Automated scanning will detect duplicate components and flag non-standard implementations.

Summary: The Current State

The Bottom Line You have a "micro frontend" architecture that delivers monolithic coupling. Teams can't deploy independently, can't test independently, and can't fail independently. The SSR runtime is a single point of failure. The build pipeline is a bottleneck. The component landscape is ungoverned.

None of this means the platform is failing. It processes orders, it serves customers, it generates revenue. But the architecture has reached the point where the cost of change is growing faster than the value of change. Every new feature takes longer. Every upgrade is riskier. Every team is slower than they should be.
