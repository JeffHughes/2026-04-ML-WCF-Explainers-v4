Training at a Glance

The 30-second version. Three tracks, hands-on modules, automated sign-off. Everything the team needs to know about how developers get trained and certified for the new platform.

The Entire System in 30 Seconds

Every developer picks one of three training tracks based on their role: Domain Developer, Component Author, or Platform Engineer. Each track has a series of hands-on modules—not slide decks, not lectures, real exercises. Each module ends with an automated exercise that the pipeline validates. Complete all modules in our track, and we earn a certification badge that grants production deploy access.

No meetings required for sign-off. No manager approval needed. The pipeline IS the sign-off.

The One-Line Summary Complete our track → pass the automated exercises → earn our badge → deploy to production. That's it.

Three Training Tracks

Every developer falls into one of three roles. Each role has a dedicated training track with progressively deeper modules.

💻

Domain Developer

6 modules • ~3 weeks
For developers who work within a single domain (Catalog, Cart, Account, etc.). The most common track. Covers domain structure, component usage, BFF integration, local dev, deployment, and debugging.

🧩

Component Author

7 modules • ~4 weeks
For developers who build shared ML- components used across domains. Covers Custom Elements, Shadow DOM, styling encapsulation, publishing, WCFramework internals, React wrappers, and documentation.

⚙️

Platform Engineer

6 modules • ~3 weeks
For the team maintaining the shell, catalog, and infrastructure. Covers shell architecture, catalog operations, API scanning, CDN management, core cadence, and monitoring.

Three tracks, one destination: every path leads to certified production access
Industry Standard: Role-Based Training Tracks Competency-based certification is the industry standard for enterprise development teams. Rather than generic training for everyone, role-specific tracks ensure each developer learns exactly what they need for their daily work—nothing more, nothing less.

How Modules Work

Every module across all three tracks follows the same structure. No surprises.

📖

1. Learn

Short documentation page covering the concept. Diagrams, examples, code snippets. Takes 30–60 minutes to read.

🛠️

2. Exercise

A hands-on task in a real codebase (sandbox environment). Build something, fix something, deploy something. Takes 1–3 hours.

✅

3. Validate

Push our exercise to the pipeline. Automated gates check our work. Pass = module complete. Fail = clear feedback on what to fix.

What does “automated validation” actually mean? How the pipeline checks our work

When we push our exercise code, the CI/CD pipeline runs a specific set of checks for that module:

  • Code structure — does our code follow the expected file layout and naming conventions?
  • Test coverage — do our tests pass, and do they cover the required scenarios?
  • Component contract — if we built a component, are all attributes documented and events typed?
  • Integration — does our code work with the rest of the domain/system?

Each check produces a pass/fail result with a descriptive message. If we fail, we fix and re-push. There is no limit on attempts.

Sign-off Is Automated

The old model: complete training, schedule a review meeting, wait for a manager to approve, get access. That model doesn't scale across timezones. It creates bottlenecks. It introduces subjectivity.

The new model: the pipeline is the sign-off.

Automated sign-off loop: exercise → pipeline → pass/fail → repeat until certified
Industry Standard: Pipeline-Based Sign-off Automated sign-off removes timezone bottlenecks, subjective reviews, and scheduling delays. If the code passes objective, reproducible criteria, it ships. This is how high-velocity enterprise teams operate at scale.

What This Means for the Team

For Directors Strategic benefits

Predictable onboarding. Every new developer follows the same documented path. We can forecast ramp-up time accurately. A domain developer is productive in 3 weeks. A component author in 4 weeks.

No bottlenecks. Sign-off doesn't depend on any individual manager's availability. The pipeline validates competency 24/7, across every timezone.

Measurable quality. Certification is objective. Either the developer's code passes the automated gates, or it doesn't. No ambiguity.

For Team Leads Operational benefits

Clear expectations. Each module has documented sign-off criteria. Our developers know exactly what “done” looks like before they start.

Track progress. The App Catalog shows each developer's certification status. We see at a glance who's certified for what and who's still in training.

No training meetings. We don't need to schedule review sessions. The pipeline handles validation. Our time is freed for actual leadership work.

For Developers Day-to-day benefits

Self-paced. Work through modules at our own speed. No waiting for cohorts or scheduled sessions. No timezone dependencies.

Hands-on. We learn by building, not by watching. Every module produces something real that we can reference later.

Clear feedback. When an exercise fails validation, we get specific, actionable feedback. Not “try harder”—exactly what check failed and why.
