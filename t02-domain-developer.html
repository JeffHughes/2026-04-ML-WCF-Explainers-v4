Track: Domain Developer

Six modules that take you from “what is a domain?” to confidently deploying production changes. The most common training path—this is where most developers start.

Track Overview

Domain Developers are the most common role in the platform. You work within a single domain—Catalog, Cart, Account, Checkout, Search, Content, or Admin. You build features using existing ML- components, connect to your domain's BFF, and deploy independently.

This track has 6 modules, each building on the last. Expected completion: ~3 weeks at normal pace.

The Domain Developer Workflow

Domain developer daily workflow: code → test locally → push → pipeline gates → staging → production

Module 1: Domain Anatomy

What you'll learn: The file structure of a domain, how routing works, where components are placed, and how your domain fits into the larger platform.

Module content File structure, routing, component slots

Key Concepts

  • Domain repository structure — every domain follows the same layout: /src, /components, /routes, /services, /tests
  • Route registration — how your domain declares its routes so the shell can load them
  • Component slots — where ML- components plug into your domain's pages
  • Domain boundaries — what belongs in your domain vs. what belongs in shared components or other domains
  • Configuration files — domain.config.json declares your domain's routes, dependencies, and BFF endpoint

Hands-on Exercise

Clone the sandbox domain repository. Identify the route configuration, the component slots on the main page, and the BFF endpoint. Add a new route that renders an existing ML- component. Push your changes.

Sign-off Criteria

  • New route is registered correctly in domain.config.json
  • Route renders the specified ML- component
  • File placement follows the standard domain structure
  • All existing tests still pass

Module 2: The ML- Component Contract

What you'll learn: How to use ML- components correctly. Attributes in, events out, Shadow DOM boundaries. The contract you follow as a consumer.

Module content Attributes, events, Shadow DOM

Key Concepts

  • Attributes in — pass data to components via HTML attributes and properties
  • Events out — components communicate back via Custom Events
  • Shadow DOM — components encapsulate their styles and DOM. You cannot reach inside. You don't need to.
  • Slots — components expose named slots for projected content
  • CSS custom properties — the only way to style a component from outside is through documented CSS custom properties

Hands-on Exercise

Build a page that uses three ML- components: ml-product-card, ml-search-bar, and ml-pagination. Wire up event listeners so search filters the cards and pagination controls the page. All data flows through attributes and events only.

Sign-off Criteria

  • Components are used via standard HTML (no reaching into Shadow DOM)
  • Data flows through attributes and events only
  • CSS custom properties used for any styling overrides
  • Event handlers are properly typed
Industry Standard: Component Contracts Well-defined component contracts (inputs, outputs, slots, styling hooks) are the industry standard for scalable component ecosystems. They enable independent teams to consume components without coordination, and allow component authors to evolve internals without breaking consumers.

Module 3: BFF Integration

What you'll learn: How your domain communicates with its backend through the Backend-for-Frontend (BFF) pattern. One domain, one BFF, one API contract.

Module content BFF pattern, API contracts, service layer

Key Concepts

  • BFF per domain — your domain has a dedicated backend service that provides exactly the data your UI needs
  • Service layer — all API calls go through a typed service layer in /services
  • API contract — your BFF publishes a typed contract (OpenAPI). The App Catalog tracks it.
  • Error handling — standard error patterns: loading states, error boundaries, retry logic
  • Mocking for local dev — a mock BFF server ships with your domain for offline development

Hands-on Exercise

Add a new API endpoint to the sandbox domain's BFF mock. Create a service function that calls it. Wire it to a page that displays the data using ML- components. Handle loading and error states.

Sign-off Criteria

  • Service function is typed and placed in /services
  • API call uses the standard fetch wrapper
  • Loading and error states are handled
  • Mock BFF endpoint is properly configured

Module 4: Local Development

What you'll learn: How to start your domain locally, mock the BFF, test in isolation, and iterate fast with hot reloading.

Module content Local setup, mock BFF, hot reload

Key Concepts

  • One-command setup — npm run dev starts your domain with mock BFF, component catalog, and hot reloading
  • Domain isolation — you develop in your domain only, without running the entire platform
  • Component catalog access — browse available ML- components in the local dev server
  • Mock BFF — the mock server provides realistic test data without needing the real backend
  • DevTools integration — custom DevTools panel shows domain state, component tree, and event log

Hands-on Exercise

Start the sandbox domain in local dev mode. Make a visible change to a page. Verify hot reload works. Open the component catalog and find an ML- component you haven't used before. Add it to your page.

Sign-off Criteria

  • Domain starts successfully with one command
  • New component is properly integrated and rendering
  • Mock BFF is serving data correctly
  • No console errors in development mode

Module 5: Deployment

What you'll learn: How to push your domain to the CDN, update the import map, and deploy via blue/green rollouts.

Module content CDN push, import map, blue/green

Key Concepts

  • Push to CDN — your domain's built assets are published to the CDN with a unique version hash
  • Import map update — the shell's import map is updated to point at the new version
  • Blue/green deployment — traffic shifts gradually from old version to new version
  • Instant rollback — revert the import map pointer to the previous version. Rollback takes seconds, not minutes.
  • Version pinning — each domain can pin its dependencies to specific component versions

Hands-on Exercise

Push your sandbox domain changes through the deployment pipeline. Watch the blue/green rollout in the staging environment. Verify your changes are live. Perform a rollback. Verify the rollback reverted your changes.

Sign-off Criteria

  • Domain deploys successfully to staging
  • Blue/green rollout completes without errors
  • Rollback restores previous version correctly
  • Import map reflects the correct version after each operation
Industry Standard: Blue/Green Deployments Blue/green deployment with instant rollback via import map versioning is the industry standard for micro frontend platforms. Zero-downtime deployments with sub-second rollback eliminate deployment risk.

Module 6: Debugging

What you'll learn: How to debug across Shadow DOM boundaries, trace events between components, and use DevTools effectively in the new architecture.

Module content Shadow DOM debugging, DevTools, event tracing

Key Concepts

  • Shadow DOM in DevTools — how to inspect elements inside Shadow DOM boundaries using Chrome DevTools
  • Event tracing — use the custom Event Log panel to trace Custom Events as they flow between components
  • Network tab for BFF calls — filter and inspect BFF requests using the domain's endpoint prefix
  • Component state inspection — inspect ML- component attributes and properties at runtime
  • Cross-domain debugging — when an issue spans domains, how to isolate which domain is the source
  • Performance profiling — identify slow components and unnecessary re-renders

Hands-on Exercise

A pre-built sandbox page has three intentional bugs: a broken attribute binding, a missing event listener, and a BFF endpoint returning incorrect data. Use DevTools to find all three, fix them, and push the fixes.

Sign-off Criteria

  • All three bugs identified and fixed
  • Fix descriptions documented in commit messages
  • No new bugs introduced
  • All existing tests pass

Module Progression

Each module builds on the previous one. By the end, you've exercised the complete domain developer workflow.

Domain Developer track: six modules with checkpoints leading to certification
Industry Standard: Progressive Skill Building Enterprise training programs build skills progressively—each module reinforces the previous one and introduces one new concept.
