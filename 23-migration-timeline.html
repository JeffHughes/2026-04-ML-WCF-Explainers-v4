The Migration Timeline: Phase by Phase

Four phases, each delivering independent value. You're better off after any phase — even if you stop. No big-bang cutover. No all-or-nothing gamble.

Four Phases, Each Standing Alone

The migration is structured as four sequential but independently valuable phases. Each phase delivers measurable improvements to the platform. If circumstances change and the migration pauses after any phase, the organization is still better off than before it started.

✅ The Key Guarantee Every phase delivers independently. Phase 1 alone reduces blast radius and enables independent deployments. Phase 2 removes the SSR bottleneck. Phase 3 gives visibility and governance. Phase 4 is continuous improvement that never "finishes" — and that's by design.

Phase 1: Domain Extraction (Q1–Q2)

The first phase uses the copy-split-strip technique to extract the three highest-impact domains from the monolith: Catalog (the pilot), then Cart, then Checkout. Each extraction produces an independently deployable micro frontend.

What's Delivered Concrete outcomes
  • 3 independent domain packages — Catalog, Cart, Checkout, each with their own repository, CI pipeline, and CDN deployment
  • Import map integration — each domain loadable via the existing shell through import map entries
  • Domain-scoped test suites — unit and integration tests for each extracted domain
  • Deployment independence — each domain can deploy without affecting the others
  • Rollback capability — import map versioning enables instant rollback per domain
What Changes Impact on teams
  • Catalog, Cart, and Checkout teams begin deploying independently
  • The monolith shrinks — these domains' code is removed from the shared codebase
  • Other domains continue operating exactly as before — zero impact
  • Shared components are still loaded from the monolith's bundle until Phase 2
  • The import map is introduced as the new deployment mechanism for extracted domains
Success Criteria How we know it's done
  • ✅ All three domains deploy independently via their own CI pipelines
  • ✅ No monolith build required for Catalog, Cart, or Checkout changes
  • ✅ Rollback demonstrated: domain reverted via import map in under 5 seconds
  • ✅ Domain test suites run in under 3 minutes each
  • ✅ User experience is identical — no visible change to end users
💡 Phase 1 Alone Is Valuable Even if the migration stopped here, the organization gains independent deployments for its three highest-traffic domains, reduced blast radius, and faster CI for those teams. This is not a "foundation that only pays off later" — it delivers immediate, measurable value.

Phase 2: Shell Swap (Q2–Q3)

Phase 2 replaces the monolith's server-side rendered shell with a new client-side rendered (CSR) shell. This removes the SSR bottleneck and establishes the standards-based component framework as the shell's technology. Existing React micro frontends are wrapped as Web Components for seamless integration.

What's Delivered Concrete outcomes
  • New CSR shell — lightweight, standards-based application shell that loads domains via import maps
  • React MFE wrapping — existing React micro frontends wrapped as Web Components, running inside the new shell
  • SSR removal — the server-side rendering layer is eliminated, reducing infrastructure complexity
  • Unified routing — shell manages top-level routing, domains handle their own internal routes
  • Shared layout — header, footer, and navigation managed by the shell using core Web Components
What Changes Impact on teams
  • All domains now load through the CSR shell — consistent experience
  • React domains continue working unchanged inside their Web Component wrappers
  • Page load model shifts from SSR to CSR — initial loads are client-rendered
  • The monolith's server component is no longer in the critical path for frontend rendering
  • New domains can be built directly using the standards-based component framework
Success Criteria How we know it's done
  • ✅ CSR shell serves all pages — SSR server decommissioned for frontend rendering
  • ✅ All existing React MFEs render correctly inside Web Component wrappers
  • ✅ Core Web Vitals (LCP, CLS, FID) remain within acceptable thresholds
  • ✅ New domain (pilot) built with WCFramework renders alongside React MFEs
  • ✅ No user-facing regressions — visual regression tests pass

Phase 3: App Catalog + BFF (Q3–Q4)

Phase 3 introduces the governance and visibility layer. The App Catalog tracks every component, the API scanner monitors every endpoint, and the BFF pattern decouples frontend from backend services. This is where the platform gains operational maturity.

What's Delivered Concrete outcomes
  • App Catalog — live registry of every component, domain, and their dependencies
  • API scanning — automated discovery and monitoring of all API endpoints across domains
  • BFF pattern — Backend-for-Frontend services deployed for pilot domains (Catalog, Cart)
  • Contract testing — consumer-driven contracts established between frontends, BFFs, and backends
  • Dependency graph — visual map showing which domains use which components
What Changes Impact on teams
  • Component authors publish to the App Catalog — consumers discover via the catalog UI
  • API changes are detected automatically — no more surprise breaking changes from backend teams
  • Frontend teams interact with their BFF, not directly with backend services
  • Test targeting becomes intelligent — only affected domains tested when a component changes
  • Governance happens automatically through the catalog validation pipeline
Success Criteria How we know it's done
  • ✅ App Catalog displays all registered components with documentation and examples
  • ✅ API scanner detects and reports endpoint changes within 24 hours
  • ✅ BFF deployed for Catalog and Cart domains — frontend decoupled from direct backend calls
  • ✅ Contract tests running in CI for all BFF-connected domains
  • ✅ Dependency graph accurately reflects component-to-domain relationships

Phase 4: Progressive Evolution (Q4+)

Phase 4 is different from the others — it never "finishes". This is the ongoing phase where teams upgrade components at their own pace, new domains are onboarded, and the platform continuously improves. It's not a project; it's a way of working.

What's Delivered Continuous outcomes
  • Component upgrades — teams replace React components with WCFramework equivalents on their own schedule
  • New domain onboarding — additional domains extracted from the monolith using the proven copy-split-strip process
  • Core component evolution — annual cadence delivers new shared components and improvements
  • Performance optimization — bundle sizes decrease as framework-specific wrappers are removed
  • Developer experience — tooling, documentation, and catalog improvements based on team feedback
What Changes Impact on teams
  • Each team chooses their own evolution pace — no organizational pressure
  • Last year's components are fine. Two years ago is also fine. The CDN serves all versions.
  • New hires onboard faster — WCFramework requires only HTML, CSS, and JavaScript knowledge
  • The platform team shifts from "building the migration" to "maintaining the platform"
  • Cross-team coordination becomes the exception, not the rule
Success Criteria Ongoing metrics
  • ✅ New domains onboarded within 2 weeks using established patterns
  • ✅ Component adoption tracked via App Catalog — upward trend over time
  • ✅ Average deploy frequency per domain: weekly or better
  • ✅ Mean time to rollback: under 30 seconds
  • ✅ Developer satisfaction surveys show improvement in deployment confidence

Cumulative Value Delivery

Each phase adds to the cumulative value delivered to the organization. The curve is front-loaded — Phase 1 delivers the most dramatic improvement (independent deployments), and each subsequent phase adds governance, visibility, and continuous improvement on top.
