The CSR Shell: How It Works

A lightweight client-side rendered application that composes domains through browser-native Web Components and import maps

What the Shell Is (and What It Isn’t)

The shell is the thinnest possible application that ties everything together. It handles routing, authentication, shared navigation, and domain composition. It does not contain any business logic, product data, cart state, or checkout flows. Think of it as a picture frame — it holds the artwork (domains) but it isn’t the artwork.

✅

Shell Owns

Routing, authentication, environment config, shared navigation (header/footer), event bus, import map resolution, error boundaries, feature flags.

❌

Shell Does NOT Own

Business logic, domain data, domain components, product APIs, cart state, checkout flow, user account management, search logic, CMS content.

💡 The 50-Line Rule If you can’t explain what the shell does in 50 lines of pseudocode, it’s too complex. The shell’s route handler is essentially: look up the URL in the import map → load the domain module → render the custom element → done. Everything else is authentication and shared chrome (header, footer).

How Routing Works

The shell uses a simple route table that maps URL patterns to domain modules. When the URL changes (navigation, browser back/forward, direct entry), the shell:

  1. Matches the URL against the route table
  2. Checks authentication — if the route requires auth and the user isn’t logged in, redirect to login
  3. Resolves the module via the import map — /products/* maps to @ml/catalog-plp
  4. Dynamically imports the module — import('@ml/catalog-plp') loads from CDN
  5. Renders the custom element — creates <ml-catalog-plp> and inserts it into the DOM outlet
  6. Passes route parameters as attributes — <ml-catalog-plp category="gloves">

Route Table Example

// Shell route configuration — maps URL patterns to domain modules
const routes = [
  { path: '/products/*',      module: '@ml/catalog-plp',       element: 'ml-catalog-plp',       auth: false },
  { path: '/product/:sku',    module: '@ml/catalog-pdp',       element: 'ml-catalog-pdp',       auth: false },
  { path: '/category/:slug',  module: '@ml/catalog-category',  element: 'ml-catalog-category',  auth: false },
  { path: '/search',          module: '@ml/search-results',    element: 'ml-search-results',    auth: false },
  { path: '/cart',            module: '@ml/cart-page',         element: 'ml-cart-page',         auth: true  },
  { path: '/checkout/*',      module: '@ml/checkout-flow',     element: 'ml-checkout-flow',     auth: true  },
  { path: '/account/*',       module: '@ml/account-dashboard', element: 'ml-account-dashboard', auth: true  },
  { path: '/',                module: '@ml/cms-homepage',      element: 'ml-homepage',          auth: false },
  { path: '/*',               module: '@ml/cms-page',          element: 'ml-cms-page',          auth: false },
];
Industry Standard: W3C Custom Elements + Shadow DOM + HTML Templates The composition boundary between the shell and domain modules is entirely based on W3C standards. Custom Elements provide the registration API, Shadow DOM provides style/DOM encapsulation, and HTML Templates provide the declarative rendering model. Every modern browser supports these natively — no polyfills needed.

The Import Map

The import map is a JSON configuration that tells the browser where to find each module. It’s a W3C standard supported natively in all modern browsers. The shell loads the import map at startup, and every import() call resolves through it.

<script type="importmap">

  "imports": {
    "@ml/catalog-plp":       "https://cdn.medline.com/domains/catalog-plp@3.1.0/index.js",

Why Import Maps Matter

Benefit Explanation
Independent versioning Each domain module has its own version. Update Catalog to 3.2.0 without touching Cart 2.0.0.
No coordinated builds Domains build and publish independently. The import map is the integration point, not a shared build step.
Instant rollback Change the import map entry to point to the previous version. The next page load uses the old code. No rebuild needed.
CDN-cached Each versioned module URL is immutable and cacheable forever. catalog-plp@3.1.0/index.js never changes.
Browser-native No bundler plugin, no custom module loader, no framework-specific federation. The browser handles it natively.

Performance Characteristics

  • Shell JS — Tiny (<30KB gzipped). Cached by CDN. Loads in milliseconds.
  • Import map resolution — Inline JSON in the HTML. Zero network requests.
  • Domain module loading — Versioned URL, cached at CDN edge. First visit downloads, subsequent visits serve from cache.
  • BFF data fetch — Only network request that’s not cache-friendly (dynamic data). This is the critical path.
  • Rendering — Domain module renders directly into the DOM. No server-side rendering step, no hydration mismatch risk.

Shell Responsibilities in Detail

Routing & Navigation URL matching, history management, route guards

The shell uses the browser’s native History API (pushState, popstate) for client-side navigation. When the URL changes:

  1. The route matcher finds the matching pattern
  2. Auth guard checks if the user has permission
  3. The current domain element is unmounted (custom element disconnectedCallback)
  4. The new domain module is loaded (or served from module cache)
  5. The new domain element is mounted with route parameters as attributes

Navigation between pages within the same domain (e.g., PLP to PDP) is handled by the domain itself — the shell only intervenes when the domain changes.

Authentication Session management, token refresh, permission context

The shell owns the authentication lifecycle:

  • On startup: Check for existing session token in secure cookies
  • If valid: Set auth context available to all domains via <ml-auth> shared component
  • If expired: Attempt silent token refresh. If that fails, redirect to login.
  • On logout: Clear session, emit ml:auth-logout event, redirect to home.

Domain modules access auth context through a shared interface — MLAuth.getUser(), MLAuth.getToken(), MLAuth.hasPermission('checkout'). They never manage tokens directly.

Event Bus Cross-domain communication without coupling

The shell provides a centralized event bus built on CustomEvent. It adds:

  • Event logging — Every event is logged for debugging (dev mode only)
  • Event replay — When a domain module loads late (lazy), it can request replay of events it missed
  • Event typing — Event names are namespaced (ml:domain:action) and payloads are validated in development mode
  • Dead letter handling — If an event has no subscribers, it’s logged as a warning (possible misconfiguration)
Environment Configuration API URLs, feature flags, CDN paths

The shell provides environment-specific configuration to all domains:

// Available to all domains via window.__ML_ENV__
{
  apiBase: 'https://api.medline.com',
  cdnBase: 'https://cdn.medline.com',
  environment: 'production',
  region: 'us-east-1',
  featureFlags: { /* from flag service */ },
  analytics: { trackingId: '...' }
}

Domains read this configuration at initialization. They never hardcode environment-specific URLs.

Error Boundaries Graceful failure when a domain module crashes

If a domain module fails to load (CDN outage) or throws an unhandled error during rendering, the shell catches it and displays a friendly error boundary:

  • Shows a “This section is temporarily unavailable” message
  • The rest of the page (header, footer, other domain modules) continues to work
  • Error is reported to the monitoring service
  • User can retry loading the failed domain with a single click

This is a critical safety property: one domain crashing never takes down the entire page. Because the shell controls rendering boundaries via Shadow DOM, there are no more weird UI flicker issues caused by competing MFE stylesheets or hydration mismatches.

✅ If the Shell Is Big, Something Is Wrong The moment you find yourself putting product logic, cart calculations, or checkout validation in the shell, stop. That code belongs in a domain module. The shell is intentionally anorexic. Its only job is composition and cross-cutting concerns.
