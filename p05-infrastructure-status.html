✅ CDN Pipeline

Every component — wrapped React or native WCFramework — is automatically built, versioned, and published to the CDN when merged to its domain’s main branch. No manual deployment steps.

Pipeline Details How the CDN pipeline works

Build Flow

  1. Developer merges PR to domain’s main branch
  2. CI runs unit tests, integration tests, and contract tests
  3. If all pass: component is built, tree-shaken, and minified
  4. Versioned artifact published to CDN: cdn.medline.com/mfe/{domain}@{version}/
  5. Import map is automatically updated with new version
  6. CDN cache is invalidated for the updated module

Key Metrics

  • Average build-to-CDN time: 3 minutes 42 seconds
  • CDN edge propagation: < 30 seconds globally
  • Rollback time: < 15 seconds (import map version revert)
Industry Standard: Immutable Artifact Deployment Every CDN artifact is immutable and versioned. Rollback means pointing the import map to the previous version — no rebuild, no redeploy. Same pattern used by npm, Docker registries, and Cloudflare Pages.

✅ Import Map Management

The import map is the single source of truth for which version of every domain module is loaded by the shell. Updates are automated; rollbacks are instant.

Import Map Lifecycle Version management details

Automated Version Updates

When a domain publishes a new version to CDN, the import map service updates the corresponding entry. The shell fetches the import map on page load, so users get the new version on their next navigation.

Rollback Procedure

The import map service maintains a history of all versions. Rollback is a single API call that reverts the entry to the previous version. The CDN still has the old artifact (immutable), so rollback takes effect in < 15 seconds.

Canary Support (Planned)

The import map service will support percentage-based routing: serve the new version to 5% of users, validate metrics, then promote to 100%. Infrastructure is ready; the canary logic is planned for Q2.

✅ API Scanner

The API scanner runs against all BFF endpoints in staging on every deployment. It discovers endpoints, extracts OpenAPI contracts, builds dependency graphs, and flags breaking changes before they reach production.

Scanner Capabilities What the scanner does

Discovery

Scans all running BFF instances, discovers endpoints via OpenAPI spec (if available) or traffic analysis. Currently tracking 23 endpoints across 4 BFF services.

Contract Extraction

Generates OpenAPI specs for every endpoint, including request/response schemas, required fields, and authentication requirements. Specs are committed to the ml-api-contracts repository.

Breaking Change Detection

Compares new contract against the baseline. Flags: removed fields, type changes, new required fields, removed endpoints. Integrated as a CI gate — PRs with breaking changes are blocked until reviewed.

Performance

  • Full scan time: 2 minutes 18 seconds
  • False positive rate: 4.3% (1 false positive per 23 endpoints)
  • Runs on: every PR to domain main branch + nightly full scan

✅ Contract Testing

Consumer-driven contract tests run in every domain’s CI pipeline. If a domain’s BFF changes break another domain’s expectations, the build fails before merge.

Contract Testing Details How contracts are validated

How It Works

  1. Each domain declares what it expects from other domains’ BFFs (consumer contracts)
  2. Provider domains run these consumer contracts as part of their CI
  3. If a provider change violates a consumer contract, the provider’s build fails
  4. Teams negotiate contract changes via PR review

Current Coverage

  • Catalog → Search: 4 contract tests (product search, typeahead, facets, suggestions)
  • Cart → Catalog: 3 contract tests (product details, pricing, availability)
  • Mini-Cart → Cart: 2 contract tests (item count, cart total)
  • Checkout → Cart: 5 contract tests (full cart, promotions, shipping, tax, payment)
Industry Standard: Consumer-Driven Contracts Consumer-driven contract testing (Pact pattern) is the standard approach for ensuring micro services and micro frontends don’t break each other. Teams move fast independently while contracts catch integration failures in CI.
