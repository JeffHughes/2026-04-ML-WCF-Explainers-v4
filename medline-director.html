1 Where You Are Today Monolithic Next.js 12.3 shell with 84+ coupled micro frontends

Your current architecture is a monolithic Next.js 12.3 shell serving 84+ micro frontends through a tightly coupled SSR rendering pipeline. This is the architecture pattern that made sense when the platform was smaller—but at enterprise scale, it has become the primary constraint on delivery speed, team autonomy, and platform reliability.

Current Pain Points

  • Tightly coupled SSR rendering — Every micro frontend shares the same server-side rendering process. One team's memory leak or slow API call degrades the entire site.
  • Shared runtime = shared risk — A single team's change can break every other team's UI. Version conflicts, dependency collisions, and CSS leaks are weekly events.
  • React 17.x components with no common standards — 84+ micro frontends, each with their own patterns, their own state management, their own build configurations. No catalog. No discoverability.
  • Every update requires full regression — Because everything is coupled, nothing deploys in isolation. A simple button color change requires the full QA cycle.
  • Monolith release train — All 84+ MFEs deploy together. The fastest team moves at the speed of the slowest.
Industry Standard: Micro Frontend Independence The W3C Custom Elements specification and Shadow DOM standard were designed specifically to solve runtime isolation problems in large-scale web applications. Every major browser vendor has committed to these standards indefinitely.
Current Architecture — Monolithic Next.js shell with all 84+ MFEs sharing a single SSR runtime
The Core Problem In this architecture, independence is an illusion. Every team shares the runtime, shares the deployment pipeline, and shares the risk. The only way to achieve true domain independence is to break the monolith apart—safely, incrementally, without a rewrite.
2 The Migration Strategy: Copy, Split, Strip The safest migration pattern in enterprise software

The safest possible migration is one where nothing changes on day one. Copy-split-strip is the enterprise standard for decomposing monoliths because it eliminates the "big bang rewrite" risk entirely.

How It Works

  1. Copy — Take the entire monolith codebase and duplicate it. This is your new domain candidate. It runs. It's the same code. Zero risk.
  2. Split — Rename the copy to its domain identity (e.g., "Catalog"). Configure it as its own deployable unit in K8s.
  3. Strip — Progressively remove everything the domain doesn't need. Cart code comes out of Catalog. Checkout code comes out of Catalog. What remains is a focused, domain-specific application.

Why This Is Zero Risk

  • The original monolith continues running, unchanged, serving all traffic
  • The copied domain runs alongside it—same code, same behavior
  • Traffic shifts gradually via routing rules (canary, blue/green)
  • At any point, you can route 100% back to the original monolith
  • This is the classic strangler fig pattern—the industry-standard approach for monolith decomposition
Industry Standard: Strangler Fig Pattern Originally described by Martin Fowler, the strangler fig pattern is the gold standard for enterprise migration. It has been used at Amazon, Netflix, Spotify, and every other enterprise that has successfully decomposed a monolith without downtime.

Concrete Example

Take the Catalog domain:

  1. Copy the entire Next.js monolith → medline-catalog
  2. Deploy medline-catalog to K8s as its own SSR service (same code, runs identically)
  3. Route /catalog/* traffic to the new service
  4. Begin stripping Cart, Checkout, Account, and other non-Catalog code from the copy
  5. The Catalog team now owns their deployment pipeline end-to-end
Copy-Split-Strip Flow — The monolith is cloned per domain, then trimmed down to only what that domain needs
Key Insight You never stop running the original monolith until you choose to. There is no flag day. There is no cutover weekend. Domains peel off one at a time, at whatever pace makes sense for your teams.
3 Domain Extraction: One Vertical at a Time Each e-commerce domain becomes its own deployable unit

The 84+ micro frontends in today's monolith represent distinct business domains. Each domain should own its own deployment lifecycle, its own release cadence, and its own team structure. Domain extraction makes this real.

E-Commerce Domain Map

  • Catalog / PLP — Product listing pages, category navigation, filtering, sorting
  • PDP (Product Detail) — Product pages, specifications, images, reviews
  • Search — Site search, typeahead, faceted search, search analytics
  • Cart / Mini-Cart — Shopping cart, quantity management, saved carts
  • Checkout — Checkout flow, payment, shipping, order confirmation
  • Account — User profiles, order history, lists, preferences
  • Content / CMS — Landing pages, promotional content, banners
  • Navigation / Shell — Global header, footer, mega-menu, breadcrumbs
  • Analytics — Tracking, event collection, A/B testing integration
  • Shared Components — Design system components used across domains

What Domain Independence Means

  • Each domain deploys independently—Catalog ships without waiting for Checkout
  • Teams own their domain end-to-end: code, tests, deployment, monitoring
  • Domain boundaries are explicit, not implicit
  • Cross-domain communication happens through well-defined contracts, not shared state
  • Drift between domains is expected and managed (covered in Section 7)
Industry Standard: Domain-Driven Design Domain-driven micro frontend architecture aligns with the bounded context patterns described by Eric Evans. Each domain is a bounded context with explicit interfaces. This is the pattern used by Amazon, Zalando, IKEA, and every other enterprise e-commerce platform operating at scale.
Domain Map — Each e-commerce domain as an independent deployable unit with explicit contracts
Team Autonomy The Catalog team can ship three times a day if they want. The Checkout team can take two weeks for a careful release. Neither is blocked by the other. This is the fundamental shift from monolith thinking to domain thinking.
4 Shell Swap: Next.js → WCFramework CSR Shell Lightweight client-side composition replaces server-side monolith

Once domains are extracted, the Next.js monolith shell is no longer needed. It gets replaced by a lightweight CSR (Client-Side Rendering) shell that composes micro frontends at the browser level using W3C Web Components.

Why CSR Over SSR for Composition

  • Separation of concerns — The BFF handles data fetching and API orchestration. The shell handles UI composition. Clean boundary.
  • No shared server resources — SSR-first architectures that made sense for small teams doing SEO-focused content sites become a bottleneck when 84 micro frontends are competing for the same server-side rendering pipeline.
  • Simpler deployment model — A CSR shell is static HTML/JS/CSS on a CDN. No Node.js servers to scale. No SSR memory leaks to debug.
  • True runtime isolation — Web Components with Shadow DOM provide browser-native encapsulation. No CSS leaks. No global state collisions.

How the Shell Works

  1. The CSR shell loads as a lightweight HTML page from CDN
  2. An import map defines which micro frontends to load and from where
  3. Each micro frontend is a Web Component, loaded dynamically
  4. Existing React 17 components are wrapped as Web Components—they work inside the new shell immediately
  5. The shell provides routing, layout, and cross-domain event bus—nothing else
Industry Standard: W3C Custom Elements & Shadow DOM Custom Elements (v1) and Shadow DOM (v1) are W3C standards implemented natively in every modern browser. They provide component registration, lifecycle hooks, and style encapsulation without any framework dependency. This is the browser's built-in component model.
The SSR Counterargument SSR is valuable for SEO-critical content pages and initial load performance. For authenticated e-commerce experiences with 84+ micro frontends, the SSR overhead creates more problems than it solves. The BFF pattern (Section 8) provides all the data performance benefits of SSR without coupling the rendering pipeline.
Before & After — SSR monolith replaced by CDN-served CSR shell with domain-specific BFFs
5 Component Transition: React 17 → HTML-First MV* Components Gradual migration from React to standards-based components

You don't rewrite 84+ micro frontends on day one. The component transition is progressive, running at each team's pace. Existing React components stay, wrapped as Web Components. New components use the WCFramework HTML-first pattern.

The Three-Stage Component Lifecycle

  1. Stage 1: React as-is — Existing React 17 components run inside the CSR shell, wrapped in a thin Web Component adapter. No code changes required.
  2. Stage 2: React wrapped — React components are formally registered as Web Components with proper Shadow DOM encapsulation. They participate in the component catalog.
  3. Stage 3: WCFramework native — When a component needs a major rewrite anyway, it's rebuilt using the WCFramework HTML-first MV* pattern. Pure standards. No framework dependency.

WCFramework Component Pattern

  • ML- prefix — All Medline components use the ml- prefix (e.g., <ml-product-card>, <ml-cart-summary>, <ml-search-bar>)
  • Single module per component — Each component is a self-contained ES module, loaded dynamically from CDN
  • HTML-first — Components are authored in HTML templates with declarative bindings. No JSX compilation step.
  • MV* pattern — Model-View-Whatever: the component defines its data model, its template, and its behavior. No prescribed state management library.
  • Shadow DOM encapsulation — Every component's styles and DOM are isolated from the rest of the page. No CSS leaks, ever.
Industry Standard: HTML Templates & ES Modules HTML Templates and ES Modules are W3C/WHATWG standards supported by every browser. Combined with Custom Elements and Shadow DOM, they form a complete component model with zero framework dependencies. This is the platform itself as the framework.
Component Lifecycle — Progressive transition from React 17 to standards-based WCFramework components
No Flag Day React components and WCFramework components coexist in the same shell, on the same page. A product listing page could have <ml-product-card> (WCFramework) next to a React-wrapped filter component. Both are Web Components to the shell. Both work.
6 The App Catalog: Your Single Source of Truth Every component, version, and dependency tracked and discoverable

With 84+ micro frontends splitting into independent domains, you need a single place that knows everything: what components exist, what version each is running, what data contracts each consumes, and what depends on what. That's the App Catalog.

What the App Catalog Tracks

  • Component registry — Every ml-* component, every React-wrapped component, every shared utility. Versioned, tagged, searchable.
  • Deployment status — Which version is deployed to which environment (Dev, QA, UAT, Prod) for every domain.
  • Dependency graph — What depends on what. If ml-product-card@2.3.1 changes its API, the catalog knows exactly which pages and domains are affected.
  • Data contracts — API scanning discovers data shapes automatically. The catalog knows the contract between every BFF and every component.
  • Breaking change detection — Before deployment, the catalog compares the new version's API surface against all consumers. Breaking changes are flagged automatically.
  • Health metrics — Bundle sizes, load times, error rates, usage frequency—all visible in one place.

Discovery Over Documentation

The App Catalog doesn't rely on teams keeping documentation up to date. It scans. It discovers. It tracks automatically. When a new component is published to CDN, the catalog picks it up. When an API contract changes, the catalog detects it.

Industry Standard: Software Catalog (Backstage Pattern) Centralized software catalogs are the industry standard for managing micro frontend and microservice architectures at scale. Spotify's Backstage, which popularized this pattern, is now a CNCF incubating project used by hundreds of enterprises. The App Catalog extends this pattern with component-level granularity and automated contract scanning.
App Catalog — Component registry with version tracking, dependency graphs, and automated contract scanning
The Question It Answers "What version of the product card is running in production, who published it, when, and what will break if we upgrade?" The App Catalog answers this in seconds, not hours of Slack threads.
7 Managing Drift: Automation Over Coordination Domains will drift — that's expected and managed automatically

Once domains split and teams own their own release cadence, they will drift. Different domains will run different versions of shared components. Different teams will adopt new patterns at different speeds. This is not a bug. This is the feature.

Why Drift Is Good

  • Team autonomy — Teams that are ready to upgrade do so. Teams that are mid-sprint on a critical feature don't get interrupted.
  • Risk isolation — If the new version of a shared component has a subtle issue, only the domains that adopted it are affected. Not all 84.
  • Natural testing — Early adopter domains serve as canaries for the rest of the organization.

The 1-Year Core Cadence

  • Core framework and shared components publish one major version per year
  • Two component sets published yearly: one in WCFramework format, one in React wrapper format
  • Domains have 12 months to adopt the new version at their own pace
  • Automated change-mod scripts handle the mechanical parts of upgrades (import paths, API signature changes, deprecation replacements)
  • Small sites and small teams can move fast. Large domains can take their time. Or stay on last year's stack—that's fine too.

How the App Catalog Manages Drift

  • The catalog shows exactly which version each domain is running
  • It provides a clear target: "here's the latest, here's what changed, here's the automated script to get there"
  • Drift metrics are visible to leadership without requiring manual reporting
  • No surprise breaks—the 1-year cadence means teams plan for upgrades, not react to them
Industry Standard: Managed Dependency Cadence Enterprise teams of 50-200 developers cannot coordinate simultaneous upgrades across 84+ deployable units. The managed cadence pattern (annual major, quarterly minor, weekly patch) provides predictability without forcing lockstep coordination. This is the same pattern used by the Java ecosystem (LTS releases), .NET (annual cadence), and every successful enterprise platform.
Version Timeline — 1-year core cadence with domains adopting at their own pace
Automation Over Coordination The change-mod scripts do the mechanical work. The App Catalog tracks the progress. Leadership sees a dashboard, not a spreadsheet maintained by a program manager. Drift is a solved problem when you have the right tooling.
8 The BFF Pattern: Backend Isolation Backend-for-Frontend isolates every domain from backend complexity

The Backend-for-Frontend (BFF) pattern gives each domain its own backend layer. Frontend teams never touch backend services directly. The BFF is the boundary between frontend concerns (presentation, interaction) and backend concerns (business logic, data, integrations).

How the BFF Works

  • Each domain has its own BFF service
  • The BFF aggregates data from multiple backend services into exactly the shape the frontend needs
  • The BFF handles authentication, caching, and error transformation
  • Frontend components call their domain's BFF and only their domain's BFF
  • Backend service changes are absorbed by the BFF layer, not by the frontend

API Contract Scanning

  • The App Catalog's API scanner automatically discovers data shapes between BFFs and backend services
  • Contract changes are detected before deployment
  • Breaking changes trigger automated alerts to affected domain teams
  • Contract tests run in CI/CD to validate compatibility

Deployment Model

  • BFFs deploy independently from both frontend and backend
  • Blue/green deployments at the BFF level provide instant rollback
  • Each BFF can be scaled independently based on its domain's traffic patterns
  • The Catalog BFF handles thousands of product queries; the Account BFF handles fewer, more complex transactions
Industry Standard: Backend-for-Frontend The BFF pattern was pioneered by Sam Newman at ThoughtWorks and is now the standard architecture for any enterprise with multiple frontend experiences (web, mobile, kiosk) consuming shared backend services. It eliminates the "one API to rule them all" problem that creates coupling between frontend and backend teams.
BFF Architecture — Each domain's frontend communicates exclusively with its own BFF, which orchestrates backend calls
Backend Changes Don't Break the Frontend When the Product Service team refactors their API, only the Catalog BFF and Cart BFF need to adapt. The <ml-product-card> component doesn't change at all. This is isolation at every layer.
9 Testing, Rollouts, and Blue/Green Deployments Each domain deploys independently with instant rollback

Independent domains mean independent deployments. Each domain can roll out changes, validate them, and roll back without affecting any other domain. No more "wait for the release train."

Blue/Green at the Domain Level

  • Each domain maintains two production slots: blue (current) and green (new version)
  • New version deploys to the green slot and gets validated
  • Traffic shifts from blue to green when validation passes
  • If anything goes wrong, traffic routes back to blue instantly
  • This is per-domain, not per-site. Catalog can be mid-rollout while Checkout is stable on its current version.

Import Map Rollback

For the CSR shell, rollback is even simpler. The import map defines which component version loads. Rolling back means updating one line in the import map to point to the previous version on CDN. The previous version is still there—immutable artifacts never get deleted.

Contract Testing

  • Component contract tests — Verify that each Web Component's public API (attributes, events, slots) matches its documented contract
  • BFF contract tests — Verify that each BFF's response shape matches what the frontend components expect
  • Cross-domain integration tests — Verify that domain interactions (e.g., "add to cart" from PDP) work correctly across boundaries
  • Visual regression tests — Automated screenshot comparison catches unexpected UI changes
Industry Standard: Immutable Artifacts + Import Maps Immutable deployment artifacts on CDN are the gold standard for frontend deployment. Combined with browser-native import maps (now supported in all major browsers), they provide declarative version management with instant rollback. This eliminates the need for complex deployment orchestration.
Blue/Green Deployment — Per-domain deployment with instant import map rollback
Rollback in Seconds, Not Hours Traditional monolith rollback means redeploying the entire application. Domain-level rollback means changing one entry in an import map. The old version is already on CDN. The switch is instant.
10 Why This Pattern Wins for Enterprise Standards-based longevity vs. framework churn

The patterns that work for small teams who throw out and start over every 6-18 months do not work for enterprise teams of 50-200 developers managing platforms that need to last 10-15 years. This is the fundamental disconnect in the current frontend ecosystem.

The Enterprise Reality

  • Frameworks with 18-month major version cycles create perpetual migration tax. Every major version means rewriting, retesting, and redeploying. With 84+ micro frontends, that migration tax is enormous.
  • Library-specific hiring pools are narrow and expensive. Developers who know "React 18 with Server Components and the App Router" are a subset of developers who know React, who are a subset of developers who know JavaScript.
  • Framework lock-in is invisible until it isn't. When the framework community moves on (jQuery, Backbone, Ember), you're left maintaining a legacy ecosystem with a shrinking talent pool.

The Standards Advantage

  • W3C Web Components are browser-native — Backed by every browser vendor (Google, Apple, Mozilla, Microsoft). Not going away. Not changing their API every 18 months.
  • No proprietary framework lock-in — Components are standard HTML elements. Any developer who knows HTML, CSS, and JavaScript can be productive.
  • 10-15 year component lifespans — Web Components written today will work in browsers 15 years from now, just like HTML written in 2010 still works today.
  • The hiring pool is everyone — The platform IS the framework. HTML, CSS, JavaScript. Every web developer already knows the stack. Onboarding time drops from weeks to days.
Industry Standard: Platform-as-Framework The web platform itself provides Custom Elements, Shadow DOM, HTML Templates, ES Modules, Import Maps, CSS Custom Properties, and the Fetch API. These are the building blocks of enterprise component development. They are maintained by the browser vendors, standardized by the W3C, and backward-compatible by design. No framework in history has matched this longevity commitment.
Framework Churn vs. Standards Longevity — 18-month migration cycles vs. 10-15 year component lifespans
The Simple Test Will the components you build today still work in a browser 10 years from now? With Web Components, the answer is yes—because the browser vendors guarantee backward compatibility. With framework-specific components, the answer depends on whether the framework community still exists in 10 years.
11 The Migration Timeline Four phases, each delivering value independently

The migration is structured as four overlapping phases. Each phase delivers value independently—you don't need to complete all four to see benefits. Phase 1 alone gives teams domain ownership and independent deployment.

Phase 1: Domain Extraction (Copy-Split-Strip)

  • Copy the monolith for each domain
  • Deploy domain copies as independent K8s services (still SSR, same code)
  • Route traffic per-domain
  • Strip non-domain code from each copy
  • Outcome: Teams own their deployment. Domains deploy independently. The monolith still runs as a fallback.

Phase 2: Shell Swap

  • Introduce the WCFramework CSR shell alongside the SSR domains
  • Wrap existing React micro frontends as Web Components
  • The CSR shell serves wrapped MFEs—same functionality, new composition model
  • Outcome: Client-side composition in place. SSR rendering dependency removed. CDN-served shell.

Phase 3: Component Catalog + BFF Integration

  • Stand up the App Catalog with API scanning
  • Register all components (React-wrapped and native) in the catalog
  • Introduce BFF services for each domain
  • Migrate frontend components from direct backend calls to BFF calls
  • Outcome: Full visibility into the component ecosystem. Backend isolation complete.

Phase 4: Progressive Component Evolution

  • As teams do major rewrites, they build with WCFramework patterns instead of React
  • New components use the ml- prefix and HTML-first pattern
  • Existing React components continue working—no forced migration
  • The 1-year core cadence provides the framework for gradual adoption
  • Outcome: Standards-based component library growing organically. Framework dependency shrinking naturally.
Migration Timeline — Four overlapping phases, each delivering value independently
Industry Standard: Incremental Migration Every successful enterprise platform migration in history has been incremental. Amazon's migration from monolith to microservices took years. Netflix's migration took years. The key is that each phase delivers value and each phase is reversible. There is no "Phase 3 fails and everything falls apart" scenario.
Each Phase Pays for the Next Phase 1 (domain extraction) immediately gives teams deployment independence. That speed improvement justifies Phase 2. Phase 2 (shell swap) simplifies the infrastructure, which makes Phase 3 straightforward. There's no "big investment now, payoff later" gamble.
12 What Changes for Your Teams Domain ownership, faster onboarding, and no more waiting

This migration changes how teams work, how they deploy, and how fast new developers become productive. Here's what's concretely different.

Deployment Cadence

  • Before: All 84+ MFEs deploy together on a release train. A single broken test blocks everyone.
  • After: Each domain deploys independently. The Catalog team can ship three times a day. The Checkout team can take two weeks. Neither blocks the other.

Team Ownership

  • Before: Teams "own" their micro frontend but share the runtime, share the build, share the deployment. Ownership is nominal.
  • After: Teams own their domain end-to-end: code, build, tests, deployment, monitoring, BFF. True ownership with true accountability.

Onboarding Speed

  • Before: New developers need to understand the Next.js monolith, the shared runtime, the build system, the deployment pipeline, and the implicit dependencies between 84 micro frontends.
  • After: New developers learn their domain's components and their domain's BFF. The component catalog tells them what exists. The standards-based stack (HTML, CSS, JS) is what they already know.
  • Offshore resources who know HTML can be productive on day one. The WCFramework pattern is HTML templates and JavaScript. No JSX. No framework-specific idioms to learn.

Upgrade Path

  • Before: Framework major version upgrades are multi-month, all-hands projects that touch every MFE.
  • After: 1-year core cadence with automated change-mod scripts. Each team upgrades at their own pace. The catalog tracks who's on what version.

Discoverability

  • Before: "Does a product card component already exist? What version? Where's the code? What data does it need?" — Slack threads, tribal knowledge, stale docs.
  • After: The App Catalog. Search for any component, see every version, see every consumer, see the data contract. Seconds, not hours.
Team Impact — From shared release train to independent domain deployment
The Bottom Line for Teams Less waiting. More shipping. Clear ownership. Better tools. And a technology stack that doesn't require a framework migration every 18 months. This is what domain independence looks like in practice.

Key Principles

Six foundational principles that drive every decision in this migration strategy.

🛡

Zero-Risk Migration

Copy, don't rewrite. The original monolith stays running until you choose to decommission it. Every step is reversible. The strangler fig pattern is the safest migration strategy in enterprise software.

🎯

Domain Independence

Ship on your own schedule. Each domain owns its code, its deployment pipeline, its release cadence, and its BFF. No more waiting for the monolith release train. No more "one team's change breaks everyone."

🌐

Standards Over Frameworks

W3C, not React. Not Vue. Not any framework. Web Components are browser-native, vendor-maintained, and backward-compatible by design. The platform is the framework. It will be here in 15 years.

⚙

Automated Governance

The App Catalog tracks every component. API scanning discovers contracts. Change-mod scripts automate upgrades. Breaking change detection prevents deployment surprises. Automation over coordination, always.

🔒

Backend Isolation

The BFF pattern ensures frontend teams never touch backend services directly. Backend changes are absorbed by the BFF layer. Frontend components only know their domain's BFF contract. Clean separation at every layer.

⏳

Enterprise Longevity

10-15 year component lifespans without framework migrations. 1-year core cadence with automated upgrade tooling. Developers who know HTML, CSS, and JavaScript are immediately productive. The hiring pool is everyone.
