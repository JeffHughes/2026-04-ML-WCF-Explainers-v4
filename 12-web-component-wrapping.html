Wrapping React Micro Frontends in Web Components

The bridge pattern — add a standard interface around existing code without changing a single line of React

The Core Idea

Every existing React micro frontend gets a thin Web Component wrapper. The wrapper creates a Shadow DOM boundary, mounts the React app inside it, and exposes a standard Web Component API (attributes in, events out). From the shell’s perspective, it’s just a <ml-catalog-plp> custom element. What’s running inside — React, vanilla JS, or anything else — is an internal implementation detail.

✅ This Is NOT a Rewrite The React code inside the wrapper doesn’t change. Not a single component, not a single state hook, not a single API call. The wrapper is an adapter that gives existing React code a standards-based interface. It’s the difference between remodeling a house and putting a new front door on it.

Wrapper Architecture

Wrapper architecture — the React application runs unchanged inside a Shadow DOM boundary, communicating via attributes and events

How It Works Step by Step

  1. The shell navigates to /products/gloves
  2. The shell creates <ml-catalog-plp category="gloves"> and inserts it into the DOM
  3. The custom element’s connectedCallback fires
  4. The wrapper creates a Shadow DOM root
  5. The wrapper injects required styles into the Shadow DOM (scoped, isolated)
  6. The wrapper calls the React render function, mounting the React app into the Shadow DOM
  7. The React app renders exactly as it always has — components, state, API calls, everything
  8. When the React app needs to communicate outward (e.g., user clicks “Add to Cart”), it calls a callback that dispatches a CustomEvent
  9. When the shell updates attributes (e.g., user navigates to a different category), the wrapper’s attributeChangedCallback passes new props to React

The Wrapper Code

Here’s the complete wrapper pattern. This is the only new code needed to make a React MFE work in the standards-based shell:

// ml-catalog-plp.js — Web Component wrapper for the Catalog PLP React app
import { render, unmount } from './react-app/index.js'; // existing React entry point

class MLCatalogPLP extends HTMLElement {
  // Declare which attributes the shell can pass
  static get observedAttributes() {
    return ['category', 'page', 'sort', 'filters'];
  }

  constructor() {
    super();
    // Create Shadow DOM for style/DOM isolation
    this.attachShadow({ mode: 'open' });
    this._mountPoint = document.createElement('div');
    this._mountPoint.id = 'react-root';
    this.shadowRoot.appendChild(this._mountPoint);
  }

  connectedCallback() {
    // Called when the element is added to the DOM
    this._renderReact();
  }

  disconnectedCallback() {
    // Called when the element is removed — clean up React
    unmount(this._mountPoint);
  }

  attributeChangedCallback(name, oldVal, newVal) {
    // Called when an observed attribute changes — re-render React with new props
    if (oldVal !== newVal && this.isConnected) {
      this._renderReact();
    }
  }

  _renderReact() {
    // Gather attributes as props
    const props = {
      category: this.getAttribute('category'),
      page: this.getAttribute('page') || '1',
      sort: this.getAttribute('sort') || 'relevance',
      filters: this.getAttribute('filters'),
      // Callback for outbound events
      onAddToCart: (detail) => {
        this.dispatchEvent(new CustomEvent('ml:add-to-cart', {
          bubbles: true, composed: true, detail
        }));
      },
      onNavigate: (path) => {
        this.dispatchEvent(new CustomEvent('ml:navigate', {
          bubbles: true, composed: true, detail: { path }
        }));
      }
    };
    // Mount the React app into the Shadow DOM
    render(this._mountPoint, props);
  }
}

// Register the custom element
customElements.define('ml-catalog-plp', MLCatalogPLP);
💡 That’s the Entire Wrapper ~50 lines of code. No build tool changes. No React component modifications. The React render() function already exists — it’s the same entry point the React app uses today. The wrapper just calls it inside a Shadow DOM instead of document.getElementById('root').

Data Flow Through the Wrapper

Data flow — attributes flow in from the shell, events flow out to the shell, React handles everything in between

The Two-Way Contract

Direction Mechanism Standard Example
In (Shell → Domain) HTML attributes W3C Custom Elements observedAttributes <ml-catalog-plp category="gloves" page="2">
Out (Domain → Shell) Custom Events W3C DOM Events with composed: true new CustomEvent('ml:add-to-cart', { detail })

This is the same interface contract used by every native HTML element. An <input> takes attributes (value, type, placeholder) and fires events (input, change, focus). Web Component wrappers work exactly the same way.

Shadow DOM: Why It Matters

Shadow DOM provides true isolation between the wrapper and the rest of the page. This solves some of the most painful problems in micro frontend architectures:

Problem Without Shadow DOM With Shadow DOM
CSS leakage Domain A’s styles bleed into Domain B. Global .button class conflicts everywhere. Styles are scoped. Domain A’s .button can’t affect Domain B.
DOM collisions Two domains using document.getElementById('root') conflict. Each domain has its own DOM tree inside Shadow DOM.
Global state pollution Domains accidentally modify each other’s DOM via document.querySelector. Shadow DOM is invisible to external querySelector calls.
Framework conflicts Two versions of React can’t coexist on the same page. Each Shadow DOM can load its own framework version in isolation.
Third-party script interference Analytics scripts, chat widgets, A/B testing tools modify the DOM unpredictably. Shadow DOM elements are protected from external DOM manipulation.
Industry Standard: Shadow DOM Encapsulation Shadow DOM is a W3C standard implemented in all modern browsers. It’s used by YouTube (the video player is a Shadow DOM tree), GitHub (many interactive elements use Shadow DOM), and Salesforce Lightning Web Components. It’s the browser-native solution to the micro frontend isolation problem.

Handling Styles Inside Shadow DOM

One question that comes up immediately: “How do I get my React app’s CSS into the Shadow DOM?” There are three approaches, and the right one depends on how our React app currently manages styles:

Approach 1: Inject CSS into Shadow DOM Best for apps with external CSS files

If our React app uses external CSS files (imported via bundler), inject them as <style> or <link> elements into the Shadow DOM root:

connectedCallback() {
  // Inject styles into Shadow DOM
  const style = document.createElement('link');
  style.rel = 'stylesheet';
  style.href = new URL('./styles.css', import.meta.url).href;
  this.shadowRoot.prepend(style);

  this._renderReact();
}

This works because styles inside Shadow DOM are scoped — they only apply to elements within that Shadow DOM tree.

Approach 2: Constructable Stylesheets Best for programmatic style injection

Use the CSSStyleSheet constructor to create a stylesheet and adopt it into the Shadow DOM:

const sheet = new CSSStyleSheet();
sheet.replaceSync(cssText); // cssText is your component's CSS string
this.shadowRoot.adoptedStyleSheets = [sheet];

Constructable Stylesheets are a browser standard that allows sharing stylesheet objects across multiple Shadow DOM trees efficiently.

Approach 3: CSS-in-JS (Emotion, Styled Components) Works automatically with Shadow DOM target

If our React app uses CSS-in-JS (Emotion, styled-components), configure the style injection target to point at the Shadow DOM root instead of <head>:

// Emotion: specify container
import { CacheProvider } from '@emotion/react';
import createCache from '@emotion/cache';

const cache = createCache({
  key: 'ml-catalog',
  container: this.shadowRoot // Styles inject here, not <head>
});

Most CSS-in-JS libraries support a custom injection point. Once configured, styles flow into the Shadow DOM automatically.

The Gradual Migration Path

The wrapper is a bridge, not the destination. Over time, teams can replace the React internals with WCFramework components at their own pace. This is a team-level decision, not a platform-wide mandate.

Stage What’s Inside the Wrapper Shell Sees When
Stage 1 Full React app, unchanged <ml-catalog-plp> Day 1 of shell swap
Stage 2 React app with some WCFramework sub-components <ml-catalog-plp> When team has capacity
Stage 3 Fully WCFramework, no React dependency <ml-catalog-plp> When team does next major rewrite

The shell never knows and never cares what’s inside the wrapper. It just renders <ml-catalog-plp> and passes attributes. Whether that element contains React 17, React 19, vanilla JS, or WCFramework components is invisible to the shell. This is the power of the Web Component boundary.

💡 No Big Bang Rewrite Teams migrate their internals when it makes sense for them: during a feature rewrite, when fixing a major bug, or when upgrading a dependency. The wrapper buys us time — potentially years. We never have to rewrite everything. We just have to wrap everything once (in ~50 lines), and then the internal migration happens organically.

Common Questions

Can two domains use different React versions? Yes, via Shadow DOM isolation

Yes. Because each domain runs inside its own Shadow DOM and loads its own bundled dependencies, Domain A can run React 17 while Domain B runs React 18 (or any other version). There’s no shared React runtime — each domain bundles its own. This eliminates the “coordinated React upgrade” problem that plagues monolithic micro frontend architectures.

What about performance? Doesn’t bundling React per domain increase total JS? Shared externals via import map

Initially, yes — each domain bundles its own React. But the import map can be used to share a single React instance across domains that use the same version:

// In the import map
"react": "https://cdn.medline.com/shared/react@17.0.2/index.js",
"react-dom": "https://cdn.medline.com/shared/react-dom@17.0.2/index.js"

Domains that want to share React reference it as an external. Domains that need a different version bundle their own. The import map gives us per-domain control.

How do I test the wrapper? Unit tests + integration tests + visual regression
  • Unit test the wrapper — Verify connectedCallback mounts React, disconnectedCallback unmounts, attributeChangedCallback re-renders with new props
  • Integration test — Load the custom element in a test harness, set attributes, verify DOM output, verify events fire correctly
  • Visual regression — Screenshot the wrapped component and compare with the unwrapped version. They should look identical.
  • Existing React tests — These continue to pass without modification. The wrapper doesn’t change React’s behavior.
What about server-side rendering? Not needed in the CSR shell

The new CSR shell doesn’t do server-side rendering. Domains render client-side inside the browser. For SEO-critical pages (PDP, PLP), pre-rendering or static site generation handles the SEO requirement (see page 13: CSR Hydration). The wrapper pattern is purely client-side.

Can I use this pattern for non-React MFEs? Yes — any framework works

The wrapper pattern works with any UI framework or library. Replace ReactDOM.render() with:

  • Vue — createApp(App).mount(this._mountPoint)
  • Svelte — new App({ target: this._mountPoint })
  • Vanilla JS — this._mountPoint.innerHTML = template(props)
  • WCFramework — Already a Web Component, no wrapper needed

The shell only cares about the custom element API. What runs inside is each team’s choice.
