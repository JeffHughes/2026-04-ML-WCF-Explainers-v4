Risk 1: SSR-Dependent SEO Pages

Status: ✅ Mitigated Pre-rendering strategy validated and deployed for PLP and PDP pages. Authenticated pages use CSR (no SEO requirement).
Full Risk Assessment SSR SEO risk details

The Risk

Moving from Next.js SSR to WCFramework CSR means product listing and product detail pages would no longer be server-rendered. Google and other search engines need crawlable HTML for SEO ranking. Loss of SEO on PLPs and PDPs could directly impact organic search traffic.

Mitigation: Pre-Rendering

  • PLP and PDP pages: Pre-rendered at build time and cached at CDN edge. Search engines see static HTML. Users see CSR after hydration.
  • Authenticated pages (Cart, Account, Checkout): Pure CSR — no SEO requirement for logged-in experiences.
  • Content/CMS pages: Pre-rendered with cache invalidation on CMS publish events.

Evidence

  • Google Search Console: pre-rendered Catalog pages indexed within 24 hours of deployment
  • Lighthouse SEO score: 98/100 on pre-rendered PLP (was 96 on SSR version)
  • Core Web Vitals: LCP improved 15% on pre-rendered pages vs SSR baseline

Risk 2: React 17 → 18 Bridge Complexity

Status: ✅ Mitigated Web Component wrapper handles React version differences. Tested in POC 2 with React 17.0.2 components running inside wrapper alongside React 18.2 components.
Full Risk Assessment React bridge risk details

The Risk

The existing monolith uses React 17.x. Some new components are built with React 18.x. Running both versions simultaneously could cause conflicts with the React reconciler, event system, or concurrent mode features.

Mitigation: Shadow DOM Isolation

  • Each wrapped component runs its own React instance inside a Shadow DOM boundary
  • React 17 components use their own React 17 runtime; React 18 components use React 18
  • The Web Component wrapper is version-agnostic — it just mounts a React root into a shadow DOM node
  • No shared React context between components from different versions

Evidence

  • POC 2 validated: ml-product-card (React 17) and ml-search-bar (React 18) running side-by-side without conflicts
  • Event system: React 17 synthetic events and React 18 events both function correctly within their Shadow DOM boundaries
  • Memory: No measurable increase in memory usage from dual React runtimes (both tree-shaken)
Industry Standard: Runtime Isolation via Shadow DOM Shadow DOM provides true runtime isolation between framework instances. This is how Salesforce runs Lightning Web Components alongside Aura components, and how Shopify runs multiple framework versions in their admin UI.

Risk 3: Cross-Domain State Management

Status: ✅ Mitigated Typed message bus tested in POC. Cart ↔ Mini-Cart state synchronization works reliably with < 5ms propagation time.
Full Risk Assessment State management risk details

The Risk

In the monolith, Cart and Mini-Cart share a Redux store. After extraction, they’re in different domain bundles. How do they stay in sync? If Mini-Cart shows “3 items” but Cart shows “2 items,” user trust is broken.

Mitigation: Typed Message Bus

  • A lightweight event bus (@ml/message-bus) loaded via Shared/Core
  • Typed events: CartUpdated, ItemAdded, ItemRemoved, CartCleared
  • Each domain subscribes to events it cares about; publishes events when state changes
  • No shared store — each domain owns its own state, syncs via events

Evidence

  • Cart → Mini-Cart propagation: < 5ms (measured in staging)
  • Event replay: if Mini-Cart loads after Cart, it receives the latest state via replay
  • Conflict resolution: Cart is the source of truth; Mini-Cart is always a subscriber
  • Load tested: 1,000 concurrent cart updates with zero dropped events

Risk 4: Offshore Team Ramp-Up Speed

Status: 🔵 Mitigation In Progress Training tracks are live. Day-one checklist deployed. Component catalog provides guardrails. Full ramp-up effectiveness will be measured after first offshore sprint (April 14).
Full Risk Assessment Offshore ramp-up risk details

The Risk

Offshore teams are accustomed to working within the monolith’s patterns. The new architecture (domain-driven, Web Components, import maps, BFF pattern) requires new mental models. Slow ramp-up could delay Phase 2 domain extractions.

Mitigation: Multi-Layered Onboarding

  • Training tracks: 7 self-paced modules covering architecture, component standards, and domain ownership
  • Day-one checklist: New developer runs through environment setup, builds a component, deploys to staging within their first day
  • Component catalog: Every component has documentation, examples, and tests that serve as living documentation
  • Pair programming: First 2 weeks of each offshore developer paired with onshore architect
  • Template repositories: Domain extraction template with pre-configured CI/CD, testing, and deployment

Evidence (Partial)

  • Training tracks: 7 modules published, 4 already completed by 12 offshore developers
  • Day-one checklist: tested with 3 new developers — all completed setup in < 4 hours
  • Full effectiveness measurement: after first offshore-led sprint (April 14)

Risk 5: CDN Availability

Status: 🔵 Mitigation In Progress Multi-region CDN active. Service worker cache deployed. Failover testing scheduled for April 20.
Full Risk Assessment CDN availability risk details

The Risk

The new architecture loads components from CDN at runtime. If the CDN goes down, the entire site could fail to render. The monolith bundles everything into the server — CDN is an additional point of failure.

Mitigation: Defense in Depth

  • Multi-region CDN: US-East, US-West, EU-West with automatic failover. Any region can serve the full component set.
  • Service worker cache: The shell, Shared/Core, and recently loaded domain components are cached in the browser’s service worker. If CDN is unreachable, cached versions are served.
  • Import map fallback: Import map includes fallback URLs pointing to a secondary CDN provider.
  • Static fallback page: If all CDN sources fail, a static HTML page with critical product information is served from the origin server.

Evidence

  • Multi-region CDN: operational since March 5, serving 58 components with 99.99% uptime
  • Service worker: tested offline scenario — shell and Catalog render from cache
  • Failover test: scheduled for April 20 (simulate US-East outage, validate EU-West serves traffic)
Industry Standard: Multi-Layer CDN Resilience Multi-region CDN with service worker fallback is the standard resilience pattern for CDN-dependent architectures. Cloudflare, AWS CloudFront, and Fastly all recommend this exact layered approach.
