Managing Drift: Automation Over Coordination

Version divergence across domains is expected, desired, and managed — not a problem to solve, but a feature to operate

Drift Is Not the Enemy

Once domains split and run independently, they will run different versions of shared components. The Catalog domain will be on ml-header@2.3. The Cart domain will still be on ml-header@2.1. Account might be on ml-header@2.0.

This is not a bug. This is the design. Independent deployment requires independent versioning. The alternative — forcing every domain to upgrade every shared component in lockstep — is what creates the coordination bottlenecks that slow enterprises down. The goal: no more dependency inconsistencies that silently break other teams' builds.

“Forced lockstep upgrades across 84 micro frontends is what got us into this situation.” The monolith’s single version for everything means one team’s upgrade blocks 83 other teams. The solution is not to replicate that problem at the component level. The solution is to manage version divergence with automation.

The Version Landscape

The App Catalog provides a real-time view of the version landscape — every domain, every component, every version. This visibility turns “unknown divergence” into “known, measured, manageable variance.”

Version Landscape — Catalog is fully current; Cart has one available upgrade; Account has three upgrades including one critical
Industry Standard: Managed Version Divergence Every package manager in the world (npm, Maven, NuGet, pip) allows consumers to pin different versions of the same dependency. Micro frontend architecture applies this same principle at the deployment level. Kubernetes, Terraform, and every infrastructure-as-code tool is designed around the assumption that different environments will run different versions simultaneously.

Automated Upgrade Proposals

The system doesn’t just show version differences — it actively proposes upgrades with full context, impact analysis, and automated scripts to execute them.

How Proposals Work

  1. Detection: The App Catalog identifies domains running older versions of shared components
  2. Analysis: The diff engine computes exactly what changed between the domain’s current version and the latest version
  3. Classification: Changes are classified as additive (safe), modified (review), or breaking (requires code changes)
  4. Script generation: For safe and review-level changes, a change-mod script is generated automatically
  5. Notification: The owning team receives a proposal with the diff, the impact, and the script
  6. Execution: The team reviews, runs the script, tests, and deploys at their own pace
Example Upgrade Proposal “Cart domain is on ml-header@2.1. Catalog and Checkout are on ml-header@2.3. Changes between v2.1 and v2.3: 2 additive attributes, 1 new event, 0 breaking changes. Change-mod script ready. Estimated effort: import map update only (no code changes).”

Change-Mod Scripts

Change-mod scripts are the engine that removes friction from version upgrades. They are automatically generated, context-specific scripts that apply known-good transformations to update a domain’s component versions.

Change-Mod Script Flow — Proposals are generated automatically; teams review and approve at their own pace; scripts apply transformations; tests verify; rollback is instant

What Change-Mod Scripts Do

Upgrade TypeScript ActionHuman Action
Import map version bump Update version pointer in import map: ml-header: "2.1.0" → "2.3.0" None — review and deploy
New optional attribute Update import map; optionally add the new attribute to usage sites Review if we want the new feature
Renamed attribute Find-and-replace old attribute name with new name across domain codebase Review replacements; run tests
Removed attribute Flag usage sites; suggest alternative if documented in changelog Manual refactor needed at each usage site
Event payload change Find event listeners; generate adapter code for new payload shape Review adapter code; update custom logic

The Safety Net

Every upgrade is a new deployment. Every deployment is reversible. This is the fundamental safety guarantee that makes drift management stress-free.

Immutable CDN Artifacts Old versions never disappear

When a team upgrades from ml-header@2.1 to ml-header@2.3, version 2.1 is still on CDN. It hasn’t been deleted. It hasn’t been overwritten. It’s still cached at the edge. If the upgrade causes any issue, reverting the import map pointer to 2.1 restores the previous behavior instantly.

  • Zero rebuild required for rollback
  • Zero coordination with the component author
  • CDN cache ensures the old version loads instantly
  • Version history in the catalog shows exactly what was rolled back
Independent Deployment Each team's upgrade, timeline, risk

Each domain’s upgrade is an independent deployment. The Cart team upgrading ml-header does not affect the Catalog team, the Account team, or any other domain. If the Cart team’s upgrade goes wrong, they roll back. Nobody else notices.

  • Upgrade and rollback happen within the domain’s deployment pipeline
  • No cross-team coordination needed for upgrade or rollback
  • Each domain can canary-test upgrades with a percentage of traffic
  • Blue-green deployment ensures zero-downtime upgrades
Automated Testing Every upgrade runs through the full test suite

Change-mod scripts don’t just apply changes — they trigger the domain’s full test suite:

  • Unit tests verify component behavior hasn’t changed
  • Visual regression tests catch pixel-level differences
  • Integration tests verify the component works within the domain’s pages
  • Contract tests verify the component’s interface matches what the domain expects
  • If any test fails, the upgrade is paused and the team is notified with the failure details

Drift Categories

Not all drift is equal. The catalog classifies drift into four categories, each with a different urgency and response:

✅

Healthy Drift

Domain is one minor version behind. No breaking changes. Upgrade is a simple import map change. No urgency. This is the normal operating state for most domains most of the time.

🟡

Attention Drift

Domain is two or more minor versions behind. New features available. Change-mod script ready. Team notified with proposal. Upgrade recommended within the current quarter.

🟠

Stale Drift

Domain is a major version behind. Some breaking changes in the upgrade path. Change-mod script available but requires review. Team should plan the upgrade in the next sprint cycle.

🔴

Critical Drift

Domain is running a version with a known security vulnerability or critical bug. Change-mod script generated as priority. Team notified immediately. Upgrade should happen within the current sprint.

Automation vs. Coordination

The traditional approach to version management is coordination: meetings, spreadsheets, upgrade deadlines, cross-team alignment sessions. The automation approach replaces every one of those activities with tooling.

ActivityCoordination ApproachAutomation Approach
Detecting drift Manual audit of package.json across repos App Catalog shows version landscape in real-time
Planning upgrades Cross-team meeting to agree on upgrade timeline Automated proposals sent to each team independently
Executing upgrades Each team manually updates dependencies and fixes breaking changes Change-mod scripts apply known-good transformations automatically
Validating upgrades Manual testing and cross-team integration verification Automated test suite runs as part of change-mod script execution
Communicating status Status updates in Slack, email threads, spreadsheet updates Health dashboard visible to all teams in real-time
Handling failures Incident response, cross-team debugging, coordinated rollback Domain rolls back independently; old version still on CDN
The Goal: Zero Coordination Meetings About Versions If a version upgrade requires a meeting, the tooling has failed. Every upgrade should be proposable, reviewable, executable, and reversible without any human in any other team needing to be involved.

The Drift Management Lifecycle

Drift management is not a one-time activity. It’s a continuous lifecycle that runs automatically alongside normal development:

  1. Publish: Component author publishes a new version to CDN
  2. Catalog: App Catalog detects the new version and computes diffs against all consumer versions
  3. Classify: Each consumer’s upgrade path is classified (healthy, attention, stale, critical)
  4. Propose: Change-mod scripts are generated for each consumer with context-specific instructions
  5. Notify: Teams receive proposals at their preferred cadence (immediately for critical, weekly for routine)
  6. Review: Teams review proposals and decide when to upgrade
  7. Apply: Change-mod scripts execute the upgrade and run the test suite
  8. Deploy: Upgraded domain deploys through its normal pipeline
  9. Monitor: Health dashboard updates to reflect the new version landscape
  10. Repeat: The cycle continues with every new component publish
Industry Standard: Automated Dependency Management Dependabot (GitHub), Renovate (Mend), and Snyk all implement this same lifecycle for npm/Maven/NuGet packages: detect drift, propose upgrades, generate pull requests, run tests. The App Catalog applies this proven pattern to CDN-distributed Web Components with import map version management.
