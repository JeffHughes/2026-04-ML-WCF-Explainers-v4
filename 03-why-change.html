The Big Picture

The Next Evolution

We've taken the canonical architecture as far as it goes. Here's the next iteration that unlocks the release velocity we need.

What You've Built — and Why It Was Right

Let's be clear about something: the current platform was the right call. Next.js, React, micro frontends on Kubernetes—this was the canonical enterprise frontend architecture. Your teams executed it well. The platform works. Orders flow. Teams ship.

The architecture delivered exactly what it was designed to deliver:

  • Component reuse across 84 micro frontends
  • SSR for performance on initial page loads
  • Kubernetes orchestration for scaling
  • React ecosystem for developer productivity and hiring

None of that was wrong. All of it was the industry standard answer at the time. The question isn't whether the current architecture was a mistake—it wasn't. The question is: have we extracted everything we can from this pattern?

No Change in Communication to Leadership This is not a "we need to start over" conversation. This is the same team, the same product, the same platform—taking the next step. The same way you moved from a monolithic backend to microservices, this is the frontend catching up to the same pattern. Just the next iteration, with additional details on how we get more releases per year.

The Ceiling We've Hit

The canonical React/Next.js micro frontend architecture has a ceiling, and your platform has reached it. This isn't a failure—every architecture pattern has limits. The signs are well-documented across the industry:

▬

Release Velocity Plateaued

Despite having capable teams, releases per quarter have flattened. Big-bang releases are the only option—the shared monolith pipeline forces coordination that caps throughput.

▭

Framework Upgrade Backlog

React 17, Next.js 12—the ecosystem has moved on. Each upgrade requires the entire monolith to move together. So the upgrade keeps getting deferred.

▮

Testing Takes Longer Than Building

Full regression runs because nobody can confidently say "this change only affects Catalog." The monolith couples everything, so you have to test everything.

▯

Every Team Waits for Every Other Team

Shared build pipeline. Shared deploy schedule. Shared failure domain. The fastest team moves at the speed of the slowest.

Add to that the day-to-day friction: dependency inconsistencies across 84 MFEs (one team's upgrade breaks another team's build), and weird UI flicker issues from competing hydration and style conflicts in the shared SSR runtime. These aren't bugs in your implementation. They're inherent limits of the pattern. Every organization that's scaled this architecture to 50+ micro frontends hits the same ceiling. The industry's answer is consistent: decompose into independent domains with a standards-based composition layer.

The Framework Treadmill

There's a second pressure beyond the coupling ceiling: the React/Next.js ecosystem moves on a cycle your platform can't keep pace with.

Figure 3-1: React ecosystem reinvents itself every 18 months. W3C Web Components have been backwards-compatible since 2018.

This isn't a criticism of React. React is excellent software. But its upgrade cadence is designed for teams that rebuild frequently. Enterprise platforms that need 5–15 year lifespans need a foundation that doesn't move under them.

W3C Web Components: The Foundation That Doesn't Move Custom Elements, Shadow DOM, HTML Templates, and ES Modules are W3C standards built into every browser. They don't have breaking version upgrades. Code written to the Web Component spec today will work in browsers a decade from now—the same way HTML from 2005 still renders correctly in 2026. This Is Not a Rewrite

The most important thing to understand about this next phase: we are not starting over.

  • The existing code stays. We copy it per domain and trim what doesn't belong.
  • The existing tests stay. Same tests, same assertions, smaller scope.
  • The existing infrastructure stays. Same K8s clusters, same CI/CD patterns.
  • The existing components stay. React components get wrapped in Web Component shells—the internals don't change.
  • The existing teams stay. Same people, same pods, clearer ownership boundaries.

What changes is the architecture boundary. Instead of 84 micro frontends sharing one monolithic shell, you get 8–12 domain apps that each pod owns completely. Each app contains its own Web Components, its own pipeline, its own deploy schedule.

Same Story, Next Chapter If anyone asks "what changed?", the answer is: nothing changed in what we deliver. We're delivering the same product, the same features, the same experience. What changed is that we can now deliver more of it, faster, with less coordination overhead. It's the same platform taking its next step.

Where the Industry Went

The pattern we're proposing isn't experimental. It's where every enterprise at this scale has ended up:

□

Web Components Everywhere

GitHub, Salesforce, Adobe, SAP, ING Bank, Microsoft, Red Hat—all rebuilt their component systems on W3C Web Components. Not because the old approach was wrong. Because the standard matured and the industry converged.

▣

Domain-Driven Decomposition

Amazon, Spotify, Zalando—the pattern is universal at scale. Independent teams need independent deployment. The micro frontend monolith was a stepping stone; domain apps are the destination.

▩

Strangler Fig Evolution

The UK Government Digital Service, BBC, and hundreds of enterprise platforms used strangler fig to evolve without rewriting. Copy the working system, split it, trim it. Never turn off what works until the new version is proven.

↦

CSR + BFF for Commerce

Netflix, Spotify, and most authenticated commerce platforms moved from SSR to CSR with BFF backends. SSR makes sense for SEO-only content pages. For authenticated commerce, CSR + BFF is faster and simpler.

Industry-Proven Patterns Domain-Driven Design (Evans, 2003), Strangler Fig (Fowler, 2004), W3C Web Components (stable 2018), Backend-for-Frontend (Newman, 2015). These aren't emerging ideas. They are the established playbook for enterprise platforms at this scale, validated across thousands of organizations over two decades.

What Happens If We Stay Put

Standing still is always an option. But standing still on a treadmill means falling behind. Here's the honest trajectory if the current architecture stays as-is:

TimelineWhat Happens
Year 1Next.js 12 loses community support. Security patches slow. New hires expect modern tooling. Release velocity stays flat despite team investment.
Year 2React 17 ecosystem shrinks. Third-party libraries drop support. Teams spend more time on workarounds than features. The gap between your stack and the industry widens.
Year 3A forced upgrade becomes unavoidable—but now it's bigger, because the monolith has accumulated 3 more years of coupling. The jump from React 17 to whatever's current in 2029 is much larger than evolving now.
The Best Time to Evolve Is When Things Work The worst time to make an architectural change is when the platform is in crisis. Right now the platform works. Teams are capable. The business is stable. That's exactly when you make a deliberate, measured, low-risk evolution—not when you're forced to.

The Next Iteration

Given where we are and where the industry has gone, the next iteration is clear:

  1. Decompose into domain apps using copy-split-strip (zero risk, same code, same tests)
  2. Evolve the shell to W3C Web Components (industry standard, no vendor lock-in)
  3. Establish an App Catalog for visibility, scanning, and automated governance
  4. Evolve components incrementally to WCFramework at each pod's pace
  5. Adopt a 1-year core cadence with automated scripts that handle the upgrade

Each step is safe. Each step is reversible. Each step delivers value independently. And critically—each step builds on everything you've already built. Nothing gets thrown away.
