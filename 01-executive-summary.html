The Big Picture

The 90-Second Pitch

What we're doing, why it works, and why the whole team sleeps better at night. The entire evolution strategy in the time it takes to get coffee.

The Core Message

This is a smooth migration to the next evolution of the platform. No new framework to learn—we are going back to first principles: real HTML, real DOM, simpler code that lasts forever. We are taking the monolith we have today—the one that works, the one that processes orders, the one our teams know inside and out—and we are splitting it into independent domains that each team can own, deploy, test, and evolve on their own schedule.

The One-Sentence Version Copy the monolith, trim it per domain, swap the shell to industry-standard Web Components, catalog everything, and automate core updates on a 1-year cadence.

Every step is reversible. Every phase delivers value on its own. If we stop at any point, we're still better off than where we started.

Five Things That Change

📋

1. Copy, Don't Rewrite

We copy the entire monolith repository for each domain, then delete what doesn't belong. The code that runs in production tomorrow is the same code running today—just with less of it.

🏗️

2. Independent Domains

Each domain becomes its own repository, its own pipeline, its own deployment. Catalog deploys without waiting for Cart. Account ships without blocking Checkout.

🔌

3. Standards-Based Shell

The orchestration layer moves from a framework-specific SSR monolith to W3C Web Components—SEO-friendly CSR that runs from a CDN. No framework lock-in. No server rendering infrastructure.

📦

4. App Catalog

A registry tracks every micro frontend: what version it's running, what APIs it calls, what components it uses, and whether it's healthy. Automated scanning catches problems before they reach production.

🔄

5. Every Step Is Reversible

At every phase, the original monolith is still running. If anything goes wrong, traffic routes back instantly. There is no point of no return. We can pause, roll back, or stop entirely—and still be better off.

🔄

6. Same Team, Same Skills

HTML, CSS, JavaScript—the skills our teams already have. No new language. No new paradigm. First-principles web development that every developer already knows and every AI tool already understands.

Why This Is Safe

Every migration strategy lives or dies on one question: what happens when something goes wrong?

Industry-Standard Safety Pattern The Strangler Fig pattern has been the go-to enterprise migration approach for over 20 years. It was named by Martin Fowler and has been validated at scale by organizations from Amazon to the UK Government Digital Service. The core principle: never turn off the old system until the new system is proven.

Here's why our approach is inherently safe:

  • Same code: When we copy-split-strip a domain, we're running the exact same code. It was tested in the monolith. It works in the monolith. It will work as a standalone domain because it is the same code.
  • Original stays running: The monolith continues to serve traffic until the domain is proven. If anything goes wrong, traffic routes back to the original.
  • Phase independence: Each phase delivers value on its own. There's no need to complete all four phases to benefit. Even stopping after Phase 1 (domain extraction) gives us independent deployment.
  • Reversibility: Every step can be rolled back. The copy-split-strip domains can be merged back. The shell swap can be reversed. The component transition is incremental.
Deep Dive: The Four Phases of Safety How each phase provides protection

Phase 1: Domain Extraction (Copy-Split-Strip)

Risk: Near zero. We're literally copying code that already works and removing the parts that don't apply to this domain. The deployment infrastructure is identical—same K8s pods, same CI/CD pipeline, same monitoring.

Rollback: Route traffic back to the monolith. The domain copy can be deleted without any impact.

Phase 2: Shell Swap

Risk: Low. The new shell loads the exact same micro frontends. It just uses Web Components as the integration layer instead of the Next.js SSR runtime. Each MFE is wrapped in a Web Component container—the internal code doesn't change.

Rollback: Switch the load balancer back to the Next.js shell. Both shells run in parallel during the transition.

Phase 3: App Catalog & Governance

Risk: None. The catalog is a read-only registry that scans and tracks what's running. It doesn't modify any deployed code. It only provides visibility and automated scanning.

Rollback: Turn off the catalog. Nothing else changes.

Phase 4: Component Evolution

Risk: Low, incremental. React components are wrapped first, then teams can incrementally migrate to WCFramework, keep using React, use plain vanilla JS, or whatever is easiest for them and their components. Each change is self-contained within a single domain. Other domains are unaffected.

Rollback: Revert the component PR. The wrapped React version continues to work.

Who Benefits and How

Technical Directors Strategic benefits
  • Predictable core updates: The 1-year cadence means we know exactly when updates happen and can plan resources accordingly.
  • Reduced coordination overhead: Domain independence means fewer cross-team meetings, fewer merge conflicts, fewer "who broke the build" investigations.
  • Industry-standard stack: W3C Web Components and standards-based patterns mean the platform is built on technology that won't be abandoned by a single vendor.
  • Incremental risk: No more big-bang releases. No flag day. Every step is proven before the next step begins. We shift left on quality—catching issues early in the pipeline instead of in production.
Development Teams Day-to-day improvements
  • Ship when ready: Each domain deploys when its tests pass. No waiting for 83 other teams.
  • Own the dependencies: Each team's node_modules, build pipeline, test suite. When something breaks, it's something the team controls.
  • Smaller blast radius: A bug in Catalog doesn't take down Checkout. A regression in Account doesn't block Cart.
  • Clearer ownership: Each team knows exactly what it owns, what APIs it consumes, and what contracts it exposes.
Operations & Platform Teams Infrastructure improvements
  • Independent scaling: Scale the Catalog domain during a flash sale without scaling Account.
  • Targeted rollbacks: Roll back a single domain without touching others.
  • Better observability: The App Catalog gives us a single pane of glass showing what's deployed, what version, and whether it's healthy.
  • Simplified SSR elimination: Moving to CSR+BFF means simpler infrastructure—static files served from a CDN plus lightweight BFF services.
