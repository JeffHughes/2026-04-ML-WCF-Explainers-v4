The Shell Swap: Next.js to WCFramework

How to replace the legacy SSR shell with a standards-based CSR shell — without downtime, without risk, with instant rollback

The Four-Phase Shell Swap

The shell swap is not a “big bang” migration. It happens in four controlled phases, each independently verifiable and reversible. At no point is there a moment where “everything switches at once.”

Four-phase shell swap — from legacy SSR to standards-based CSR, each phase independently reversible

Phase 1: Domains Are Extracted

Before the shell swap begins, domains have already been extracted using the Copy-Split-Strip pattern (pages 5–9). Each domain runs as its own service with its own SSR shell — still Next.js, still React, but independently deployed.

What This Looks Like

  • Catalog runs at medline-catalog serving /products/*, /product/*, /category/*
  • Cart runs at medline-cart serving /cart
  • Each service has its own Next.js instance doing SSR
  • An ingress controller routes traffic to the right service based on URL path
  • Users don’t notice any difference — the experience looks identical
💡 Why Not Jump Straight to CSR? Because you need to prove domain independence first. If domains can’t deploy independently with their existing SSR shells, they certainly can’t do it with a new CSR shell. Phase 1 validates the domain boundaries before introducing a new rendering model.

Phase 2: New Shell Deployed Alongside

The new WCFramework CSR shell is deployed as a separate service. It’s not replacing anything yet — it runs alongside the old shell, accessible via a feature flag, an internal-only URL, or a specific cookie.

Deployment Setup

Aspect Old Shell (Next.js SSR) New Shell (WCFramework CSR)
URL www.medline.com/* preview.medline.com/* (or feature flag)
Rendering Server-side via Node.js Client-side via static HTML + JS from CDN
Domain modules React MFEs loaded via Module Federation Same React MFEs wrapped as Web Components
Infrastructure K8s pods running Node.js SSR servers Static files on CDN (no server)
Who sees it All production users Internal testers only

During Phase 2, the new shell is thoroughly tested by internal teams. Every page is compared side-by-side with the old shell. Visual regression tests run against both. Performance metrics are compared. Only when confidence is high does the team move to Phase 3.

Phase 3: React MFEs Wrapped as Web Components

This is where the magic happens. Existing React micro frontends are wrapped in thin Web Component adapters (detailed on the next page). From the new shell’s perspective, every domain is just a custom element: <ml-catalog-plp>, <ml-cart-page>, <ml-checkout-flow>.

What Changes for Domain Teams

  • Add a Web Component wrapper — A thin adapter (~50 lines) that mounts the React app inside a custom element with Shadow DOM
  • Register the custom element — customElements.define('ml-catalog-plp', CatalogPLPElement)
  • Publish to CDN — The domain module (React code + wrapper) is built and published as a versioned ES module
  • No React code changes — The React components inside the wrapper don’t change at all. They render exactly as before.
Industry Standard: Progressive Migration via Web Component Wrappers Wrapping existing framework components in Web Components is a proven migration strategy used by Google (wrapping Polymer in Lit), GitHub (wrapping jQuery in Web Components), and Salesforce (wrapping Aura in Lightning Web Components). The wrapper provides a standards-based interface while the internals migrate at their own pace.

Phase 4: Traffic Shift

With the new shell tested and all domains wrapped, traffic is gradually shifted from the old shell to the new one. This uses the same percentage-based rollout used for domain extraction:

Rollout Schedule

Stage New Shell Traffic % Duration Gate Criteria
Canary 1% 2 days Error rate < baseline, Core Web Vitals stable
Early Adopter 5% 3 days No P1/P2 incidents, conversion rate stable
Ramp 1 25% 5 days Performance metrics match or exceed old shell
Ramp 2 50% 5 days No regressions at scale
Full 100% — Old shell decommissioned
Traffic migration flow — gradual shift with instant rollback via feature flag or load balancer rules

Why WCFramework?

The new shell is built with the WCFramework — a standards-based component framework that uses browser-native APIs exclusively. Here’s why this choice matters for a platform that needs to last 10+ years:

Factor Next.js / React Shell WCFramework CSR Shell
Standards compliance Proprietary JSX, virtual DOM, React-specific APIs W3C Custom Elements, Shadow DOM, HTML Templates, ES Modules
Framework lock-in All MFEs must use React. Version upgrades affect everyone. No lock-in. Domains can use any internal framework (or none).
Upgrade path React 17 → 18 → 19 → ... requires coordinated migration Browser standards are forever. No framework upgrade cycle.
Server infrastructure Node.js SSR servers (compute, scaling, memory) Static files on CDN (no servers to manage)
Expected lifespan React’s API surface changes every 2–3 years Browser standards last 10–20+ years
Team independence Shared React runtime creates coupling Web Component boundaries provide true isolation
✅ The Shell Becomes Infrastructure, Not Application With WCFramework, the shell is more like a web server than an application. It routes requests, serves static content, and provides shared infrastructure. It has no opinions about what framework domains use internally. It just needs them to register a custom element. That’s the entire contract.

Rollback: The Safety Net

At every phase of the swap, rollback is instant and requires no code changes:

Phase 2 Rollback New shell is preview-only, no traffic impact

The new shell is only accessible to internal testers. To “rollback,” simply turn off the preview URL or remove the feature flag. Zero production impact.

Phase 3 Rollback Web Component wrappers are additive, not destructive

The Web Component wrappers are added alongside the existing React code. They don’t modify the React components. If wrapping causes issues, simply don’t load the wrapper and the React MFE continues to work in the old shell as before.

Phase 4 Rollback Route traffic back to old shell instantly

Traffic split is controlled by a feature flag or load balancer rule. To rollback: set the new shell percentage to 0%. All traffic immediately returns to the old Next.js shell. The old shell is kept running and warmed until the new shell has been at 100% for an agreed period (typically 2–4 weeks).

  • No redeployment needed
  • No data migration to reverse
  • No code changes required
  • Takes effect in seconds
Industry Standard: Blue-Green Deployment Running the old and new shells simultaneously is a blue-green deployment pattern. Netflix, Amazon, and Google all use this approach for major infrastructure transitions. The key principle: never remove the old thing until the new thing has been running successfully in production for an extended period.

What Teams Need to Do

The shell swap requires minimal work from domain teams. Here’s the exhaustive list:

Task Effort Who When
Add Web Component wrapper to domain module ~50 lines of code Domain team Phase 3
Register custom element in module entry point 1 line of code Domain team Phase 3
Publish domain module to CDN CI/CD pipeline update Domain team + Platform team Phase 3
Verify domain works in new shell Testing (automated + manual) Domain team Phase 3
Switch to event-based communication If not already done during extraction Domain team Phase 3
💡 Total Effort per Domain: 1–2 Days The Web Component wrapper is a thin adapter. It doesn’t require rewriting any React code. A senior developer can add the wrapper, register the element, update the build pipeline, and verify functionality in 1–2 days. The rest of the team continues shipping features uninterrupted.
