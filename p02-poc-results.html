What We Did

  1. Forked the full monolith repository (next-monolith, 2.4GB)
  2. Deleted all non-Catalog page routes, components, and API handlers
  3. Removed unused node_modules dependencies (saved 340MB)
  4. Deployed to isolated K8s namespace with existing Helm charts
  5. Ran full test suite against the extracted domain

Key Finding

The extraction took 3 days, not the 3 weeks originally estimated. The monolith’s route-based structure made it straightforward to identify Catalog-owned files. The biggest risk — hidden imports from other domains — was resolved by the TypeScript compiler flagging 14 missing references, all of which were utility functions that were moved to Shared/Core.

Evidence

  • Test results: 347 passed, 0 failed, 0 skipped (Jest output)
  • K8s deployment: Catalog pod running in ml-catalog-poc namespace, healthy for 72 hours
  • Performance: p50 latency 91ms (PDP), 108ms (PLP) vs monolith baseline 87ms / 112ms
Industry Standard: Copy-Split-Strip Validation The Strangler Fig pattern, validated here, is the industry-standard approach to monolith decomposition. Netflix, Amazon, and Shopify all used variations of this exact pattern. The key: our existing code IS the migration — no rewrite.

Components Wrapped

  1. ml-product-card — PLP product tile (Catalog domain)
  2. ml-search-bar — Header search with typeahead (Search domain)
  3. ml-mini-cart — Cart flyout with item count (Cart domain)
  4. ml-breadcrumb — Navigation breadcrumbs (Shared/Core)
  5. ml-promo-banner — Promotional banner with countdown (Content domain)

Wrapper Pattern

Each wrapper is a W3C Custom Element that: (1) creates a Shadow DOM root, (2) injects the component’s stylesheet into the shadow root, (3) renders the React component into a mount point inside the shadow, and (4) forwards attributes as React props. Average wrapper: ~40 lines of code.

Key Finding

The wrapper is ~40 lines of boilerplate code per component, and it’s identical across all wrappers except for the React component import and prop mapping. This means wrapping can be automated with a code generator. No React source code modifications were needed — the wrapper sits completely outside the React component tree.

Evidence

  • Render comparison: Pixel-identical output between monolith and wrapped versions (Chromatic visual diff)
  • Performance: First Contentful Paint within +12ms of unwrapped baseline across all 5 components
  • Shadow DOM audit: Zero CSS declarations leaked between components (DevTools audit)
Industry Standard: W3C Web Components Shadow DOM isolation via W3C Web Components is the browser-native solution to micro frontend CSS conflicts. It’s the same pattern used by GitHub (github-elements), Salesforce (Lightning Web Components), and Google (Material Web). No polyfills needed in modern browsers.

What We Scanned

All running BFF instances in the staging environment: Catalog BFF (8 endpoints), Cart BFF (6 endpoints), Search BFF (5 endpoints), and Account BFF (4 endpoints).

Breaking Changes Detected

  1. Cart BFF /api/cart/items: Response field unitPrice renamed to itemPrice in a pending PR. Scanner caught it via contract diff.
  2. Account BFF /api/account/addresses: New required field countryCode added to request body. Would have broken Checkout’s address lookup.

Hidden Dependencies Found

  1. Mini-Cart was calling Catalog BFF directly for product names instead of going through Cart BFF
  2. Search was calling Account BFF for personalized results — undocumented requirement
  3. Checkout was calling Cart BFF’s internal health endpoint for availability checks

Evidence

  • Full dependency graph generated as interactive visualization (deployed to internal tools)
  • 23 OpenAPI specs committed to ml-api-contracts repository
  • Breaking change report integrated into PR review workflow
Industry Standard: Contract Testing & API Governance Automated contract discovery and breaking change detection is what separates enterprise micro frontend architectures from prototypes. Pact, Spring Cloud Contract, and Backstage all implement variations of this pattern. We catch breaking changes in CI, not in production.
