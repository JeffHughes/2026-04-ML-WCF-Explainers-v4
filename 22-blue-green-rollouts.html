Blue/Green Deployments at Domain Level

When deployment is boring, we deploy daily. Import-map switching gives every domain instant, zero-downtime deployments with sub-second rollback.

Why Deployment Shouldn't Be Scary

In the current monolith, every deployment is a coordinated event. Multiple teams must align schedules, merge code into a shared release branch, and deploy together during a designated window. If anything goes wrong, everyone rolls back — even teams whose changes were fine.

🚨 The Monolith Deployment Tax When deployment is scary, we deploy less. When we deploy less, each deployment carries more changes. More changes mean more risk. More risk makes deployment scarier. It's a vicious cycle that the domain-level blue/green model breaks entirely.

The modernized architecture inverts this: no more big-bang releases. Each domain deploys independently via blue/green switching—or canary rollouts for higher-risk changes. No coordinated release trains. No deployment windows. No all-hands-on-deck releases. Just faster independent deployments—a team pushing their code through their own pipeline whenever they're ready.

How Blue/Green Works with Import Maps

The import map is the deployment mechanism. It's a JSON file that tells the browser which version of each domain's JavaScript bundle to load from the CDN. Switching from blue (current) to green (new) is simply updating a URL in that JSON file.

Blue/Green Deployment via Import Map

Fig 22.1 — Blue/green deployment: the import map points to CDN-hosted domain bundles; switching versions is a URL update

The Import Map Explained

The import map is a standard browser feature (supported in all modern browsers). It maps module specifiers to URLs. Here's what it looks like in practice:

{
  "imports": {
    "@ml/catalog": "https://cdn.medline.com/mfe/catalog/2.4.1/index.js",
    "@ml/cart":    "https://cdn.medline.com/mfe/cart/1.8.0/index.js",
    "@ml/checkout":"https://cdn.medline.com/mfe/checkout/3.1.0/index.js",
    "@ml/account": "https://cdn.medline.com/mfe/account/1.2.0/index.js"
  }
}

To deploy the Catalog domain's new version, we change one line:

// Before (Blue)
"@ml/catalog": "https://cdn.medline.com/mfe/catalog/2.4.1/index.js"

// After (Green)
"@ml/catalog": "https://cdn.medline.com/mfe/catalog/2.5.0/index.js"
✅ That's the Entire Deployment Update one URL. The next page load picks up the new version. No server restarts. No build pipeline waiting. No other domain affected. The old version remains on the CDN — rollback is reverting that one URL.

Instant Rollback

Rollback is the inverse of deployment: revert the import map entry. Because the previous version's bundle is still on the CDN (it was never removed), reverting is instant. No rebuilding, no redeploying, no waiting.

Rollback Procedure

  1. Detect issue — monitoring alerts fire (error rate spike, performance degradation)
  2. Revert import map — change the URL back to the previous version (2.5.0 → 2.4.1)
  3. Verify — confirm the issue is resolved via the same monitoring
  4. Investigate — the green version remains on the CDN for debugging
⏱ Sub-Second Rollback The entire rollback process — from "something is wrong" to "it's fixed" — takes seconds. No rebuild, no redeploy, no approval chain. The import map update propagates on the next page load. For existing sessions, a soft refresh picks up the reverted version.

Canary Rollouts: Gradual Traffic Shifting

For high-risk changes, teams can opt into canary deployments — routing a small percentage of traffic to the green version while monitoring for issues before switching fully.

Canary Rollout with Monitoring

Fig 22.2 — Canary rollout: gradual traffic shift with real-time metric comparison
Step-by-Step Canary Deployment Gradual promotion

Phase 1: Initial Canary (5% traffic)

  • Deploy green version to CDN
  • Configure edge router to send 5% of traffic to green
  • Monitor for 30 minutes: error rate, p95 latency, core web vitals
  • If metrics are healthy, proceed to Phase 2

Phase 2: Expanded Canary (25% traffic)

  • Increase traffic split to 25%
  • Monitor for 1 hour with broader traffic patterns
  • Validate business metrics (conversion rates, add-to-cart rates)

Phase 3: Majority Traffic (50% traffic)

  • Increase to 50% — now half of users are on the new version
  • Monitor for 2 hours across peak and off-peak periods
  • Compare aggregate metrics between blue and green cohorts

Phase 4: Full Promotion (100%)

  • Update the import map to point to green version for all traffic
  • Blue version remains on CDN as instant rollback target
  • Canary complete — domain is fully deployed
Rollback Procedure When canary goes wrong

If metrics degrade at any canary phase:

  1. Automated alert fires — threshold breach on error rate or latency
  2. Traffic immediately routed 100% to blue — edge router removes green from rotation
  3. Team investigates — green version remains accessible for debugging at its CDN URL
  4. Fix and retry — deploy a new green version and restart the canary process

The key insight: at no point during a canary does the blue version stop being available. It's always there as the safety net.

Self-Service Domain Deployment

Every domain team controls their own deployment pipeline. No tickets. No approvals from other teams. No waiting for a shared deployment window. The pipeline is:

  1. Push to main — triggers CI pipeline
  2. Quality gates pass — unit tests, contracts, visual regression, security scan
  3. Bundle published to CDN — new version available at its unique URL
  4. Import map updated — either directly (blue/green) or via canary configuration
  5. Monitoring confirms — dashboards verify the deployment is healthy
Industry Standard: Self-Service Deployment Amazon deploys to production every 11.7 seconds on average. This is only possible because teams own their own deployment pipelines. The platform provides the infrastructure (CDN, import maps, monitoring), and teams control the timing. This is the same model applied to domain-level micro frontends.
💬 The Mindset Shift "When deployment is scary, we deploy less. When deployment is boring, we deploy daily." The goal is to make deployment so routine, so automated, and so safe that it's just another part of the development workflow — not a special event.
