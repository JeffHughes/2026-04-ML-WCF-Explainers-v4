Testing Strategy: Domain-Level Confidence

How domain-scoped testing, contract verification, and automated quality gates give every team confidence to ship independently — without standing up the entire monolith.

The Monolith Testing Problem

In the current monolith, testing a single button change requires standing up the entire application. Build times are long, test suites are fragile, and a failure in one domain's tests can block every other domain's deployment. This is the testing equivalent of a traffic jam — one accident stops everyone.

⚠️ The Anti-Pattern If our test suite requires standing up the entire monolith, our test suite is the problem. Domain-scoped testing inverts this: each team tests their own domain in isolation, with contract checks at the boundaries.

The modernized architecture changes testing fundamentally. Because each domain is independently deployable, each domain is also independently testable. Most testing happens within a single domain's boundary, with lightweight contract checks at the integration points.

The Domain Testing Pyramid

Each domain follows a structured testing pyramid. The base is wide (fast, cheap, plentiful unit tests) and the top is narrow (slow, expensive, minimal E2E tests). This is not a new concept — but applying it at the domain level rather than the application level is what makes it work at scale.

Layer-by-Layer Breakdown

🎯 Unit Tests — Fast, Local, CI-Friendly Base of the pyramid

Each Web Component is tested in isolation using standard DOM testing utilities. Tests run in milliseconds, require no network, and execute in CI without any special infrastructure.

  • Scope: Single component, single function, single utility
  • Speed: Entire domain suite runs in under 30 seconds
  • Tools: Standard test runners (Vitest, Web Test Runner) with DOM assertion libraries
  • Coverage target: 80%+ line coverage per domain
  • Who runs them: Every developer, on every commit, in every CI pipeline
// Example: Testing a product-card Web Component
describe('product-card', () => {
  it('renders product name in shadow DOM', async () => {
    const el = document.createElement('ml-product-card');
    el.setAttribute('name', 'Surgical Gloves');
    document.body.appendChild(el);
    const nameEl = el.shadowRoot.querySelector('.product-name');
    expect(nameEl.textContent).toBe('Surgical Gloves');
  });
});
🔗 Integration Tests — Domain + BFF Real HTTP, mocked backends

Integration tests verify that a domain's frontend communicates correctly with its BFF (Backend-for-Frontend). The BFF is started locally, and backend services behind it are mocked using contract stubs.

  • Scope: Full domain frontend + its BFF, mocked downstream services
  • Speed: Suite runs in 1–3 minutes
  • Key validation: HTTP request/response shapes, error handling, loading states
  • Who runs them: CI pipeline on every pull request
📜 Contract Tests — API Boundary Verification Prevents integration drift

Contract tests verify that the agreed-upon API shapes between services are maintained. If the Catalog BFF expects a certain response format from the Product Service, the contract test catches any deviation before deployment.

  • Scope: API boundaries — frontend↔BFF and BFF↔backend
  • Pattern: Consumer-driven contracts (Pact or similar)
  • Key insight: Contracts are checked independently by both sides — no coordination needed
  • Failure mode: Contract mismatch fails the CI pipeline of the breaking side
📷 Visual Regression — Screenshot Comparison Catch UI drift

Every component in the App Catalog has baseline screenshots. When a component changes, visual regression compares the new rendering against the baseline. Pixel-level differences are flagged for human review.

  • Scope: Individual components and key page compositions
  • Tools: Chromatic, Percy, or similar screenshot diffing services
  • Threshold: Configurable pixel-diff tolerance per component
  • Approval flow: Visual changes require explicit sign-off in the PR
🎯 E2E Tests — Critical User Journeys Top of the pyramid

End-to-end tests cover the handful of critical user journeys that span multiple domains: login → search → product detail → add to cart → checkout. These are expensive, slow, and deliberately minimal.

  • Scope: Multi-domain journeys running against a staging environment
  • Count: 10–20 scenarios, not hundreds
  • Speed: Suite runs in 5–10 minutes
  • When: Nightly against staging, and before production import-map updates
  • Key insight: Cross-domain tests are minimal because contracts are enforced at the boundaries

Contract Testing: The Boundary Guardian

Contract testing (covered in API Scanning and Data Contracts) ensures domains can evolve independently without breaking each other. In the testing context, contracts replace the need for full-stack integration tests by verifying that each side of an API boundary honors the agreed-upon shape.

App Catalog-Driven Test Targeting

The App Catalog doesn't just track which components exist — it also tracks which domains use which components. This dependency graph powers intelligent test targeting: when a shared component changes, only the domains that consume it are tested.

How Test Targeting Works

  1. A developer changes the ml-data-table core component
  2. The App Catalog identifies that Catalog, Orders, and Inventory domains consume ml-data-table
  3. CI triggers test suites only for those three domains
  4. Cart, Checkout, and Account domains are unaffected — their pipelines don't run
✅ Smart Test Targeting Instead of running every test for every change, the system knows which domains are affected and runs only those tests. This cuts CI time dramatically and eliminates false-positive failures from unrelated domains.

CI Pipeline Integration Automated test selection

The CI pipeline queries the App Catalog's dependency graph at the start of every build. It compares the changed files against the component-to-domain mapping and generates a targeted test plan.

  • Changed file analysis: Git diff → affected components
  • Catalog query: Affected components → consuming domains
  • Test plan generation: Consuming domains → test suites to execute
  • Parallel execution: Each domain's tests run in parallel workers
  • Result aggregation: All results reported back to the PR as status checks

Automated Quality Gates

Every domain deployment passes through a series of automated quality gates. These gates are enforced by CI — no human approval needed for standard deployments. The gates are:

✅

Unit Test Gate

All unit tests pass. Coverage threshold met (80%+ lines). No new warnings introduced.

📜

Contract Gate

All consumer and provider contracts verified. No breaking changes at any API boundary.

📷

Visual Regression Gate

No unintended visual changes. Pixel-diff within tolerance. Intentional changes approved in PR.

📚

Catalog Validation Gate

Component metadata valid. Documentation complete. Required props documented. Examples render correctly.

🔓

Security Gate

Dependency audit clean. No known CVEs. CSP headers verified. No inline scripts or eval() usage.

♿

Accessibility Gate

axe-core automated scan passes. No critical or serious violations. ARIA attributes validated.

Industry Standard: Shift-Left Quality Netflix, Spotify, and Amazon all enforce automated quality gates in their CI pipelines. The principle is "shift left" — catch defects as early as possible, ideally before the code leaves the developer's machine. This includes automated CVE (Common Vulnerabilities and Exposures) scanning on every build so known security vulnerabilities are caught before they ever reach production. Automated gates make this happen without manual review bottlenecks.
