Account, Search, Content, and Admin Domains

Completing the domain map — every bounded context defined, every API dependency mapped, every shared component accounted for

The Complete Domain Map

After extracting Catalog (page 7) and Cart/Mini-Cart/Checkout (page 8), four more domains remain: Account, Search, Content/CMS, and Admin. Plus the Shared/Core layer that provides cross-cutting capabilities to every domain.

All domains with their BFFs — the CSR shell routes traffic and provides shared capabilities via the event bus

Account Domain

The Account domain owns everything related to the authenticated user’s profile and history. It’s a medium-complexity domain with moderate release cadence.

Account Capabilities Profile, orders, addresses, payments, preferences
👤

User Profile

Name, email, phone, company info, job title. Profile editing with validation and confirmation.

📦

Order History

Past orders with line items, status tracking, reorder functionality, invoice download, returns initiation.

🏠

Address Book

Saved shipping and billing addresses. Default address selection. Address validation and standardization.

💳

Payment Methods

Saved payment cards (tokenized), purchase order defaults, payment preferences for B2B accounts.

⚙

Preferences

Communication preferences, notification settings, display preferences (list vs. grid), language/locale.

❤

Wishlists

Multiple wishlists, shared wishlists for team purchasing, wishlist-to-cart bulk add.

Cross-Domain Dependencies

Account data is consumed by other domains (Checkout reads saved addresses, Mini-Cart uses account preferences), but Account never imports code from other domains. All cross-domain reads go through the Account BFF’s internal API.

Search Domain

Search is a cross-cutting capability — the search bar appears on every page (provided by the shell), but the search results page and the search logic are owned by the Search domain.

Search Capabilities Bar, autocomplete, results, filters, recent searches

Search Bar (Shell-Hosted)

The search bar component (<ml-search-bar>) lives in the shell header. It fires ml:search-submit events and renders autocomplete suggestions from the Search BFF. It’s a shared component, not owned by the Search domain, but it consumes Search domain APIs.

Autocomplete / Typeahead

As the user types, the search bar calls /search/suggest on the Search BFF. Results include product suggestions, category matches, and recent searches (per-user, from cookies or account data).

Search Results Page

The full search results experience at /search?q=... is owned by the Search domain. It includes:

  • Result grid with product cards (uses <ml-product-card> from Catalog’s shared component library)
  • Faceted filtering (brand, category, price range, availability, attributes)
  • Sort options (relevance, price, newest, best-selling)
  • Pagination with “load more” and traditional page numbers
  • Search result count and active filter summary
  • “Did you mean?” spelling corrections
  • Zero-results page with suggestions

Recent Searches

Per-user recent search history, stored client-side (localStorage) for anonymous users and server-side for authenticated users. Displayed as suggestions when the search bar is focused.

Content / CMS Domain

The Content/CMS domain owns all non-product content pages: marketing landing pages, promotional banners, help articles, and dynamically managed content blocks.

Content Capabilities Marketing, banners, landing pages, help center
📰

Marketing Pages

Landing pages for campaigns, seasonal promotions, brand spotlights. Content managed via headless CMS, rendered as static pages or CSR components.

🏷

Promotional Banners

Site-wide banners, category-specific promotions, hero carousels. Served as Web Components that any domain can embed: <ml-promo-banner slot="category-hero">.

📖

Help / Support Pages

FAQ, how-to guides, contact forms, shipping information. Managed as structured content in the headless CMS.

🎨

Content Blocks

Reusable content slots that other domains embed: “related resources,” “seasonal picks,” “featured brands.” Served from CMS BFF as Web Components.

Admin Domain

The Admin domain is an internal-only application used by operations, merchandising, and customer support teams. It has fundamentally different users, different auth requirements, and a completely different deployment cadence compared to the customer-facing domains.

Admin Capabilities Order management, inventory, content editing, user admin

Order Management

  • View and search all orders by status, date, customer, or order number
  • Manual order modifications (cancel, modify, refund)
  • Shipping label generation and tracking updates
  • Customer service notes and communication log

Inventory Management

  • Stock level dashboard with alerts for low inventory
  • Manual stock adjustments with audit trail
  • Warehouse assignment and transfer management
  • Backorder management and estimated availability dates

Content Editing

  • Headless CMS admin interface for creating/editing pages, banners, and content blocks
  • Preview and publish workflow with approval gates
  • Scheduled content publishing for campaigns

User Management

  • Internal user account management and role assignment
  • Customer account lookups and support tools
  • B2B account hierarchy management (parent/child organizations)
  • Permission management and access control lists
⚠️ Admin Is a Separate Application Admin does not run in the same shell as the customer-facing domains. It has its own shell, its own authentication (internal SSO), its own deployment pipeline, and its own security policies. It happens to talk to some of the same backend services, but it is architecturally independent.

Shared / Core Layer

Some capabilities don’t belong to any single domain — they’re platform primitives used by every domain. These are published as versioned packages to CDN and loaded via the import map.

How Shared Components Work

  1. Published to CDN — Shared components are built, versioned, and uploaded to CDN as ES modules
  2. Loaded via import map — The shell’s import map maps @ml/shared/header to https://cdn.medline.com/shared/header@2.1.0/index.js
  3. Every domain uses the same version — The import map is the single source of truth for which version of a shared component is active
  4. Version updates are atomic — Update the import map entry, and every domain immediately loads the new version on next page load
Shared component distribution — all domains load shared modules through the import map from CDN

Component Tag Responsibility Consumed By

Industry Standard: Import Maps & ES Modules Import maps are a W3C standard (supported in all modern browsers) that allow controlling module resolution at runtime without a build step. Combined with ES modules served from CDN, they provide a dependency management system that requires no bundler, no shared node_modules, and no coordinated builds. This is how browsers were designed to load code.

Extraction Priority and Sequencing

Not all domains need to be extracted simultaneously. The sequencing is driven by three factors: independence value (how much does this team gain from independent deployment?), extraction complexity (how tangled is this code in the monolith?), and risk (what breaks if something goes wrong?).

Priority Domain Independence Value Extraction Complexity Risk
1 Catalog High — highest traffic Low — read-only Low
2 Cart High — frequent promo changes Medium — event wiring Medium
3 Mini-Cart Medium — small surface Low — 2-3 components Low
4 Search High — search improvements Medium — cross-cutting bar Low
5 Account Medium Medium Low
6 Checkout High — payment flows High — many backends High — payments
7 Content/CMS Medium Low — mostly static Low
8 Admin High — internal tool Medium — different auth Low — internal only
💡 Checkout Goes Late, Not Early Checkout is the riskiest domain to extract because it handles payments. It goes after Cart (step 6, not step 2) because by that point, the team has extracted five domains successfully, the pattern is proven, and confidence is high. Never start with the riskiest piece.

Key Takeaway

Every domain follows the same pattern: Copy → Strip → Build BFF → Route Traffic. The differences are in scope, risk profile, and timing — not in approach. Once Catalog proves the pattern, every subsequent extraction is a repeatable playbook with decreasing uncertainty.

Industry Standard: Bounded Contexts with Explicit Interfaces Each domain is a bounded context (Domain-Driven Design) with an explicit API contract (the BFF). Cross-domain communication uses events or BFF-to-BFF calls — never shared database tables, never shared state stores, never direct code imports. This is the architecture pattern used by Amazon (each team owns a service), Spotify (each squad owns a domain), and Zalando (each vertical owns their micro frontend).
