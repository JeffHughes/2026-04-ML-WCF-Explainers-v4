Deep Dive: The Catalog Domain

The pilot domain extraction — what gets copied, what gets stripped, and how the Catalog BFF owns its data contract

Why Catalog as the Example

Catalog is the most obvious example domain—it has clear ownership boundaries and was the domain Vincent picked for the initial proof of concept. Product listing pages (PLP) and product detail pages (PDP) are overwhelmingly read-only — they display data fetched from the product API. They don’t write to carts, they don’t process payments, they don’t manage user sessions. That makes it a clean, easy-to-follow illustration of how Copy-Split-Strip works.

Example, Not Mandate We use Catalog throughout these documents because it’s the most straightforward example. The actual extraction order will be simplest first, hardest last—determined by the domain committee when the time comes.

Once any domain is running independently, the pattern is proven. Every subsequent domain extraction follows the same playbook, with decreasing risk each time.

What the Catalog Domain Owns

After extraction, the Catalog domain has sole responsibility for the following capabilities. No other domain touches these pages or data flows:

📄

Product Listing Pages (PLP)

Category browsing, grid/list views, pagination, sort options, filter facets applied to results. Every URL matching /products/* or /category/*.

🔍

Product Detail Pages (PDP)

Individual product view with images, specifications, pricing, availability, variant selection, and related products. URLs like /product/:sku.

🔎

Product Search

Search results pages, search suggestions, typeahead/autocomplete, recent searches. The search bar component is shared; the results page is owned by Catalog.

📁

Category Navigation

Category tree, breadcrumbs, browse-by-category menus. The taxonomy data is owned by the Catalog BFF and served as a structured endpoint.

🌄

Product Images & Media

Image gallery, zoom, 360° views, product videos. All media assets served from CDN, referenced by the product data API.

🚀

Product Data API

The Catalog BFF exposes endpoints for product search, product detail, category tree, and faceted filtering. This is the single source of truth for product information.

What Gets Deleted from the Copy

After copying the monolith to create medline-catalog, the strip phase removes everything that doesn’t belong. This is a subtractive process — you start with everything and remove what you don’t need.

Cart Logic — Completely Removed Cart pages, add-to-cart handlers, cart state management

All cart-related routes (/cart, /cart/*), cart state slices, add-to-cart button handlers, and cart API clients are deleted. The Catalog domain has zero knowledge of cart state.

The “Add to Cart” button on PDP remains, but it fires a domain event (ml:add-to-cart) that the Cart domain’s micro frontend catches. The Catalog domain doesn’t know or care what happens next.

// Catalog domain emits a custom event — that's it
this.dispatchEvent(new CustomEvent('ml:add-to-cart', {
  bubbles: true, composed: true,
  detail: { sku: 'MED-12345', qty: 1 }
}));
Checkout Flows — Completely Removed Multi-step checkout, payment, address validation

Every checkout route (/checkout/*), payment gateway integration, address validation logic, order submission handlers, and confirmation page templates are removed. The Catalog service never processes a transaction.

Account Pages — Completely Removed Profile, order history, saved addresses, payment methods

All account management routes (/account/*), profile forms, order history views, address book, and saved payment methods are deleted. Account is its own domain with its own extraction timeline.

Admin Console — Completely Removed Internal tools, inventory management, user admin

Admin routes (/admin/*), internal dashboards, inventory management screens, and user administration tools are removed. These are a completely separate bounded context with different users (internal staff), different auth requirements, and different deployment cadence.

CMS Pages — Completely Removed Marketing pages, banners, landing pages

Content marketing pages, promotional landing pages, banner management, and any CMS-driven content outside product data are removed. The Content/CMS domain owns these independently.

What Stays As-Is in Phase 1

In Phase 1 we just copy the shell—the code stays exactly as it is. The header, footer, flyouts, mini-cart, all of it comes along in the copy. Nothing gets rewritten yet. The goal is a working independent deployment, not a refactor.

Later—when the new WCF shell is ready—we turn the header, footer, and flyouts into Web Components and let the shell hydrate them. That’s when the shared UI becomes truly shared (loaded once, used everywhere) instead of copied into each domain.

Phase 1 = Copy & Ship. Refactor Later.

Don’t overthink what stays and what goes during extraction. Copy it all, delete what obviously belongs to other domains, ship it. The Web Component conversion of shared chrome (header, footer, flyouts) happens in the shell swap phase, not during extraction.

The Catalog BFF (Backend-for-Frontend)

The Catalog BFF is a lightweight API layer that sits between the Catalog micro frontend and the underlying backend services. It owns the data contract — the shape of the JSON that Catalog pages consume.

Why a BFF Instead of Direct Backend Calls?

  • Contract stability — Backend services can change their internal APIs without breaking the frontend. The BFF absorbs the translation.
  • Aggregation — A single PDP page might need data from Product Service, Pricing Service, Inventory Service, and Media Service. The BFF combines them into one response.
  • Performance — The BFF runs server-side, close to the backends. Multiple backend calls collapse into a single HTTP response to the browser.
  • Security — Internal service credentials never reach the browser. The BFF authenticates the user and passes sanitized, authorized data.
Catalog API surface — the BFF aggregates multiple backend services into a clean frontend-specific contract

Catalog BFF API Endpoints

Endpoint Method Description Consumer
/products GET Paginated product listing with sort and filter support PLP Component
/products/:id GET Full product detail including variants, pricing, media, availability PDP Component
/categories GET Category tree for navigation menus and breadcrumbs Category Nav, Breadcrumbs

Deployment: Same Infrastructure, Independent Service

The extracted Catalog domain runs in the same Kubernetes cluster, uses the same container registry, and follows the same CI/CD pipeline patterns as the monolith. The only difference is that it has its own deployment pipeline.

What Changes

  • Own Kubernetes deployment — medline-catalog runs as a separate deployment with its own replicas, scaling rules, and health checks
  • Own CI/CD pipeline — Catalog team merges to main and deploys without waiting for any other team
  • Own monitoring — Dedicated dashboards, alerts, and SLOs for Catalog-specific metrics
  • Traffic routing — Ingress routes /products/*, /product/*, /category/*, /search/* to the Catalog service

What Stays the Same

  • Same Kubernetes cluster and namespace conventions
  • Same container base images and security scanning
  • Same logging, tracing, and observability stack
  • Same DNS, SSL certificates, and CDN configuration
  • Same feature flag service and configuration management
Industry Standard: Independent Deployability Independent deployment per bounded context is the defining characteristic of a micro frontend architecture. Amazon, Spotify, Zalando, and IKEA all structure their platforms this way. Each team owns their deploy pipeline end-to-end. The platform team provides the infrastructure primitives; the domain teams compose them.

Catalog Extraction Checklist

Here’s the step-by-step checklist for extracting Catalog. Each step is a discrete, verifiable task:

Phase 1: Copy Create the independent Catalog service
  1. Fork the monolith repository → medline-catalog
  2. Set up independent CI/CD pipeline for the new repository
  3. Deploy to Kubernetes as a separate service (same code, runs identically)
  4. Configure ingress to route /catalog-preview/* to the new service (internal only)
  5. Verify: Every Catalog page renders identically in both services
Phase 2: Strip Remove non-Catalog code
  1. Delete all cart routes, components, and state
  2. Delete all checkout routes, components, and state
  3. Delete all account routes, components, and state
  4. Delete all admin routes, components, and state
  5. Delete all CMS routes, components, and state
  6. Run the full test suite — fix any broken imports or missing dependencies
  7. Verify: Catalog pages still render correctly. All other routes return 404.
Phase 3: BFF Create the Catalog Backend-for-Frontend
  1. Define the Catalog BFF API contract (OpenAPI spec)
  2. Implement BFF endpoints: /products, /products/:id, /categories, /search, /facets, /suggestions
  3. Wire the Catalog frontend to call the BFF instead of direct backend services
  4. Add BFF-level caching (product data changes infrequently)
  5. Verify: All Catalog pages load data exclusively through the BFF
Phase 4: Go Live Route production traffic to Catalog service
  1. Enable feature flag for Catalog routing (10% → 25% → 50% → 100%)
  2. Monitor error rates, latency, and Core Web Vitals at each percentage
  3. Confirm rollback works: flip the feature flag back, traffic returns to monolith
  4. At 100%: remove Catalog routes from the monolith (the monolith gets smaller)
  5. Celebrate: The first domain is independently deployed 🎉

Risk Mitigation

Risk Mitigation Rollback
Catalog pages break after stripping Full visual regression test suite runs before each deployment Route traffic back to monolith via feature flag (instant)
BFF introduces latency BFF runs in same cluster as backends; response time monitored via SLOs Catalog can temporarily bypass BFF and call backends directly
Shared header/footer diverge between Catalog and monolith Shared components published to CDN; both services consume the same version Pin to known-good version; investigate drift before updating
Search index consistency Catalog BFF reads from the same search index as the monolith No change to underlying search infrastructure
