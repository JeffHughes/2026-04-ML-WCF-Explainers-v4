Component Standards: The ML- Convention

One naming convention, one contract model, total independence — the rules every Medline component follows

Why Standards Matter at Scale

When you have 84+ micro frontends, each owned by a different team, each deployed independently, the only thing that keeps the system coherent is a shared convention. Not a shared codebase. Not a shared build. A convention.

The ml- prefix convention gives every team a predictable, discoverable, and collision-proof naming system. Every component in the Medline ecosystem starts with ml-, followed by a domain-specific or shared identifier. No exceptions.

The Rule Is Simple If it renders pixels on a Medline page, it’s a Web Component with an ml- prefix. If it doesn’t have the prefix, it’s not a Medline component and the shell won’t load it.

The Component Contract

Every ml- component follows a strict interface contract. This contract is what makes components framework-agnostic from the outside, regardless of what technology powers them internally.

Three Pillars of the Contract

  • Documented Attributes — Every input the component accepts is a standard HTML attribute. No framework-specific bindings. <ml-product-card sku="12345" variant="compact">
  • Documented Events — Every output the component emits is a standard CustomEvent. No framework-specific event systems. el.addEventListener('ml-add-to-cart', handler)
  • Documented Slots — Content projection uses standard <slot> elements. No framework-specific content projection. <ml-card><span slot="title">My Title</span></ml-card>
Industry Standard: Custom Elements v2 The W3C Custom Elements specification defines exactly this pattern — attributes in, events out, slots for composition. Every major browser supports it natively. No polyfills needed. This is the same component model used by GitHub, Salesforce Lightning, Adobe Spectrum, and SAP UI5.

Shadow DOM Isolation

Every ml- component uses Shadow DOM to encapsulate its styles. This means:

  • Component CSS cannot leak out into the page
  • Page CSS cannot accidentally break the component
  • Two components from two different teams, built at two different times, with two different CSS methodologies — they coexist peacefully
  • No more !important wars. No more naming collision debugging. No more “it worked on my page but broke on yours.”

Self-Registration

Every component self-registers via customElements.define(). There is no central registration file. No manifest that needs updating. When the browser loads ml-product-card.js, the component announces itself to the DOM and is immediately available.

// ml-product-card.js
class MlProductCard extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
  }
  // ... component logic
}
customElements.define('ml-product-card', MlProductCard);

Component Anatomy

Every ml- component follows the same internal architecture, regardless of size or complexity. Understanding this anatomy makes building and debugging components predictable across every team.

CDN Module Loading

Components are distributed via CDN, not npm. This is a critical distinction: build once, deploy many. When a component is loaded from CDN, every consumer gets the exact same artifact. No build-step variations. No version resolution surprises. No node_modules deduplication games.

The Version Path Convention

Every component version is immutable on CDN. The URL encodes the exact version:

/components/ml-product-card/2.1.0/ml-product-card.js
/components/ml-cart-summary/1.4.2/ml-cart-summary.js
/components/ml-address-form/3.0.0/ml-address-form.js

The shell’s import map resolves component names to CDN paths. Each domain’s import map can pin to a different version. Catalog can use ml-header@2.3 while Cart stays on ml-header@2.1. No coordination required.

CDN Module Loading — Import map resolves names to versioned CDN URLs; browser registers the Custom Element on load
Cache Forever Because each version has its own URL path, CDN cache headers can be set to max-age=31536000 (one year). Users who have loaded a version once never download it again. New versions get new URLs — no cache busting needed.

Shared npm Package vs. CDN Web Component

The traditional approach of distributing UI components as npm packages creates hidden coupling. CDN Web Components eliminate every one of those coupling points.

DimensionShared npm PackageCDN Web Component
Style Isolation CSS leaks between consumers; requires BEM/CSS Modules/naming discipline Shadow DOM enforces isolation at the browser level
Versioning npm resolution can install different versions across consumers; deduplication is unpredictable Exact version in URL path; every consumer of v2.1.0 gets identical artifact
Independence Consumer must rebuild and redeploy to pick up a new version Update the import map — no rebuild, no redeploy of consumer
Build Coupling Package must be compatible with consumer’s bundler, framework version, and build config Pre-built ES module; no bundler interaction needed
Runtime Sharing Shared dependency must match consumer’s version exactly or risk duplicates Component bundles its own dependencies inside Shadow DOM
Framework Lock-in Package assumes React (or specific framework); consumers must use the same Standard HTML element; works with any framework or no framework
Rollback Revert package version, rebuild, test, redeploy Change import map version pointer; old artifact still on CDN
Industry Standard: CDN-First Component Distribution Salesforce Lightning, Adobe Spectrum, and Shopify Polaris all distribute their design system components as pre-built modules on CDN. The npm package is a convenience wrapper; the CDN artifact is the source of truth. This pattern has been proven at scales far larger than Medline.

The Standard Interface Contract

Attribute Contract How data flows in

Every attribute a component accepts is documented in the App Catalog. Attributes follow standard HTML conventions:

  • Lowercase, hyphen-separated: product-id, display-mode, max-items
  • Primitive types only: strings, numbers, booleans (via presence/absence)
  • Complex data passed via properties (set via JavaScript): el.items = [...]
  • Changes reflected via attributeChangedCallback for reactive updates
<ml-product-card
  sku="ML-12345"
  variant="compact"
  show-price
  locale="en-US"
></ml-product-card>
Event Contract How data flows out

Every event a component emits is a standard CustomEvent with a documented detail payload:

  • Event names prefixed with ml-: ml-add-to-cart, ml-navigate, ml-filter-change
  • Payload in event.detail: { sku: 'ML-12345', quantity: 1 }
  • Events bubble through Shadow DOM boundary (composed: true)
  • Consumers listen with standard addEventListener — no framework binding needed
// Inside the component
this.dispatchEvent(new CustomEvent('ml-add-to-cart', {
  detail: { sku: this.getAttribute('sku'), quantity: 1 },
  bubbles: true, composed: true
}));

// Consumer code — any framework
document.querySelector('ml-product-card')
  .addEventListener('ml-add-to-cart', (e) => {
    console.log('Added:', e.detail.sku);
  });
Slot Contract How content is composed

Named slots allow consumers to project content into specific regions of the component:

  • Named slots documented in the App Catalog: title, actions, badge, footer
  • Default slot for unnamed content projection
  • Slot content styled by the consumer, not the component (light DOM CSS applies)
  • Fallback content inside the slot for when nothing is projected
<ml-card>
  <span slot="title">Featured Product</span>
  <img slot="badge" src="sale-badge.svg" alt="On Sale">
  <div slot="actions">
    <button>Add to Cart</button>
  </div>
  <p>Default slot: This product is amazing.</p>
</ml-card>

Enforcement & Governance

Standards only work if they’re enforced. Here’s how the ml- convention is maintained across all teams:

🛡

CI Pipeline Validation

Every component PR runs automated checks: ml- prefix present, Shadow DOM attached, contract documented, no global style leakage.

📋

App Catalog Registration

Components that don’t register in the App Catalog cannot be published to CDN. Registration includes attribute, event, and slot documentation.

🔍

Runtime Scanner

The API scanner detects undocumented components in production pages and flags them for review. No rogue components in the wild.

📦

Template Generators

Team scaffolding tools generate new components with all conventions pre-applied. The right structure is the default structure.

Standards Without Enforcement Are Suggestions The difference between a standard and a suggestion is automation. Every convention described on this page is enforced by CI, validated by scanners, and baked into templates. Teams don’t have to remember the rules — the tools enforce them.
