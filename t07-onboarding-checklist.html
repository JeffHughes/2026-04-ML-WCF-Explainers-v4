Day One: New Developer Checklist

The literal step-by-step for our first day. Follow these instructions in order. A new developer should see their first change live within 4 hours of cloning the repo.

The Promise

A new developer—whether they sit next to us or are offshore in a different timezone—should be able to follow this page and see their first change deployed to staging within 4 hours. No tribal knowledge required. No Slack messages asking “how do I set up my environment?” Everything is here.

4-Hour Target Clone the repo at 9:00 AM. See our change live in staging by 1:00 PM. This is the standard we hold ourselves to.

Day One Timeline

Day one timeline: from clone to live change in 4 hours

Step 1: Clone Our Domain Repository

We will be assigned to a domain (Catalog, Cart, Account, Checkout, Search, Content, or Admin). Clone that domain's repository.

How to clone ~5 minutes
  1. Open our terminal
  2. Navigate to our projects directory
  3. Run: git clone [your-domain-repo-url]
  4. Enter the cloned directory: cd [domain-name]

Our team lead will provide the repository URL. If we don't have it, check the App Catalog—every domain's repository URL is listed there.

Step 2: Run the Local Dev Setup Script

One command installs everything we need: dependencies, mock BFF, local dev server, and component catalog.

How to set up ~10 minutes
  1. Make sure we have Node.js installed (version specified in .nvmrc)
  2. Run: npm run setup
  3. Wait for it to complete. We will see: ✓ Setup complete. Run 'npm run dev' to start.
  4. Run: npm run dev
  5. Open our browser to http://localhost:3000

If setup fails, check the error message. The most common issues are: wrong Node.js version (check .nvmrc), missing git credentials (check with our team lead), or a firewall blocking npm registry access.

Industry Standard: One-Command Setup A single-command local development setup is the industry standard for developer experience. Every major platform team (Shopify, Stripe, Airbnb) provides a setup script that gets a new developer running in minutes, not hours. If setup takes more than 15 minutes, something is wrong.

Step 3: Browse the Component Catalog

Before we write any code, spend 20 minutes browsing the Component Catalog. See what ML- components already exist. This prevents us from building something that already exists.

How to browse ~20 minutes
  1. With our local dev server running, open: http://localhost:3000/__catalog
  2. Browse the component list. Each component has a name, description, and interactive demo.
  3. Click on 3–5 components that seem relevant to our domain
  4. For each, look at: the attributes it accepts, the events it emits, the slots it provides, and the CSS custom properties for theming
  5. Try changing attribute values in the interactive playground to see how the component responds

The catalog is our primary reference for what components are available. Bookmark it. We'll come back here constantly.

Step 4: Complete Module 1 of Our Training Track

Open the training materials for our track and complete Module 1. For most developers, this is the Domain Developer track, Module 1: Domain Anatomy.

What we'll do ~2 hours
  1. Read the Module 1 documentation (30–60 minutes)
  2. Complete the hands-on exercise in the sandbox environment (60–90 minutes)
  3. Push our exercise code to trigger the automated validation
  4. If validation passes: Module 1 complete. Move on.
  5. If validation fails: read the feedback, fix the issues, re-push

Module 1 covers the basics: domain file structure, route configuration, component slots. By the end, we understand how our domain is organized and how it fits into the platform.

Step 5: Make Our First Change

This is a guided exercise. We will make a small, visible change to our domain. Something we can see in the browser.

What to change ~30 minutes
  1. Pick a page in our domain that uses an ML- component
  2. Add a new ML- component to that page (pick one from the catalog we browsed in Step 3)
  3. Wire it up: pass in the required attributes, handle any events
  4. Verify it renders correctly in our local dev server
  5. Write a simple test that verifies the component is rendered on the page

Keep it simple. This is not about building a feature. This is about proving we can make a change, see it locally, and push it through the pipeline.

Step 6: Push to Our Domain's Staging Environment

Push our change to the staging branch. The pipeline runs automatically.

How to push ~15 minutes
  1. Stage our changes: git add .
  2. Commit with a descriptive message: git commit -m "Add ml-info-card to product page"
  3. Push to the staging branch: git push origin staging
  4. Open the pipeline dashboard (URL provided by our team lead)
  5. Watch the seven gates run. Green checkmarks mean pass. Red X means fail.
  6. If all gates pass, our change deploys to staging automatically

If a gate fails, read the failure report. It will tell us exactly which gate failed, what the specific error is, and how to fix it. Fix, commit, push again. There's no limit on retries.

Industry Standard: Continuous Deployment to Staging Automatic deployment to staging on every push is the industry standard for modern development teams. It gives developers immediate feedback on whether their code works in a production-like environment. Fast feedback loops are the foundation of developer productivity.

Step 7: See Our Change Live in Staging

Open the staging URL for our domain. See the change we just made, running in a production-like environment, composed with all other domains in the shell.

How to verify ~10 minutes
  1. Open the staging URL: https://staging.[your-domain].medline.com
  2. Navigate to the page where we added the component
  3. Verify the component renders correctly
  4. Verify the component responds to interactions (click, hover, etc.)
  5. Check the browser console for any errors

Congratulations. We just went from cloning a repo to seeing our change live in staging. That's the developer experience we've built. Every subsequent change follows the same path, and it only gets faster from here.

Tips for Success

📖

Read the Error Messages

Every error message in the pipeline includes what failed and how to fix it. Read the full message before asking for help. The answer is almost always in the output.

🔍

Use the Component Catalog

Before building anything, search the catalog. The component we need probably already exists. The catalog shows live demos, API docs, and usage examples.

🚀

Push Early, Push Often

Don't wait until our code is “perfect” to push. Push early, get pipeline feedback, iterate. The pipeline catches issues faster than we can spot them manually.

🎯

Stay in Our Domain

Our domain is our sandbox. We don't need to understand every other domain. Focus on ours. Use components from the catalog. Talk to our BFF. That's it.

Industry Standard: Developer-First Onboarding Developer-first onboarding—where new team members can self-serve through a documented, automated process without depending on another person's availability—is the defining characteristic of world-class developer experience. It's what separates productive platforms from platforms that waste developer time.
