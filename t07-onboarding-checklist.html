Day One: New Developer Checklist

The literal step-by-step for your first day. Follow these instructions in order. A new developer should see their first change live within 4 hours of cloning the repo.

The Promise

A new developer—whether they sit next to you or are offshore in a different timezone—should be able to follow this page and see their first change deployed to staging within 4 hours. No tribal knowledge required. No Slack messages asking “how do I set up my environment?” Everything is here.

4-Hour Target Clone the repo at 9:00 AM. See your change live in staging by 1:00 PM. This is the standard we hold ourselves to.

Day One Timeline

Day one timeline: from clone to live change in 4 hours

Step 1: Clone Your Domain Repository

You will be assigned to a domain (Catalog, Cart, Account, Checkout, Search, Content, or Admin). Clone that domain's repository.

How to clone ~5 minutes
  1. Open your terminal
  2. Navigate to your projects directory
  3. Run: git clone [your-domain-repo-url]
  4. Enter the cloned directory: cd [domain-name]

Your team lead will provide the repository URL. If you don't have it, check the App Catalog—every domain's repository URL is listed there.

Step 2: Run the Local Dev Setup Script

One command installs everything you need: dependencies, mock BFF, local dev server, and component catalog.

How to set up ~10 minutes
  1. Make sure you have Node.js installed (version specified in .nvmrc)
  2. Run: npm run setup
  3. Wait for it to complete. You will see: ✓ Setup complete. Run 'npm run dev' to start.
  4. Run: npm run dev
  5. Open your browser to http://localhost:3000

If setup fails, check the error message. The most common issues are: wrong Node.js version (check .nvmrc), missing git credentials (check with your team lead), or a firewall blocking npm registry access.

Industry Standard: One-Command Setup A single-command local development setup is the industry standard for developer experience. Every major platform team (Shopify, Stripe, Airbnb) provides a setup script that gets a new developer running in minutes, not hours. If setup takes more than 15 minutes, something is wrong.

Step 3: Browse the Component Catalog

Before you write any code, spend 20 minutes browsing the Component Catalog. See what ML- components already exist. This prevents you from building something that already exists.

How to browse ~20 minutes
  1. With your local dev server running, open: http://localhost:3000/__catalog
  2. Browse the component list. Each component has a name, description, and interactive demo.
  3. Click on 3–5 components that seem relevant to your domain
  4. For each, look at: the attributes it accepts, the events it emits, the slots it provides, and the CSS custom properties for theming
  5. Try changing attribute values in the interactive playground to see how the component responds

The catalog is your primary reference for what components are available. Bookmark it. You'll come back here constantly.

Step 4: Complete Module 1 of Your Training Track

Open the training materials for your track and complete Module 1. For most developers, this is the Domain Developer track, Module 1: Domain Anatomy.

What you'll do ~2 hours
  1. Read the Module 1 documentation (30–60 minutes)
  2. Complete the hands-on exercise in the sandbox environment (60–90 minutes)
  3. Push your exercise code to trigger the automated validation
  4. If validation passes: Module 1 complete. Move on.
  5. If validation fails: read the feedback, fix the issues, re-push

Module 1 covers the basics: domain file structure, route configuration, component slots. By the end, you understand how your domain is organized and how it fits into the platform.

Step 5: Make Your First Change

This is a guided exercise. You will make a small, visible change to your domain. Something you can see in the browser.

What to change ~30 minutes
  1. Pick a page in your domain that uses an ML- component
  2. Add a new ML- component to that page (pick one from the catalog you browsed in Step 3)
  3. Wire it up: pass in the required attributes, handle any events
  4. Verify it renders correctly in your local dev server
  5. Write a simple test that verifies the component is rendered on the page

Keep it simple. This is not about building a feature. This is about proving you can make a change, see it locally, and push it through the pipeline.

Step 6: Push to Your Domain's Staging Environment

Push your change to the staging branch. The pipeline runs automatically.

How to push ~15 minutes
  1. Stage your changes: git add .
  2. Commit with a descriptive message: git commit -m "Add ml-info-card to product page"
  3. Push to the staging branch: git push origin staging
  4. Open the pipeline dashboard (URL provided by your team lead)
  5. Watch the seven gates run. Green checkmarks mean pass. Red X means fail.
  6. If all gates pass, your change deploys to staging automatically

If a gate fails, read the failure report. It will tell you exactly which gate failed, what the specific error is, and how to fix it. Fix, commit, push again. There's no limit on retries.

Industry Standard: Continuous Deployment to Staging Automatic deployment to staging on every push is the industry standard for modern development teams. It gives developers immediate feedback on whether their code works in a production-like environment. Fast feedback loops are the foundation of developer productivity.

Step 7: See Your Change Live in Staging

Open the staging URL for your domain. See the change you just made, running in a production-like environment, composed with all other domains in the shell.

How to verify ~10 minutes
  1. Open the staging URL: https://staging.[your-domain].medline.com
  2. Navigate to the page where you added the component
  3. Verify the component renders correctly
  4. Verify the component responds to interactions (click, hover, etc.)
  5. Check the browser console for any errors

Congratulations. You just went from cloning a repo to seeing your change live in staging. That's the developer experience we've built. Every subsequent change follows the same path, and it only gets faster from here.

Tips for Success

📖

Read the Error Messages

Every error message in the pipeline includes what failed and how to fix it. Read the full message before asking for help. The answer is almost always in the output.

🔍

Use the Component Catalog

Before building anything, search the catalog. The component you need probably already exists. The catalog shows live demos, API docs, and usage examples.

🚀

Push Early, Push Often

Don't wait until your code is “perfect” to push. Push early, get pipeline feedback, iterate. The pipeline catches issues faster than you can spot them manually.

🎯

Stay in Your Domain

Your domain is your sandbox. You don't need to understand every other domain. Focus on yours. Use components from the catalog. Talk to your BFF. That's it.

Industry Standard: Developer-First Onboarding Developer-first onboarding—where new team members can self-serve through a documented, automated process without depending on another person's availability—is the defining characteristic of world-class developer experience. It's what separates productive platforms from platforms that waste developer time.
