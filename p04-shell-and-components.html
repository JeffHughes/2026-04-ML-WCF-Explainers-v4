Shell Architecture Details How the shell works

Import Map Configuration

The shell loads a single import-map.json from the CDN that maps module specifiers to versioned URLs. When a domain releases a new version, only the import map entry is updated — no shell rebuild required.

{
  "imports": {
    "@ml/catalog": "https://cdn.medline.com/mfe/catalog@2.4.1/index.js",
    "@ml/cart": "https://cdn.medline.com/mfe/cart@1.8.3/index.js",
    "@ml/core": "https://cdn.medline.com/mfe/core@3.1.0/index.js"
  }
}

Domain Loading Flow

When a user navigates to a route owned by a domain: (1) shell resolves the route to a domain module, (2) dynamically imports the module via import map, (3) module registers its Web Component custom elements, (4) shell renders the domain’s root component into the page slot.

Industry Standard: Import Map Module Federation Import maps are a W3C standard (natively supported in all modern browsers) for controlling module resolution at runtime. This is the same pattern Spotify and IKEA use for their micro frontend shells — zero bundler coupling, instant rollback via version change.

High-Priority Components (Wrapped First)

The 42 wrapped components were prioritized by: (1) presence on high-traffic pages (PLP, PDP, header), (2) cross-domain usage, and (3) visual prominence. These 42 components cover 78% of user-visible surface area across Catalog, Cart, Search, and Shared/Core.

Native WCFramework Components

16 components were built natively in WCFramework from scratch. These are primarily new UI elements that didn’t exist in the React monolith: the new breadcrumb system, enhanced product quick-view, accessible modal system, and responsive navigation drawer.

React-Only (89 Components)

These components remain in their original React form within their domain bundles. They’ll be wrapped as their domains are extracted, following the same template used for the first 42.
